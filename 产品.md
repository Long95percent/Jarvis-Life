# Jarvis-Life / ShadowLink 产品说明

> 整理时间：2026-04-24 19:05（Asia/Shanghai）  
> 文档视角：产品定位、目标用户、核心场景、功能边界与当前产品形态。  
> 关联技术说明：`项目说明.md`

## 1. 一句话定位

**Jarvis-Life / ShadowLink 是一个面向个人工作与生活场景的 AI 工作台，帮助用户把大模型对话、本地知识库、任务管理、文件理解、工具调用和多 Agent 协作整合到一个可长期记忆、可主动辅助、可本地化运行的个人智能助理中。**

## 2. 产品愿景

让每个用户拥有一个长期陪伴自己的“个人 AI 操作系统”：

- 能理解用户的资料、项目、任务和偏好。
- 能连接本地文件、知识库、网页、模型和外部工具。
- 能在不同工作模式下切换角色和能力。
- 能从被动问答进化到主动提醒、主动整理、主动协作。
- 能兼顾本地隐私、桌面效率和云端/服务化扩展。

## 3. 当前产品形态

当前仓库中产品存在两个阶段/形态：

### 3.1 当前可用形态：PyQt 桌面端一体化应用

这是目前更接近实际可运行产品的版本，主要面向个人本地使用。

核心特点：

- 桌面端启动和使用，入口集中在 `launcher.py` / `main.py`。
- 集成 AI 对话、模型配置、本地 RAG、任务卡片、本地资源管理、历史记录等能力。
- 更适合作为早期 MVP、个人版工具、功能验证原型。

### 3.2 演进形态：ShadowLink 分层架构版本

这是产品后续平台化、服务化、多端化的方向。

核心特点：

- `shadowlink-web`：Web 前端。
- `shadowlink-electron`：桌面壳与原生能力。
- `shadowlink-server`：Java 后端，承载认证、会话、业务和网关。
- `shadowlink-ai`：Python AI 服务，承载 Agent、RAG、MCP、文件解析、工具执行。
- `deploy` / `proto`：部署与协议基础设施。

适合未来发展为多端产品、团队协作产品、私有化部署产品或更完整的个人 AI 平台。

## 4. 目标用户

### 4.1 核心用户

1. **知识工作者**
   - 需要频繁阅读、整理、检索资料。
   - 希望让 AI 帮助理解文档、总结内容、生成方案和追踪任务。

2. **开发者 / 独立产品人**
   - 需要在项目资料、代码、需求、计划之间快速切换。
   - 希望 AI 能理解本地项目上下文，辅助编码、规划、调试和文档整理。

3. **研究者 / 学生**
   - 需要管理论文、笔记、课程资料和长期学习计划。
   - 希望 AI 能成为个人知识库问答和学习助理。

4. **高频使用 AI 的个人用户**
   - 已经习惯使用大模型，但需要更强的本地资料接入、记忆、任务和工具能力。

### 4.2 潜在扩展用户

- 小团队内部知识库助手。
- 企业私有化 AI 助理。
- 本地优先的 AI 桌面工作流工具用户。
- 多 Agent 自动化工作流爱好者。

## 5. 用户痛点

当前产品试图解决以下问题：

1. **大模型不知道用户自己的资料**
   - 普通聊天工具无法直接理解本地文档、项目文件和历史资料。
   - 用户需要反复复制粘贴上下文。

2. **AI 对话和实际工作流割裂**
   - 问答、任务、文件、链接、知识库、工具分散在不同软件中。
   - AI 生成了建议，但很难继续沉淀为任务、模式或长期记忆。

3. **个人知识难以长期复用**
   - 文档很多，但检索、总结、追问成本高。
   - 历史对话和个人偏好没有形成持续积累。

4. **AI 缺少执行能力**
   - 只能回答问题，不能调用工具、打开资源、处理文件或协同多个 Agent 完成复杂任务。

5. **隐私和本地化需求**
   - 一些资料不适合上传到公网服务。
   - 用户希望能接入本地模型、本地索引和本地存储。

## 6. 产品价值主张

### 6.1 对用户的核心价值

- **更懂我**：通过历史记录、用户偏好、记忆模块和本地资料，让 AI 更贴近用户上下文。
- **更懂我的资料**：通过 RAG 和文件解析，让 AI 能检索和理解本地知识库。
- **更能办事**：通过工具调用、Agent 执行和本地资源管理，让 AI 从“回答”走向“执行”。
- **更适合长期使用**：通过任务、模式、场景、会话和记忆，把一次性对话变成持续工作台。
- **更可控**：支持本地模型、本地索引、本地数据库和可私有化部署方向。

### 6.2 与普通 AI 聊天产品的差异

| 维度 | 普通 AI 聊天 | Jarvis-Life / ShadowLink |
| --- | --- | --- |
| 使用场景 | 单次问答为主 | 长期个人工作台 |
| 上下文 | 主要依赖当前对话 | 可接入本地文件、知识库、历史和偏好 |
| 能力边界 | 以文本生成和问答为主 | 问答 + 检索 + 工具 + 任务 + Agent 协作 |
| 部署形态 | 多为云端产品 | 支持桌面、本地资源、未来私有化部署 |
| 产品目标 | 通用聊天助手 | 个人 AI 操作系统 / AI 工作流中枢 |

## 7. 当前已具备的产品功能

### 7.1 AI 聊天助手

用户可以通过桌面端与大模型进行对话，获得回答、建议、总结和内容生成。

当前能力：

- 支持多模型配置。
- 支持 OpenAI 兼容接口、DeepSeek、本地模型/Ollama 等方向。
- 支持流式响应。
- 支持聊天历史持久化。
- 支持模型设置弹窗。

用户价值：

- 把通用 AI 问答能力整合到本地工作台中。
- 用户可根据成本、速度、隐私和能力选择不同模型。

### 7.2 本地知识库 / RAG 问答

用户可以把本地文档或资料构建成知识库，通过 AI 进行检索问答。

当前能力：

- 文档采集。
- 文本切分。
- 向量化。
- FAISS 等索引方向。
- 本地 RAG 查询。
- Web/服务化版本中规划了更完整的混合检索、重排序、多查询等能力。

用户价值：

- 不用反复复制粘贴长文档。
- 能围绕自己的资料追问、总结和提炼。
- 适合项目文档、学习资料、论文、产品资料、个人笔记等场景。

### 7.3 本地资源管理

用户可以在产品中维护和打开本地文件、目录、链接等资源。

当前能力：

- 添加/展示本地资源。
- 判断资源类型。
- 打开文件、目录或网页链接。
- 检测本地目录是否可启动。
- 资源与工作模式/任务场景结合。

用户价值：

- 把“资料入口”和“AI 对话入口”放在一起。
- 降低在文件夹、浏览器、聊天窗口之间切换的成本。

### 7.4 任务与工作模式

产品中已经出现任务卡片、模式管理和任务配置能力。

当前能力：

- 任务卡片展示。
- 工作模式 ID 和配置管理。
- 任务配置持久化。
- 不同模式可绑定不同资源、配置或上下文。

用户价值：

- 用户可以围绕不同目标组织工作，如写作、开发、阅读、项目管理等。
- 让 AI 能在特定场景下提供更相关的帮助。

### 7.5 Agent 与工具调用

产品具备从普通聊天向 Agent 执行演进的基础。

当前能力：

- 桌面版有技能接口和内置技能。
- AI 服务版包含 ReAct、Plan-and-Execute、多 Agent、Brainstorm 等执行框架。
- 支持系统工具、文件工具、代码执行、本地搜索、知识库搜索、网页搜索等工具方向。
- 支持 MCP 协议适配方向。

用户价值：

- AI 不只是回答问题，还可以拆解任务、调用工具、执行步骤。
- 为后续自动化工作流和复杂任务处理打基础。

### 7.6 多 Agent 协作 / 圆桌能力

AI 服务版中已经有多 Agent、Brainstorm、Shadow Roundtable 等模块。

当前能力：

- 多角色 Agent 协作。
- Supervisor / Swarm / Hierarchical 等多 Agent 模式。
- 圆桌会话管理。
- 协作记忆。
- 头脑风暴执行器。

用户价值：

- 适合复杂决策、方案评审、头脑风暴、产品规划和多视角分析。
- 可模拟多个专业角色共同讨论，而不是单一助手视角。

### 7.7 记忆与个性化

产品中已具备记忆和偏好学习的模块基础。

当前能力：

- 聊天历史数据库。
- 短期、长期、语义、情节记忆模块。
- 用户偏好学习。
- 用户设置。
- 上下文总线。

用户价值：

- AI 可以逐步记住用户习惯、常用资料和长期目标。
- 降低重复说明背景的成本。

### 7.8 文件解析与内容理解

AI 服务版具备多格式文件处理能力。

当前能力：

### 7.9 Web 前端与桌面壳

产品已有向多端扩展的前端和 Electron 基础。

当前能力：

## 8. 典型使用场景

### 8.1 个人知识库问答

用户把文档、笔记、项目资料放入知识库，然后直接提问：

- “这个项目现在有哪些模块？”
- “帮我总结这批文档的关键结论。”
- “根据已有资料写一个产品方案。”

### 8.2 项目开发助手

用户把代码项目、需求文档、任务列表和常用链接绑定到一个工作模式中，让 AI 辅助：

- 理解项目结构。
- 生成开发计划。
- 分析 Bug。
- 写说明文档。
- 管理下一步任务。

### 8.3 学习/研究助手

用户把论文、课程资料、阅读笔记加入知识库，让 AI 辅助：

- 总结论文。
- 对比概念。
- 生成学习路线。
- 针对资料进行问答。

### 8.4 写作与创作助手

用户在创意写作或内容生产模式下，让 AI 辅助：

- 生成大纲。
- 扩写段落。
- 改写语气。
- 维护素材库。
- 多角色讨论创意方向。

### 8.5 个人事务与本地生活助理

产品中的 Jarvis 模块已经具备本地生活聚合、偏好、主动引擎等方向，未来可用于：

- 根据用户习惯给出主动建议。
- 汇总日程、资料、任务和本地信息。
- 在合适时机提醒用户下一步行动。

## 9. 产品当前成熟度判断

### 9.1 已经比较明确的部分

- 产品方向：个人 AI 工作台 / 本地知识助理 / Agent 工作流中枢。
- 核心技术路线：LLM + RAG + Agent + 工具 + 多端架构。
- 桌面端原型：已有 PyQt 一体化版本。
- 服务化架构：已有 Web、Electron、Java 后端、Python AI 服务的模块雏形。

### 9.2 仍需产品化打磨的部分

- 需要明确主线产品形态：继续强化 PyQt 单机版，还是转向 ShadowLink 分层版本。
- 需要统一命名：Jarvis-Life、ShadowLink、Jarvis、FocusFlow 等名称目前并存。
- 需要明确核心用户场景优先级：知识库、开发助手、个人生活助理、多 Agent 圆桌、任务管理等都已出现，但需要排序。
- 需要梳理从“功能模块”到“用户闭环”的路径，如导入资料 → 提问 → 生成任务 → 沉淀记忆 → 下次主动辅助。
- 需要清理历史快照、压缩包、重复模块和编码问题，降低维护成本。

## 10. 建议的产品主线

建议将产品主线收敛为：

**一个本地优先的个人 AI 工作台，先服务“知识工作 + 项目推进”，再扩展到“长期记忆 + 主动生活助理”。**

推荐优先级：

1. **AI 对话稳定可用**
   - 模型配置清晰。
   - 流式回答稳定。
   - 历史记录可查。

2. **本地知识库体验闭环**
   - 用户能导入资料。
   - 能构建索引。
   - 能基于资料问答。
   - 能看到引用来源或检索结果。

3. **工作模式 / 项目空间**
   - 每个模式对应一个目标，如“产品规划”“代码开发”“论文阅读”。
   - 每个模式绑定资料、模型、提示词、任务和历史。

4. **任务与行动沉淀**
   - AI 回答可以转成任务。
   - 任务可以关联资料、对话和截止时间。
   - 后续可以主动提醒或继续推进。

5. **Agent 工具执行**
   - 先从安全、可控的工具开始，如搜索、文件读取、知识库查询、打开资源。
   - 再扩展到代码执行、系统命令、多 Agent 协作。

6. **记忆与主动辅助**
   - 先记住用户偏好和常用上下文。
   - 再做主动建议、每日总结和场景提醒。
