7.圆桌模式架构：轻量、表演性强、能帮助决策的多角色讨论

圆桌模式不应该做成复杂的后台多 Agent 协同系统。它更像一个产品化的“可见讨论舞台”：用户能看到几个角色围绕同一个问题表达观点、产生分歧、相互补充，最后给出一个清晰结论。它需要有表演性，因为用户进入圆桌就是希望看到不同角色的性格和观点碰撞；它也必须有实用性，因为讨论不能只是热闹，最后要能帮助用户做决策或拓展想法。考虑到外接 API 慢，圆桌必须限制轮次、限制角色数、限制工具调用，默认不跑复杂规划和长链路。

圆桌建议分成两种模式：`策略决策圆桌` 和 `自由头脑风暴圆桌`。两者共用同一个前端舞台，但目标、输出和后续处理不同。策略决策圆桌服务于具体问题，例如“今天怎么安排”“要不要休息”“这个任务先做哪个”“今晚要不要继续学”；它的输出应该是一个可执行建议，用户点同意后回交给总管/Maxwell 去分发任务、写日程或回复用户。自由头脑风暴圆桌服务于思想、创意、人生想法、产品构思、故事设定等开放讨论；它的输出可以是灵感摘要、观点列表、可继续追问的问题，不默认落盘执行。

第一层是触发入口。圆桌有两种触发方式：用户主动点击“圆桌模式”，或者某个角色在私聊中推荐。推荐不是自动启动，而是一个轻量 escalation hint，例如：“这个问题可能适合圆桌讨论，要不要让 Maxwell、Mira、Leo 一起给你几个角度？”用户确认后才进入圆桌。这样普通聊天不会被拖慢，也不会突然出现多个角色插话。

第二层是模式选择。进入圆桌时先判断是 `decision` 还是 `brainstorm`。如果用户问题里有明确决策、取舍、今天安排、是否休息、要不要做某事，就进入策略决策圆桌；如果用户说“聊聊这个想法”“帮我发散一下”“我有个设定/产品/人生问题想讨论”，就进入自由头脑风暴圆桌。用户也可以手动切换。模式字段可以简单保存为 `roundtable_mode=decision|brainstorm`。

第三层是角色配置。为了速度和可读性，每次圆桌默认 3 个角色，最多 4 个，不要全员上场。角色选择可以按场景固定：
1. 日程/任务决策：Maxwell（执行与现实约束）、Mira（身心状态）、Leo（生活体验）。
2. 学习/工作策略：Maxwell（计划）、Athena（知识与方法）、Mira（压力与可持续）。
3. 生活选择：Leo（体验）、Mira（状态）、Maxwell（时间和成本）。
4. 创意头脑风暴：Athena（结构和知识）、Leo（想象和趣味）、Mira（人性和情绪）。

如果用户正在和某个角色聊天后进入圆桌，该角色默认保留为主持或第一发言者。这样产品上有连续感：不是突然换场，而是“这个角色把你带进圆桌”。

第四层是轻量讨论流程。圆桌不采用自由群聊，而采用固定脚本式流程，保证快和稳定：
1. `主持开场`：用 1-2 句话复述用户问题，说明本轮目标。
2. `第一轮观点`：每个角色各说一段，控制在 80-150 字。
3. `交锋/补充`：每个角色只允许针对别人观点补一句，控制在 50-100 字。
4. `主持总结`：给出最终建议、分歧点、可选方案。
5. `用户选择`：同意、调整、继续讨论、返回。

这样一轮圆桌最多 3-4 个角色发言 + 1 个总结，不会无限扩散。实现上可以一次 LLM 生成完整多角色剧本，也可以每个角色单独调用。考虑 API 慢，建议默认“一次 LLM 生成整轮舞台文本”，只有高级模式才逐角色调用。一次生成时要让模型严格输出 JSON：`opening, turns:[{agent_id,agent_name,stance,content}], crossfire:[...], summary, options, recommended_option`。

第五层是策略决策圆桌。这个模式的目标是“讨论出一个可执行策略”。它的输入通常包含用户当前上下文：今天日程、任务压力、心理状态、作息、用户原始问题。讨论结束后必须产出结构化 `decision_result`：
1. `recommendation`：最终建议，例如“今天休息 40 分钟后再做最重要的一件事”。
2. `rationale`：为什么这么建议。
3. `tradeoffs`：牺牲了什么、保留了什么。
4. `actions`：可执行动作，例如移动任务、添加休息、减少计划。
5. `handoff_target`：交给 Maxwell、Mira、Planner service 或只回复用户。

用户点“同意”后，圆桌本身不直接写库，而是把 `decision_result` 回交给总管/Committer。如果 actions 里包含日程改动，Maxwell 生成 pending action；如果只是建议，直接作为 agent 回复保存；如果涉及心理关怀，Mira 生成 care intervention；如果用户点“返回”，则把总结以一条普通消息回到原私聊，不做执行。

第六层是自由头脑风暴圆桌。这个模式的目标不是立刻执行，而是让用户和角色一起发散。它可以更有表演性：角色可以有风格、有比喻、有不同立场，但仍要控制轮次。输出不是 `decision_result`，而是 `brainstorm_result`：
1. `themes`：讨论出的主题。
2. `ideas`：具体想法列表。
3. `tensions`：有趣的矛盾或分歧。
4. `followup_questions`：值得继续追问的问题。
5. `save_as_memory`：用户是否想保存为灵感记录。

头脑风暴模式不默认交给总管执行，也不默认改日程。用户可以选择“保存灵感”“继续聊这个方向”“让 Maxwell 帮我把它变成计划”。只有用户主动选择转计划，才回到轻量管家团队链路。

第七层是表演性设计。圆桌的价值一半来自内容，一半来自角色感。每个角色发言都应该有明确姿态：Maxwell 冷静现实、Mira 温柔关注状态、Leo 轻快重体验、Athena 理性结构化。前端可以用座位、头像、短暂停顿、发言高亮、主持总结卡来增强舞台感。但表演性不能牺牲可读性：每个角色一段话，不要长篇独白；冲突要围绕决策点，不要空泛争吵；主持总结必须清楚。

第八层是性能策略。默认圆桌尽量一次 LLM 生成整轮，避免多次 API 调用。Prompt 里传入角色设定、用户问题、必要上下文、输出 JSON schema。禁止工具调用，除非进入策略决策圆桌前由总管预先读取了日程/心理状态/RAG 摘要。也就是说，圆桌讨论阶段只讨论，不查工具；上下文准备阶段由轻量管家完成。这样响应速度可控。

第九层是和总管团队的关系。圆桌不是默认管家链路的一部分，而是一个独立模式。默认管家可以推荐圆桌；圆桌结束后可以把结果回交默认管家。关系是：
1. `私聊 -> 推荐圆桌`：当前角色发现问题复杂，推荐进入。
2. `圆桌 -> 总管`：策略结果被用户同意后，交给 Committer/Maxwell 执行。
3. `圆桌 -> 私聊`：用户返回时，把总结作为一条消息带回原角色。
4. `头脑风暴 -> 计划`：用户要求把灵感落地时，再交给 Maxwell 或 Athena。

第十层是数据结构。圆桌只需要简单表，不需要复杂 team task：
1. `roundtable_sessions`：保存 `id, mode(decision|brainstorm), source_session_id, source_agent_id, title, user_prompt, status, created_at, updated_at`。
2. `roundtable_turns`：保存每个角色发言 `session_id, agent_id, speaker_name, turn_type(opening|opinion|crossfire|summary), content, sort_order, created_at`。
3. `roundtable_results`：保存结构化结果 `session_id, result_type, result_json, user_choice, handoff_status, created_at`。

如果现有项目已经有 `roundtable_sessions / roundtable_turns`，可以复用，只新增 `mode`、`result_json`、`handoff_status` 等字段。

第十一层是接口设计：
1. `POST /roundtable/start`：创建圆桌，传入 `mode`、用户问题、来源 agent/session。
2. `POST /roundtable/{id}/run`：生成一轮固定脚本讨论。
3. `POST /roundtable/{id}/continue`：用户继续追问，追加一轮短讨论。
4. `POST /roundtable/{id}/accept`：用户接受策略结果，回交总管/Maxwell。
5. `POST /roundtable/{id}/return`：用户返回私聊，把总结带回原会话。
6. `POST /roundtable/{id}/save`：头脑风暴结果保存为灵感或记忆。

第十二层是典型策略决策流程。用户和 Mira 聊天时说“我今天好累，但还有学习任务，要不要继续？”Mira 推荐圆桌。用户同意后，系统预先读取今天日程、任务压力和 mood snapshot，然后启动 `decision` 圆桌。Maxwell 说现实上还有任务但可压缩，Mira 说身体状态需要恢复，Athena 或 Leo 提出更低成本的替代方案。主持总结建议“先休息 40 分钟，再做 1 个最重要任务，其余挪到明天”。用户点同意后，结果交给 Maxwell，生成待确认日程调整卡。

第十三层是典型头脑风暴流程。用户主动进入圆桌说“我想设计一个陪伴型 AI，但不知道它应该有什么气质”。系统启动 `brainstorm` 圆桌。Athena 从产品结构说，Mira 从情感安全说，Leo 从体验趣味说。最后总结出几个方向和追问问题。用户可以继续讨论，也可以点“保存灵感”。除非用户说“帮我做成计划”，否则不会进入任务落盘。

这个圆桌模式的核心原则是“轻量固定轮次、角色感强、结论清楚、执行需确认、Brainstorm 独立”。策略圆桌帮助用户做选择，结果能回交总管执行；头脑风暴圆桌帮助用户发散，默认只保存灵感不落地。这样既能保留多角色表演和思维碰撞，又不会把默认系统拖成复杂慢速多 Agent 架构。
