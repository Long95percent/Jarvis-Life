1.历史聊天记录看不到
问题描述：我在任意打开一个跟智能体的私聊后，再点击返回，并没有看到历史记录里存在一条新的记录
我的要求：只要用户点开一个聊天窗口，一旦发送文字，不必等待智能体回复，就写入历史记录，并且可以通过点击那条历史记录进入那个会话
目前链路以及问题所在：

问题拆分：
1. “历史聊天记录”指左侧/弹窗里的会话历史列表，不是单个 agent 的消息气泡历史。
2. 需求要求的触发时机是“用户发送文字后立刻入库”，不能等智能体回复成功。
3. 历史记录点击后要能恢复到同一个 agent、同一个 session，并拉取该 agent 的聊天内容。

当前链路：
1. 前端从 `shadowlink-web/src/components/jarvis/JarvisHome.tsx` 点击 agent 后调用 `useJarvisStore.openPrivateChat(agent.id)`，进入私聊页面。
2. `shadowlink-web/src/stores/jarvisStore.ts` 的 `openPrivateChat` 目前只写 `localStorage` 和 Zustand 状态：`activeAgentId / interactionMode / sessionId`，不会调用后端 `saveConversationHistory`。
3. 用户在 `shadowlink-web/src/components/jarvis/AgentChatPanel.tsx` 发送消息时，面板层会尝试调用 `jarvisApi.saveConversationHistory(...)`，但这段逻辑处在前端发送流程里，和真实后端聊天入库链路不是一个原子业务动作；如果前端提前返回、异常、路由到 Maxwell、或后端对话失败，历史列表可能没有稳定生成。
4. 后端 `shadowlink-ai/app/api/v1/jarvis_router.py` 的 `/chat` 会在进入 LLM 前调用 `save_conversation(...)`，但当前保存时机跟聊天接口绑定，前端历史列表刷新依赖 `window.dispatchEvent("jarvis:conversation-history-changed")`，而该事件现在是在完整 `jarvisApi.chat(...)` 返回后才触发。
5. 单条消息历史保存在 `shadowlink-ai/app/jarvis/persistence.py` 的 `agent_chat_turns` 表里，但该表没有 `session_id` 字段，`get_chat_history(agent_id)` 只能按 agent 拉最近消息，无法精确恢复“那条历史记录对应的那一次会话”。
6. 会话历史保存在 `conversation_history` 表里，字段包含 `session_id / agent_id / route_payload`，点击历史时 `openConversation` 会切回 `private_chat` 并调用 `loadChatHistory(conversation.agent_id)`，因此 UI 能跳回 agent，但消息列表仍是按 agent 混合加载，不是按 session 加载。

根因：
1. 会话历史的创建不是“用户发送消息”这条业务链路的第一步，而是散落在前端面板和后端聊天接口里；前端刷新事件又等后端回复后才发出，导致用户返回时看不到即时历史。
2. 聊天消息持久化缺少 `session_id`，使“历史记录 -> 对应会话消息”没有闭环，只能按 agent 粗粒度展示。
3. `openPrivateChat` 复用全局 `sessionId`，多个 agent 私聊可能共用同一 session，历史记录的边界不清晰。

解决方案（不允许只写兜底代码，必须修改实际业务链路，让它能真实运行）：

实施计划：
1. 先把“发送用户消息”拆成明确的后端业务步骤：创建/更新会话历史 -> 保存用户消息 -> 再启动 LLM 回复。
2. 改造聊天消息表和接口，让消息按 `session_id + agent_id` 存取。
3. 改造前端 store，让发送消息时先等待“用户消息已落盘”的响应并立刻刷新历史列表，再异步等待 agent 回复。
4. 改造历史点击恢复逻辑，按历史项里的 `session_id` 拉取对应消息。

具体改造：
1. 后端 `shadowlink-ai/app/jarvis/persistence.py`：
   - 给 `agent_chat_turns` 增加 `session_id TEXT` 字段，并增加索引 `idx_chat_session_agent(session_id, agent_id, timestamp)`。
   - 修改 `save_chat_turn(...)` 入参，必须支持 `session_id`；历史数据迁移时允许旧数据 `session_id` 为空。
   - 修改 `get_chat_history(...)` 支持可选 `session_id`，如果传入 session 就按 `session_id + agent_id` 查询，否则保留兼容的 agent 查询。
2. 后端 `shadowlink-ai/app/api/v1/jarvis_router.py`：
   - 在 `/chat` 开始处，确定最终 `routed_agent_id` 后，立即执行 `save_conversation(...)`，然后立即执行 `save_chat_turn(session_id=req.session_id, agent_id=routed_agent_id, role="user", content=req.message)`。
   - LLM 完成后只保存 agent 回复：`save_chat_turn(... role="agent", content=clean_reply, actions=action_results)`，不要再重复保存 user turn。
   - `/chat/{agent_id}/history` 增加 query 参数 `session_id`，传给 `get_chat_history(agent_id, session_id=session_id, limit=...)`。
   - 返回体可增加 `conversation` 或 `conversation_id`，让前端知道本轮消息已经落到哪条历史记录。
3. 前端 `shadowlink-web/src/services/jarvisApi.ts`：
   - `getChatHistory(agentId, limit, sessionId?)` 拼接 `session_id`。
   - `chat(...)` 保留原接口，但类型上接收后端返回的 `conversation_id`，便于刷新历史。
4. 前端 `shadowlink-web/src/stores/jarvisStore.ts`：
   - `openPrivateChat(agentId)` 如果是从主页新开私聊，应生成新的 `sessionId = private-${agentId}-${Date.now()}`；如果是历史记录打开，则使用历史项里的 sessionId。
   - `sendMessage(agentId, message, sessionId)` 前端可以继续先乐观追加用户气泡，但必须在调用 `/chat` 后端接口时由后端先落库用户消息；接口成功建立会话后立即 `window.dispatchEvent(new Event("jarvis:conversation-history-changed"))`，不要等 agent 回复渲染完成才刷新。
   - `loadChatHistory(agentId)` 改成 `loadChatHistory(agentId, sessionId)`，从当前 `state.sessionId` 或历史项传入 sessionId。
   - `openConversation(conversation)` 点击历史项时设置 `sessionId=conversation.session_id`，再调用 `loadChatHistory(conversation.agent_id, conversation.session_id)`。
5. 前端 `shadowlink-web/src/components/jarvis/AgentChatPanel.tsx`：
   - 移除或弱化面板层自行 `saveConversationHistory(...)` 的职责，避免前端和后端重复创建历史；历史创建统一由 `/chat` 业务接口负责。
   - 发送按钮点击后，即使 agent 回复还在等待，返回历史列表也能看到刚创建/更新的会话。
6. 验证：
   - 打开任意 agent 私聊 -> 发送“你好” -> 立刻返回 -> 历史列表出现该 agent 私聊。
   - 点击该历史项 -> 恢复到相同 agent、相同 session，并只看到这次 session 内的消息。
   - 模拟 LLM 失败 -> 历史记录和用户消息仍存在，agent 错误消息作为后续 turn 保存或前端展示。

2.智能体回复了两句。
问题描述：我跟智能体说“你好”,它回复了两条，但是我只需要一条回复，不然很乱
我的要求：每一轮对话智能体仅一条回复
目前链路以及问题所在：

问题拆分：
1. “两句”可能来自两种来源：后端一次 `/chat` 返回了两个 agent turn；或前端除后端回复外又额外生成了一条确认/提示气泡。
2. 需求要求的是“一轮用户输入只产生一条 agent 消息气泡”，工具执行结果、待确认卡片、日程卡片都应挂在这一条 agent 消息的 `actions` 上展示，而不是追加第二条 agent 文本。

当前链路：
1. 前端发送消息走 `AgentChatPanel.handleSend -> useJarvisStore.sendMessage -> jarvisApi.chat`。
2. `jarvisStore.sendMessage` 会先乐观追加一条 `{role:"user"}`，后端返回后追加一条 `{role:"agent", content: response.content, actions: response.actions}`。
3. 后端 `shadowlink-ai/app/api/v1/jarvis_router.py` 的 `/chat` 调用 `run_agent_turn(...)`，`run_agent_turn` 在 `shadowlink-ai/app/jarvis/tool_runtime.py` 中执行“两段式 LLM”：第一次让模型决定是否输出工具块；如果有工具块，执行工具后第二次调用 LLM 生成最终自然语言回复。
4. `run_agent_turn(...)` 正常只返回 `clean_final` 一段文本和 `tool_results`，后端 `/chat` 也只返回一个 `content` 字段；从后端结构看，理论上一轮只有一条 agent content。
5. 当前容易出现“双回复”的业务点有两个：
   - `AgentChatPanel.tsx` 针对 `calendar.add` / `task.plan` 等 pending action 会展示额外的确认状态文本，例如“已保存到后台任务清单，可在日历 → 查看所有任务里看到”；如果这类提示被作为单独消息渲染，用户会感知为 agent 回复两条。
   - `jarvisStore.sendMessage` 在发生路由时会同时往原 agent 和 `routedAgentId` 的 `chatHistory` 写入同一个 `agentTurn`；如果当前页面 agentId 与 routedAgentId 切换/重渲染时没有去重，可能出现同一回复被渲染两次。
6. 另外，后端 `/chat` 当前在保存历史时会保存 user 和 agent 两边；如果前端之后再 `loadChatHistory`，乐观消息和后端消息没有 turn id 去重，也可能造成刷新瞬间重复显示。

根因：
1. 系统没有“一轮对话 turn id”的概念，前端只能按数组 append，无法判断某条 agent 回复是否已经属于当前用户消息。
2. 工具确认卡片/状态提示和 agent 自然语言回复没有统一为“同一条 agent 消息的结构化附件”，UI 层容易把它们渲染成两条回复。
3. 跨 agent 路由时，同时向两个 agent 的本地历史数组写入同一 turn，缺少 active session 下的唯一消息键。

解决方案（不允许只写兜底代码，必须修改实际业务链路，让它能真实运行）：

实施计划：
1. 为每次用户发送建立唯一 `turn_id`，贯穿前端、后端、消息表和 UI 渲染。
2. 后端保证一个 `turn_id` 最多保存一条 agent 消息。
3. 前端保证一个 `turn_id` 最多渲染一条 agent 气泡，工具卡片作为该气泡的一部分。
4. 路由到其他 agent 时移动会话归属，不在两个 agent 面板同时追加同一回复。

具体改造：
1. 后端 `persistence.py`：
   - 给 `agent_chat_turns` 增加 `turn_id TEXT`、`message_id TEXT` 或至少 `client_message_id TEXT`，并增加唯一约束或业务去重索引：`UNIQUE(session_id, agent_id, turn_id, role)`。
   - `save_chat_turn(...)` 保存前按 `session_id + agent_id + turn_id + role` upsert，避免同一轮 agent 回复重复入库。
2. 后端 `jarvis_router.py`：
   - `AgentChatRequest` 增加 `turn_id`，如果前端没传就后端生成。
   - `/chat` 保存 user turn 使用这个 `turn_id`；保存 agent turn 也使用同一个 `turn_id`。
   - `run_agent_turn(...)` 的两段式 LLM仍保留，但只允许第二段 `clean_final` 作为 `content` 返回；第一段 `draft_text` 只能作为内部上下文，不能入库、不能返回给前端。
   - 对 `action_results` 保持结构化返回，不再额外生成第二条自然语言“操作成功”消息；确认/取消后的结果只更新 pending action 或 calendar/task 状态。
3. 前端 `jarvisApi.chat(...)`：
   - 发送时携带 `turn_id`。
4. 前端 `jarvisStore.sendMessage(...)`：
   - 本地消息结构增加 `turnId` 和 `messageId`。
   - 先追加 user 消息；收到后端响应时，用 `turnId` 查找当前轮是否已有 agent 消息：有则 update，没有才 append。
   - 如果 `routedAgentId !== agentId`，不要在两个历史数组里都追加完整 agent 消息；应切换 activeAgent 到 `routedAgentId` 后，只在最终归属 agent/session 下保存和展示。
5. 前端 `AgentChatPanel.tsx`：
   - pending action、日程确认、后台任务确认卡片都从 `message.actions` 渲染在同一个 agent 气泡下。
   - `confirmPending(...)`、`cancelPending(...)` 不再向 `chatHistory` 追加新的 agent 文本，只更新卡片状态和刷新日历/任务列表。
   - 如果确实需要提示“已保存”，显示为卡片内状态文案，不作为新消息。
6. 验证：
   - 输入“你好”只出现一条 agent 气泡。
   - 输入“帮我安排今晚 8 点学习”只出现一条 Maxwell 回复，下面挂待确认日程卡；确认后卡片状态变化，不新增第二条 agent 回复。
   - 跨 agent 路由时，原 agent 面板不重复显示 Maxwell 的同一条回复。

3.日程安排并没有实际落盘，我点开打开日历-查看所有任务什么都看不到，即使我让它安排了雅思学习计划
问题描述：我发现计划安排只是一个浮于表面的前端
我的要求：用户输入一句话，智能体先判断意图是否要进行日程安排，并自动判断是长期还是短期的哪类，如果是长期就在获取足够的信息后进行长期日程规划，并加上标签（长期还是短时，短时表示今天之内要做的，长期表示今天之外的）、规划周期、以及LLM给的规划安排写入数据库并落盘，同时让用户在打开日历-查看所有任务后能看到所有长期安排、短期安排。同时，日历界面不是有日视图、周视图、月视图吗，要把落盘后的任务和安排展示在这些视图上，视图上可能写不下，但是尽量展现一个标题，用户鼠标移动到上面就可以查看本日应该干什么。一个大任务，本地要记载下来LLM给的安排等基本信息（在那之前要跟LLM说给出任务周期内每天的规划），以每天的任务为一个单位保存到本地数据库，并当那天来到时，只要程序打开，凌晨一点时就把带有当天标记的细分任务推入秘书工作台的待处理列表，（如果程序没打开，就在用户第一次打开时计算，并告诉用户正在初始化）那样n个任务就都可以安排到，那天的所有安排交给秘书去协调。
如果用户当天没有确认完成任务，那就在凌晨1点修改保存在本地的规划安排的时间，并后台发送给LLM，告诉他说此任务时间少了一条但是用户什么都没做，重新安排
同时任务要支持用户自主删除，修改，确认完成

为了完成这个架构，我觉得你需要设计一个合理的多智能体存储系统和信息交换系统

目前链路以及问题所在：

问题拆分：
1. 意图识别：用户自然语言输入后，系统要判断是否是日程/计划需求，并区分短期（日内）与长期（跨天）。
2. 计划生成：短期应生成具体日程；长期应先确认必要信息，再生成任务周期内“每天的规划”。
3. 落盘：不能只显示前端卡片，必须写入本地数据库。
4. 展示：日历日/周/月视图和“查看所有任务”都要来自同一份落盘数据。
5. 运行时调度：当天凌晨 1 点或用户首次打开时，把当天细分任务推入秘书 Maxwell 工作台的待处理列表。
6. 逾期重排：当天未确认完成的细分任务，需要触发 LLM 重排并更新数据库。
7. 用户操作：任务支持删除、修改、确认完成。
8. 多智能体协作：非 Maxwell agent 识别到日程意图时要路由给 Maxwell；Maxwell 负责计划、协调、落盘和后续提醒，其他 agent 只提供上下文或建议。

当前链路：
1. 意图识别已有雏形：`shadowlink-ai/app/api/v1/jarvis_router.py` 的 `_build_schedule_intent(req.message, req.agent_id)` 会用关键词判断日程/长期任务，命中后把请求路由到 Maxwell。
2. `/chat` 中如果 LLM 没有主动调用工具，会有 fallback：短期调用 `jarvis_calendar_add`，长期调用 `jarvis_task_plan_decompose`。这说明当前系统已经试图做“兜底生成”，但这正是问题：它不是完整业务链路，很多情况下只是生成待确认 action 或粗略任务。
3. 工具执行在 `shadowlink-ai/app/jarvis/tool_runtime.py`：写操作工具会先变成 `pending_confirmation`，`/chat` 再把 pending action 保存到 `pending_actions` 表。
4. 用户确认 pending action 后，`jarvis_router.py` 的 `/pending-actions/{pending_id}/confirm`：
   - `calendar.add` 会调用 `/calendar/events` 真实写入日历。
   - `task.plan` 会调用 `save_background_task(...)` 写入 `background_tasks` 表。
5. 日历事件接口已经存在：`/calendar/events` 支持 list/add/update/delete，底层使用 `app.mcp.adapters.calendar_adapter` 管理事件；前端 `CalendarPanel.tsx` 的日/周/月视图已经用 `eventsForDay(events, day)` 渲染 `events`。
6. 后台任务接口也存在：`/background-tasks` 读取 `background_tasks` 表；前端 `CalendarPanel.tsx` 的“查看所有任务”读取 `tasks` 并展示 `task.milestones / task.subtasks / task.calendar_candidates`。
7. 现有问题在于：长期计划确认后只保存一个大任务 JSON 到 `background_tasks`，没有把“每天的任务”拆成可调度、可完成、可延期的本地实体，也没有同步生成每天的 calendar events，所以日/周/月视图只看 `calendar_events` 时看不到长期计划。
8. `background_tasks` 目前只有 `milestones / subtasks / calendar_candidates` JSON 字段，没有独立的 daily task 表、完成状态、计划日期、逾期重排状态，也没有凌晨 1 点扫描器。
9. 前端“查看所有任务”只展示后台任务列表（后台任务列表也看不到，现在没有任何显示），没有把短期 `calendar_events` 合并进任务清单，也没有对长期任务提供删除、修改、完成的真实接口。
10. 当前确认流程要求用户点 pending action；如果用户说“安排雅思学习计划”后只是看到 agent 回复但没有确认卡片或没有点确认，就不会真正写入 `background_tasks` 或 `calendar_events`，这也要求系统能正确弹出确认卡片。

根因：
1. “日程计划”被拆成了日历事件、pending action、后台任务三个松散模块，缺少统一的计划领域模型。
2. 长期任务只保存大 JSON，不保存按天拆分的可执行任务，所以日历视图和秘书工作台无法消费。
3. 短期/长期判断依赖关键词和 fallback，缺少 LLM 结构化意图输出和必要信息补齐状态机。
4. 没有每日调度器与逾期重排任务，任务不会在当天自动进入秘书待处理列表。
5. 前端任务列表与日历视图的数据源不统一，导致“查看所有任务”和日/周/月视图各看各的。

解决方案（不允许只写兜底代码，必须修改实际业务链路，让它能真实运行）：

实施计划：
1. 建立统一计划领域模型：`plans` 表保存大任务，`plan_days` 表保存每天安排，`workbench_items` 表保存秘书待处理项。
2. 改造 Maxwell 聊天链路：意图识别 -> 信息补齐 -> LLM 结构化规划 -> 保存计划 -> 生成日历投影 -> 返回同一条 agent 回复与确认卡片。

4.整体架构描述：从“聊天”到“计划落盘”再到“秘书执行”的闭环

这个架构的核心目标不是再给前端补几个兜底按钮，而是把 Jarvis 的聊天、会话历史、日程计划、多智能体协作和秘书工作台整理成一条真实可运行的业务链路。用户看到的是“我和某个智能体说一句话”，但系统内部必须把这句话拆成几个明确阶段：先识别这是普通聊天还是任务/日程意图，再决定由哪个 agent 负责，再把用户消息和会话历史落盘，再让 LLM 生成回复或结构化计划，最后把计划写入数据库，并通过日历、任务清单、秘书工作台持续消费这些数据。

整体可以分成六层：前端交互层、会话层、Agent 编排层、计划领域层、日历投影层、后台调度层。前端交互层负责让用户进入私聊、发送消息、查看历史、查看日历和任务；会话层负责保证每一次用户发送都能形成可恢复的 conversation 和 session；Agent 编排层负责判断意图、路由 agent、调用 LLM 和工具；计划领域层负责把 LLM 生成的长期/短期计划变成数据库里的真实业务对象；日历投影层负责把这些业务对象展示到日/周/月视图；后台调度层负责在每天凌晨或用户启动时把当天任务推送给 Maxwell，并处理未完成任务的重排。

第一条主链路是会话链路。用户从主页点击某个 agent，只是进入私聊状态；真正创建历史记录的时机应该是用户第一次发送文字。前端发送时生成 `session_id` 和 `turn_id`，调用后端 `/chat`。后端在进入 LLM 之前必须先完成两个动作：写入 `conversation_history`，并把用户消息写入 `agent_chat_turns`。这样即使 LLM 还没回复、LLM 报错、用户立刻返回，历史列表里也已经有这条会话。之后 LLM 返回时，后端再用同一个 `turn_id` 写入唯一一条 agent 回复。历史列表点击某条 conversation 时，前端使用该 conversation 的 `session_id + agent_id` 拉取消息，而不是只按 agent 拉最近消息。这样“历史记录 -> 会话 -> 消息”才是闭环。

第二条主链路是单轮回复链路。每次用户输入都被视为一个 turn，这个 turn 有唯一 `turn_id`。前端可以先乐观显示用户消息，但 agent 回复必须按 `turn_id` upsert：如果该 turn 已经有 agent 消息，就更新它；没有才新增。后端的 `run_agent_turn` 可以继续采用两段式 LLM：第一段让模型判断是否需要工具，第二段在工具结果基础上生成自然语言回复。但是第一段草稿永远不能入库、不能返回给前端，只有第二段最终 `clean_final` 能成为 agent 气泡。工具结果、待确认卡片、日程卡片、任务卡片都挂在这条 agent 消息的 `actions` 字段下渲染，不再作为第二条 agent 文本追加。这样用户每发一句话，界面上最多只出现一条 agent 回复。

第三条主链路是意图路由链路。不是所有 agent 都应该直接改日程。用户可能正在和 Mira、Leo 或其他角色聊天，但只要输入内容是“帮我安排”“制定计划”“明天提醒我”“30 天雅思学习计划”这类日程/计划意图，后端就应该把它路由给 Maxwell。这里不能只靠关键词做最终判断，应该采用“规则快速命中 + LLM 结构化分类”的组合：规则先识别明显计划词，LLM 再输出 JSON，说明 `intent=schedule|none`、`horizon=short_term|long_term`、`confidence`、`missing_fields`、`reason`。如果置信度足够且信息完整，Maxwell 直接规划；如果缺少周期、截止日期、每日可用时间等关键字段，Maxwell 只追问，不落盘。这样既不会误把闲聊写成任务，也不会把真正的计划停留在自然语言回复里。

第四条主链路是计划落盘链路。计划不能只存在于 LLM 回复文本里，而要进入统一的计划领域模型。短期计划和长期计划都写入 `jarvis_plans`，区别在于 `plan_type`：今天之内的安排是 `short_term`，今天之外或跨多日的是 `long_term`。每个计划再拆成若干 `jarvis_plan_days`。短期计划通常只有一天，长期计划则按日期保存每天的标题、描述、开始时间、结束时间、状态和原始 LLM payload。这样一个“30 天雅思学习计划”不再只是 `background_tasks` 里一段大 JSON，而是一个 plan 加 30 个可执行 day。每个 day 都可以单独完成、修改、延期、重排，也可以投影到日历。

第五条主链路是确认与写入链路。涉及写操作时，系统仍然应该保留 pending action 机制，避免 agent 未经确认直接改用户日程。区别是 pending action 不能只是前端浮层，它必须保存到 `pending_actions` 表。用户确认短期日程时，后端同时写入 `calendar_events`、`jarvis_plans(plan_type=short_term)` 和对应的 `jarvis_plan_days`；用户确认长期计划时，后端写入 `jarvis_plans(plan_type=long_term)`，批量写入 `jarvis_plan_days`，并为有明确时间段的 day 生成对应 `calendar_events`。用户取消时，pending action 状态改为 `cancelled`，不写计划表。用户修改待确认卡片时，修改的是 pending action 的 arguments，再确认时用修改后的 arguments 落盘。

第六条主链路是展示链路。日历视图不能只读手工创建的 `calendar_events`，任务列表也不能只读 `background_tasks`。后端应该提供统一查询接口，比如 `GET /planner/calendar-items?start=...&end=...`，把普通日历事件和计划 day 合并成统一结构返回。前端 `CalendarPanel` 的日视图、周视图、月视图都只消费这个统一结构：格子里尽量展示标题，鼠标悬停或点击时展示完整描述、来源计划、状态、时间段。另一个接口 `GET /planner/tasks` 返回计划列表，包含短期日程、长期计划、当天待办和状态统计。这样用户打开“日历 -> 查看所有任务”时，看到的是同一套计划数据，而不是另一套和日历脱节的数据。

第七条主链路是秘书工作台链路。Maxwell 不只是聊天 agent，它还应该是计划执行协调者。每天 Asia/Shanghai 凌晨 1 点，后端调度器扫描 `jarvis_plan_days`：凡是今天的、状态还是 `pending` 的 day，都生成一条 `jarvis_workbench_items`，分配给 Maxwell，并把 day 状态改成 `pushed`。如果程序凌晨没开，下一次启动时执行补偿扫描：检查上次调度时间，如果错过了今天 1 点，就立即初始化今天的待处理项，并通过消息系统告诉用户“正在初始化今日安排”。这样长期计划不会只是躺在数据库里，而是会在每天来到时自动进入秘书待办。

第八条主链路是逾期与重排链路。每天 1 点调度器在推送今天任务之前，先检查昨天以及更早的 `plan_days`。如果某个 day 仍然是 `pending` 或 `pushed`，并且没有被用户确认完成，就标记为 `missed`。然后系统把原始计划、剩余周期、已错过的内容、用户未完成的事实发给 Maxwell/LLM，让它重新安排未来的 day。LLM 返回新的结构化计划后，后端只更新未来日期的 `jarvis_plan_days` 和关联 `calendar_events`，不覆盖已经完成或已经错过的历史记录。这样计划会随着用户真实执行情况动态变化，而不是生成一次就失效。

第九条主链路是用户操作链路。用户必须能自主删除、修改、确认完成任务。删除一个 plan 时，不应该物理删除历史，而是把 `jarvis_plans.status` 改为 `cancelled`，同时取消未来未执行的 `plan_days` 和关联 calendar events。修改某一天任务时，更新 `jarvis_plan_days`，如果它有 `calendar_event_id`，同步更新日历事件。确认完成时，把 day 状态改为 `completed`，对应 workbench item 改为 `done`，并写一条 agent event。手动延期时，更新 day 日期和日历投影，并记录重排原因。这样所有用户操作都经过 planner service，而不是前端分别改日历、改任务、改本地状态。

第十条主链路是多智能体信息交换链路。系统里可以有很多 agent，但不能让每个 agent 都直接写数据库。应该建立统一事件表 `jarvis_agent_events`，所有重要变化都写事件：`plan.created`、`day.pushed`、`day.completed`、`day.missed`、`plan.rescheduled`、`calendar.changed`。Maxwell 订阅计划和工作台事件，负责解释、协调和提醒；Mira 可以根据压力和睡眠状态提出调整建议；Leo 可以根据天气和空闲时间提出生活安排；Shadow 只观察偏好，不直接说话也不改计划。各 agent 之间交换的是事件和结构化上下文，不是互相改对方的数据表。真正能修改计划状态的只有 planner service。

最终用户体验应该是这样的：用户随便找一个 agent 说“帮我安排 30 天雅思学习计划，每天 2 小时”。系统识别为长期计划，路由 Maxwell；Maxwell 如果信息足够，就调用结构化规划工具生成 30 天安排，并返回一条回复，下面挂一个待确认计划卡。用户确认后，后端写入 plan、每天的 plan_day、必要的 calendar_event。用户打开日历，日/周/月视图能看到每天的雅思学习标题；打开“查看所有任务”，能看到这个 30 天计划及每天安排。第二天凌晨 1 点，今天的雅思任务进入 Maxwell 工作台。用户完成后点完成，状态更新；如果没完成，次日系统标记 missed，并让 Maxwell 重新安排剩余计划。整个过程从聊天到落盘、从落盘到展示、从展示到执行、从执行到重排，形成一个真实闭环。

这个架构里最重要的约束是：前端只负责展示和发起用户动作，不能承担真实业务落盘；LLM 只负责理解、生成和重排计划，不能绕过工具直接改数据；所有写操作都必须经过后端 planner service；所有可恢复状态都必须进 SQLite 数据库；所有跨 agent 协作都通过结构化事件和统一领域模型完成。只有这样，Jarvis 才不会停留在“看起来像安排了”的聊天效果，而是能真正成为会记住、会展示、会提醒、会调整的个人秘书系统。


5.心理关怀模块架构：从“情绪观测”到“长期趋势”再到“温和干预”的闭环

心理关怀模块应该建立在前面那套计划和多智能体事件架构之上，而不是单独做一个“心情建议弹窗”。它的本质是一个长期状态追踪系统：持续观察用户在聊天、作息、日程、任务完成、深夜使用、压力密度等方面留下的信号，把这些信号转成可解释的心理状态指标，再按天、周、月、年形成趋势，最后由关怀层在合适时机触发温和、克制、有边界的心理建议。这个模块不应该诊断用户，更不能替代医生；它的定位是陪伴、提醒、减压和帮助用户觉察。

整体可以分成七层：情绪采集层、行为采集层、压力评估层、心理状态聚合层、趋势分析层、关怀触发层、关怀交互层。情绪采集层从聊天内容中提取情绪；行为采集层记录用户最后操作时间、深夜使用、连续在线、打开/关闭程序时间；压力评估层读取日程密度、任务堆积、逾期任务和计划重排次数；心理状态聚合层把这些信号合成每天的 mood snapshot；趋势分析层生成周/月/年的心情波动图；关怀触发层判断是否需要弹出建议；关怀交互层由 Mira 或专门的 care agent 给出建议、陪伴、呼吸练习、休息提醒或求助提示。

第一条主链路是聊天情绪观测链路。每次用户发送消息后，除了保存到 `agent_chat_turns`，还应该进入一个轻量的 emotion analyzer。这个 analyzer 可以先用规则和词典做快速判断，例如疲惫、焦虑、愤怒、低落、开心、兴奋、无助、拖延、自责等；再在必要时调用 LLM 输出结构化 JSON：`primary_emotion`、`secondary_emotions`、`valence`、`arousal`、`stress_score`、`fatigue_score`、`risk_level`、`confidence`、`evidence_summary`。这些结果写入 `jarvis_emotion_observations`。注意这里保存的不是用户原文全文，而是摘要和结构化指标，原文仍在聊天表里，避免心理模块重复存储隐私内容。

第二条主链路是作息行为观测链路。用户在个人资料里已经能写入作息，例如 bedtime / wake。系统需要增加一个 activity tracker，记录用户最后操作程序时间、当天首次打开时间、最后关闭或最后活跃时间、深夜活跃时长、是否超过设定 bedtime。比如用户设置 23:00 睡觉，但连续多天 01:30 之后仍在聊天或安排任务，就写入 `jarvis_behavior_observations`：`observation_type=late_night_usage`、`expected_bedtime=23:00`、`actual_last_active_at=01:35`、`deviation_minutes=155`。这个信号本身不是心理结论，但它会进入疲劳和压力评估。

第三条主链路是日程压力观测链路。心理关怀不能只看用户说了什么，还要看用户被安排了什么。压力评估层需要读取 `jarvis_plan_days`、`calendar_events`、`jarvis_workbench_items` 和逾期/重排日志，计算当天日程密度、连续高压天数、任务逾期数量、未完成任务数量、休息窗口是否不足、晚上是否安排过重。比如一天内超过 8 小时任务、连续 3 天没有明显休息窗口、凌晨 1 点重排多次，都应该形成 `schedule_pressure_observation`。这些观测进入 `jarvis_stress_signals`，并能解释“为什么系统认为今天压力偏高”。

第四条主链路是每日心理快照链路。系统每天至少生成一条 `jarvis_mood_snapshots`，代表当天用户的综合状态。它不是单条聊天情绪，而是综合情绪、作息、日程、任务完成和历史基线后的结果。字段可以包括：`date`、`mood_score`、`stress_score`、`energy_score`、`sleep_risk_score`、`schedule_pressure_score`、`dominant_emotions`、`positive_events`、`negative_events`、`risk_flags`、`summary`、`confidence`。比如用户当天说话偏焦虑、日程很满、昨天任务未完成且凌晨还在使用程序，那么当天 snapshot 的 `stress_score` 会升高，`energy_score` 会降低，`risk_flags` 可能包含 `late_night_overuse` 和 `schedule_overload`。

第五条主链路是长期趋势链路。月度/年度心情波动图不应该前端临时算，而应该由后端提供聚合接口。后端从 `jarvis_mood_snapshots` 聚合出 `GET /care/trends?range=month|year`，返回每天/每周的 mood、stress、energy、sleep risk 和 schedule pressure。前端可以画折线图、热力图或日历热度图：横轴是日期，纵轴是心情/压力分数，颜色代表风险等级。点击某一天可以展开“这天为什么压力高”：比如“当天 5 个任务、2 个逾期、最后活跃时间 01:42、聊天中多次表达焦虑”。这样用户看到的不是冷冰冰的分数，而是可解释的自我观察。

第六条主链路是关怀触发链路。关怀层不应该每次情绪低落都打扰用户，而是使用触发规则和冷却时间。典型触发条件包括：连续 3 天 stress_score 高于阈值；连续 2 天超过 bedtime 90 分钟仍活跃；当天日程密度过高且休息窗口不足；用户聊天中出现强烈自责、无助、崩溃表达；任务连续 miss 导致重排；月度趋势显示心情持续下降。触发后写入 `jarvis_care_triggers`，包含 `trigger_type`、`severity`、`reason`、`evidence_ids`、`cooldown_until`、`status`。同类关怀必须有冷却时间，例如轻度提醒 12 小时内不重复，中度关怀 24 小时内不重复，避免系统变成新的压力源。

第七条主链路是关怀建议链路。触发关怀后，不应该直接弹出长篇说教，而应该由 Mira 或 care agent 根据触发原因生成短、柔和、可执行的建议。轻度触发可以是非阻塞 toast：“你今天的任务排得有点满，要不要留 20 分钟空档？”中度触发可以是卡片：“这两天你都在睡觉时间后很久还在使用 Jarvis，我建议今晚把最后一个任务挪到明天。”强触发可以建议用户联系信任的人或专业帮助，但必须用谨慎语言，并声明系统不能替代专业心理支持。所有建议都写入 `jarvis_care_interventions`，记录是否展示、用户是否接受、是否关闭、是否稍后提醒。

第八条主链路是主动减负链路。心理关怀模块不能只给建议，它还应该能和 planner service 互动。比如检测到日程过紧时，Mira 不能直接改日程，但可以发起一个 `care.suggest_schedule_relief` 事件给 Maxwell。Maxwell 收到后，基于 planner service 找出可移动任务，生成一个待确认的“减负调整卡”：把某个低优先级任务挪到明天、缩短某个任务、插入休息窗口。用户确认后才真正修改 `jarvis_plan_days` 和 `calendar_events`。这样心理关怀和日程系统形成闭环：不是只安慰用户，而是帮用户把压力源结构性降低。

第九条主链路是隐私和安全边界。心理关怀涉及高度敏感数据，因此必须有明确边界。用户应能在设置里开启/关闭心理关怀、选择是否分析聊天情绪、是否记录深夜使用、是否生成长期趋势、是否允许弹窗提醒。数据库中的心理观测应尽量保存结构化指标和摘要，不额外复制完整聊天文本。高风险内容只能触发支持性建议和求助提醒，不能输出诊断结论，例如不能说“你得了抑郁症”，只能说“我注意到你最近多次表达了强烈低落和无助，如果这种状态持续，建议联系信任的人或专业支持”。

第十条主链路是多智能体分工。Mira 是心理关怀主 agent，负责情绪解释、温和建议、减压练习和关怀语气；Maxwell 负责把关怀建议转成可确认的日程调整；Shadow 负责长期偏好学习，例如用户更接受文字提醒还是卡片提醒、喜欢晚上复盘还是早晨总结；其他 agent 可以贡献上下文，但不能直接写心理状态表。所有 agent 的关怀相关动作都通过 `jarvis_agent_events` 交换，例如 `care.triggered`、`care.intervention_shown`、`care.relief_requested`、`care.relief_confirmed`。

数据表可以这样设计：
1. `jarvis_emotion_observations`：保存单次聊天或文本片段的情绪识别结果，包括 `id, session_id, turn_id, agent_id, primary_emotion, valence, arousal, stress_score, fatigue_score, risk_level, confidence, evidence_summary, created_at`。
2. `jarvis_behavior_observations`：保存用户行为信号，包括 `id, observation_type, observed_at, expected_value, actual_value, deviation_minutes, payload_json, created_at`。
3. `jarvis_stress_signals`：保存日程和任务压力信号，包括 `id, signal_type, signal_date, severity, score, reason, related_plan_id, related_day_id, payload_json, created_at`。
4. `jarvis_mood_snapshots`：保存每日综合心理状态，包括 `date, mood_score, stress_score, energy_score, sleep_risk_score, schedule_pressure_score, dominant_emotions_json, risk_flags_json, summary, confidence, created_at, updated_at`。
5. `jarvis_care_triggers`：保存触发记录，包括 `id, trigger_type, severity, reason, evidence_ids_json, status, cooldown_until, created_at, resolved_at`。
6. `jarvis_care_interventions`：保存关怀建议和用户反馈，包括 `id, trigger_id, agent_id, intervention_type, content, suggested_action_json, status, user_feedback, shown_at, acted_at, created_at`。

接口可以这样设计：
1. `GET /care/snapshot/today`：返回今天的综合状态、主要风险和建议。
2. `GET /care/trends?range=month&date=2026-04`：返回月度心情、压力、能量、作息风险曲线。
3. `GET /care/trends?range=year&year=2026`：返回年度趋势和每月摘要。
4. `GET /care/interventions?status=pending`：返回待展示或待处理的关怀建议。
5. `POST /care/interventions/{id}/ack`：用户确认已读。
6. `POST /care/interventions/{id}/snooze`：稍后提醒。
7. `POST /care/interventions/{id}/accept`：接受建议；如果涉及日程调整，则转交 planner service 生成待确认调整卡。
8. `PATCH /care/settings`：开关心理关怀、情绪分析、行为追踪、趋势图和弹窗提醒。

前端展示上可以增加一个“心理关怀 / 心情趋势”面板。首页只展示轻量摘要，例如“今天压力偏高，建议留一段休息时间”；详细页展示月度折线图、年度热力图、压力来源拆解、睡眠作息偏移、日程负荷趋势。关怀弹窗应该分级：轻度用角落 toast，中度用卡片，强风险用更明显但仍然温和的提示。任何提示都应该有“我知道了”“稍后提醒”“不要再提示此类内容”“调整今天日程”这些操作，避免让关怀变成干扰。

最终用户体验应该是这样的：用户连续几天在 23:00 作息后仍然 01:30 使用 Jarvis，同时聊天里多次表达“好累”“压力好大”，日程里又连续排满学习和工作。系统不会立刻诊断用户，而是先在后台形成 emotion observation、late night behavior observation 和 schedule pressure signal。当天或次日生成 mood snapshot，发现 stress 连续升高，于是触发 care trigger。Mira 弹出一张温和卡片：“我注意到你这两天都很晚还在处理任务，而且今天安排很满。要不要我帮你留出 20 分钟恢复时间？”如果用户接受，Maxwell 生成一个待确认日程调整卡，把低优先级任务挪开并插入休息窗口。到了月底，用户可以打开趋势图，看到哪些天压力高、原因是什么、作息是否偏移、心情是否在改善。

这个模块最重要的原则是“长期、克制、可解释、可退出”。长期意味着它不是一次性聊天建议，而是持续形成趋势；克制意味着它不会频繁打扰用户；可解释意味着每个分数和触发都有证据来源；可退出意味着用户可以随时关闭任何追踪或提醒。心理关怀模块只有和会话、日程、任务、作息、多 agent 事件系统打通，才会真正有价值：它不仅知道用户说自己累，也知道用户为什么累、什么时候更容易累、哪些安排正在加重负担，以及系统能如何温和地帮用户减轻负担。

6.轻量管家团队架构：以成熟 ReAct / LangGraph 思路为原型的可运行版本

这一版管家团队不再追求一开始就做复杂的多 Agent 公司式组织，而是先做一个轻量、稳定、响应快、容易落地的“管家内核”。它的目标是保留基础能力：角色聊天、MCP 接口、skill 使用、ReAct 工具调用、记忆读取、日程/关怀/任务落盘；同时避免过多 agent 互相讨论导致外接 API 慢、成本高、链路难 debug。复杂圆桌和 Brainstorm 先独立出去，不放进默认管家链路。默认管家模式应该像一个安静高效的单入口系统：用户说一句话，系统快速判断是否需要工具，需要就调用，不需要就直接回答。

这个架构可以参考 LangGraph 的轻量 StateGraph 和 ReAct Agent，而不是采用全自由多 Agent 群聊。核心图非常简单：`Input -> Router -> Context Loader -> Role Agent(ReAct) -> Tool/MCP/Skill Executor -> Committer -> Response`。其中 Router 只做轻量路由，Role Agent 只选一个主要角色执行，Tool Executor 统一管理 MCP 和 skill，Committer 负责把会话、计划、日程、心理观测等真实写入数据库。整个链路最多一次主 LLM 调用，只有在需要工具结果二次总结时才做第二次调用，避免外接 API 响应过慢。

第一层是统一入口。用户仍然可以从任意角色私聊、日历、任务、心理关怀卡片进入，但后台统一封装成 `jarvis_request`：`session_id, turn_id, user_message, active_agent_id, source_surface, user_confirmed_action, context_refs`。前端不需要知道管家团队怎么协作，只负责传入当前用户在哪个界面、正在和谁说话、消息是什么。后端收到请求后，第一步仍然是立即保存 conversation 和 user turn，保证历史记录可见。

第二层是轻量 Router。Router 不做复杂规划，只做四类判断：`chat` 普通聊天、`schedule` 日程/任务、`care` 心理关怀、`knowledge_or_tool` 知识/工具需求。Router 可以先用本地规则判断，只有模糊时才调用 LLM 做 JSON 分类。这样大部分简单消息不需要额外 API 调用。Router 输出字段控制在很少几个：`intent`、`target_agent`、`needs_tool`、`needs_confirmation`、`reason`。例如“你好”直接路由到当前 agent；“帮我安排明天学习”路由 Maxwell；“我最近好累”路由 Mira；“帮我总结这个文件”路由 Athena/Hermes。

第三层是角色 Agent。默认每轮只激活一个主 agent，不做多 agent 互相讨论。建议保留少量稳定角色：
1. `Maxwell`：秘书和计划执行，负责日程、任务、工作台、重排。
2. `Mira`：心理关怀，负责情绪理解、压力建议、作息提醒。
3. `Leo`：生活建议，负责活动、休息、生活安排。
4. `Athena`：知识和学习，负责 RAG、资料总结、学习辅助。
5. `Hermes`：工具执行身份，不直接聊天，只作为 Tool Executor 的内部能力名。
6. `Shadow`：偏好学习器，异步静默运行，不进入主回复链路。

这里不再设置 Auditor、复杂黑板、团队任务分派等机制。质量约束写进 system prompt 和工具 schema，由 Committer 做硬校验。比如 Maxwell 可以生成计划，但真正写库前必须经过 planner service 校验；Mira 可以生成关怀建议，但不能诊断；Hermes 可以执行工具，但所有危险操作必须确认。

第四层是 Agent 基础能力层。这里不应该只保留 ReAct，而是保留现有 agent 基础能力，只是在默认管家链路里按需选择，避免每轮都跑复杂流程。基础能力包括：ReAct 工具调用、Plan-and-Execute 分步规划、RAG 检索、长期/短期记忆读取、MCP 工具、内置 skill、结构化 action 输出、流式回复、取消执行和错误恢复。默认聊天优先走轻量 ReAct：先看用户消息和必要上下文，再判断是否需要工具；不需要工具就直接回答，需要工具就输出结构化 tool call。遇到长期计划、文件处理、复杂资料整理时，可以切换到 Plan-and-Execute，但必须由 Router 明确触发，不能所有请求默认进入复杂规划。Brainstorm 能力保留，但不进入默认管家链路，只作为用户确认后的独立圆桌模式。这里必须限制单轮工具数，例如最多 3 个，避免慢；也要限制二次 LLM，只有工具成功返回且需要自然语言解释时才进行二次总结。对于日程、任务、心理观测这类写操作，agent 只能生成 pending action 或结构化候选，不直接越权写库。

第五层是 MCP 和 Skill 接口。Tool Executor 统一管理两类工具：MCP 工具和本地 skill。MCP 用来接外部或标准化能力，例如日历、文件、搜索、浏览器、本地系统能力；skill 用来封装项目内部稳定能力，例如 `create_plan`、`analyze_emotion`、`summarize_file`、`update_calendar_event`。角色 agent 不直接知道具体实现，只看到工具 schema。这样以后替换 MCP server 或新增 skill，不需要重写 agent。工具返回必须统一成：`ok, data, error, requires_confirmation, pending_action_id, affected_resources`。

第六层是 Committer。Committer 是轻量架构里最关键的业务安全层。它不负责聊天，只负责把工具候选结果变成真实业务状态：保存会话、保存 agent 回复、保存 pending action、确认后写 planner/calendar/care 数据库、同步事件。所有写操作都从 Committer 进入，不允许 agent 直接写 SQLite。这样即使后续换成 LangGraph、AutoGen 或别的框架，业务落盘规则仍然稳定。

第七层是异步观察能力。为了速度，Shadow 偏好学习、心理长期趋势、月度统计、年度统计不应该阻塞用户当轮聊天。主回复完成后，后台异步任务再做：抽取偏好、分析情绪、更新 mood snapshot、更新日程压力信号。用户需要立即看到的只保留轻量结果，比如本轮 Mira 的关怀回复或 Maxwell 的待确认卡。长期趋势图可以由后台定时聚合，不要每次聊天都实时重算全年数据。

第八层是 Brainstorm / 圆桌模式边界。Brainstorm 不放进默认管家团队架构。它是一个独立产品模式：当用户主动点击圆桌，或者某个角色在聊天中判断“这个问题复杂、涉及多视角、需要讨论”时，才推荐用户进入圆桌。推荐方式可以是一个 escalation hint：“这个问题可能适合圆桌模式，要不要让 Maxwell、Mira、Athena 一起讨论？”用户确认后才启动 Brainstorm。默认私聊链路不自动拉多个 agent 发言，避免响应慢和回复混乱。

第八点五层是和其它模块的协作关系。轻量管家团队不是替代日程、心理关怀、历史会话、圆桌模式，而是它们的统一入口和调度层。和会话模块的关系是：所有请求先保存 conversation、session、turn，再进入 Router，保证历史可恢复。和日程模块的关系是：Maxwell 通过 planner skill 生成计划候选，Committer/Planner service 负责确认后写入 `jarvis_plans`、`jarvis_plan_days` 和日历投影。和心理关怀模块的关系是：Mira 可以读取 mood snapshot、作息偏移和日程压力信号，主链路只给即时建议，长期情绪统计交给异步 care service。和 MCP/Skill 模块的关系是：所有外部能力都挂在 Tool Executor 下，agent 只看 schema，不直接碰实现。和 Brainstorm 的关系是：默认管家只推荐圆桌，不自动启动；圆桌结束后把结论回交给 Maxwell、Mira 或 planner service 落盘。这样各模块各司其职，管家团队只负责“选谁处理、调什么能力、把结果交给哪个业务 service”。

第九层是性能策略。由于外接 API 慢，默认链路要尽量少调用模型：
1. 普通聊天：一次 LLM。
2. 明确工具请求：一次 LLM 生成工具调用 + 工具执行 + 必要时一次 LLM 总结。
3. 明确日程/任务：规则先路由 Maxwell，能用本地模板补全的先生成 pending action，复杂长期计划再调用 LLM。
4. 情绪/偏好分析：主链路不阻塞，后台异步。
5. 历史上下文：只取当前 session 最近少量 turn + 必要摘要，不把所有历史塞进 prompt。
6. 工具结果缓存：天气、日历、RAG 检索结果短时间缓存。

第十层是最小数据表。轻量版不需要复杂 team task 表，只保留必要扩展：
1. `agent_chat_turns`：增加 `session_id`、`turn_id`，保证会话恢复和单轮去重。
2. `pending_actions`：所有写操作先进入待确认。
3. `jarvis_agent_events`：保存关键事件，例如 `chat.routed`、`tool.called`、`plan.created`、`care.triggered`。
4. `tool_call_records`：记录工具调用审计。
5. 计划和心理模块继续使用前面设计的 `jarvis_plans / jarvis_plan_days / jarvis_mood_snapshots` 等领域表。

第十一层是接口设计。可以先做很少几个接口：
1. `POST /jarvis/chat`：统一聊天入口，内部完成 Router + Role Agent + ReAct + Committer。
2. `GET /jarvis/chat/{agent_id}/history?session_id=...`：按 session 恢复历史。
3. `GET /jarvis/tools`：返回当前可用 MCP/skill 工具和 schema，便于调试。
4. `POST /jarvis/pending-actions/{id}/confirm`：确认写操作。
5. `POST /jarvis/pending-actions/{id}/cancel`：取消写操作。
6. `GET /jarvis/events?session_id=...`：查看本轮路由、工具和落盘事件，便于 debug。

第十二层是技术实现建议。当前项目 `shadowlink-ai/app/agent/` 已有 ReAct、Plan-and-Execute、MultiAgent、Brainstorm 等目录，轻量管家团队可以优先实现为一个新的 `jarvis_orchestrator`，内部用很简单的状态对象：`request -> route -> context -> tool_calls -> action_results -> response`。如果后续要接成熟框架，最适合的是 LangGraph StateGraph，因为它可以把 Router、Agent、ToolExecutor、Committer 做成节点，同时保留状态可控和可测试。AutoGen/CrewAI 暂时只借鉴思想，不建议作为默认执行框架，因为默认群聊/多角色协作会增加 API 次数和不确定性。

第十三层是典型流程。用户说“你好”，Router 判断 `chat`，目标 agent 是当前私聊角色，Context Loader 取少量历史，Role Agent 一次 LLM 回复，Committer 保存 agent turn，结束。用户说“帮我安排 30 天雅思学习计划”，Router 判断 `schedule`，目标 Maxwell，Maxwell ReAct 调用 `create_plan` skill，生成待确认计划卡，Committer 保存 pending action，用户确认后 planner service 写入计划表和日历投影。用户说“最近好累还睡得晚”，Router 判断 `care`，目标 Mira，Mira 读取近期 mood 和作息信号后给出一条关怀建议，后台异步更新心理趋势。

第十四层是和圆桌模式的关系。默认管家团队解决“快速、稳定、可落盘”的日常问题；圆桌/Brainstorm 解决“复杂、开放、需要多视角讨论”的问题。二者不混在一起。默认管家可以推荐圆桌，但不能擅自启动圆桌；圆桌讨论结束后，可以把最终结论交给 Maxwell 或 planner service 落盘。这种边界能保证日常体验快，复杂讨论也有独立空间。

这个简化版管家团队的核心原则是“单轮一个主 agent、工具统一执行、写操作统一提交、重分析异步、圆桌独立”。它仍然具备成熟 Agent 架构的基础能力：Router、ReAct、Plan-and-Execute、RAG、MCP、Skill、记忆、工具审计、确认机制和异步观察；但不会引入过重的多 Agent 协商。这样更适合当前外接 API 较慢的现实情况，也更容易先把历史、日程、心理关怀这些真实业务链路跑通。


7.圆桌模式架构：轻量、表演性强、能帮助决策的多角色讨论

圆桌模式不应该做成复杂的后台多 Agent 协同系统。它更像一个产品化的“可见讨论舞台”：用户能看到几个角色围绕同一个问题表达观点、产生分歧、相互补充，最后给出一个清晰结论。它需要有表演性，因为用户进入圆桌就是希望看到不同角色的性格和观点碰撞；它也必须有实用性，因为讨论不能只是热闹，最后要能帮助用户做决策或拓展想法。考虑到外接 API 慢，圆桌必须限制轮次、限制角色数、限制工具调用，默认不跑复杂规划和长链路。

圆桌建议分成两种模式：`策略决策圆桌` 和 `自由头脑风暴圆桌`。两者共用同一个前端舞台，但目标、输出和后续处理不同。策略决策圆桌服务于具体问题，例如“今天怎么安排”“要不要休息”“这个任务先做哪个”“今晚要不要继续学”；它的输出应该是一个可执行建议，用户点同意后回交给总管/Maxwell 去分发任务、写日程或回复用户。自由头脑风暴圆桌服务于思想、创意、人生想法、产品构思、故事设定等开放讨论；它的输出可以是灵感摘要、观点列表、可继续追问的问题，不默认落盘执行。

第一层是触发入口。圆桌有两种触发方式：用户主动点击“圆桌模式”，或者某个角色在私聊中推荐。推荐不是自动启动，而是一个轻量 escalation hint，例如：“这个问题可能适合圆桌讨论，要不要让 Maxwell、Mira、Leo 一起给你几个角度？”用户确认后才进入圆桌。这样普通聊天不会被拖慢，也不会突然出现多个角色插话。

第二层是模式选择。进入圆桌时先判断是 `decision` 还是 `brainstorm`。如果用户问题里有明确决策、取舍、今天安排、是否休息、要不要做某事，就进入策略决策圆桌；如果用户说“聊聊这个想法”“帮我发散一下”“我有个设定/产品/人生问题想讨论”，就进入自由头脑风暴圆桌。用户也可以手动切换。模式字段可以简单保存为 `roundtable_mode=decision|brainstorm`。

第三层是角色配置。为了速度和可读性，每次圆桌默认 3 个角色，最多 4 个，不要全员上场。角色选择可以按场景固定：
1. 日程/任务决策：Maxwell（执行与现实约束）、Mira（身心状态）、Leo（生活体验）。
2. 学习/工作策略：Maxwell（计划）、Athena（知识与方法）、Mira（压力与可持续）。
3. 生活选择：Leo（体验）、Mira（状态）、Maxwell（时间和成本）。
4. 创意头脑风暴：Athena（结构和知识）、Leo（想象和趣味）、Mira（人性和情绪）。

如果用户正在和某个角色聊天后进入圆桌，该角色默认保留为主持或第一发言者。这样产品上有连续感：不是突然换场，而是“这个角色把你带进圆桌”。

第四层是轻量讨论流程。圆桌不采用自由群聊，而采用固定脚本式流程，保证快和稳定：
1. `主持开场`：用 1-2 句话复述用户问题，说明本轮目标。
2. `第一轮观点`：每个角色各说一段，控制在 80-150 字。
3. `交锋/补充`：每个角色只允许针对别人观点补一句，控制在 50-100 字。
4. `主持总结`：给出最终建议、分歧点、可选方案。
5. `用户选择`：同意、调整、继续讨论、返回。

这样一轮圆桌最多 3-4 个角色发言 + 1 个总结，不会无限扩散。实现上可以一次 LLM 生成完整多角色剧本，也可以每个角色单独调用。考虑 API 慢，建议默认“一次 LLM 生成整轮舞台文本”，只有高级模式才逐角色调用。一次生成时要让模型严格输出 JSON：`opening, turns:[{agent_id,agent_name,stance,content}], crossfire:[...], summary, options, recommended_option`。

第五层是策略决策圆桌。这个模式的目标是“讨论出一个可执行策略”。它的输入通常包含用户当前上下文：今天日程、任务压力、心理状态、作息、用户原始问题。讨论结束后必须产出结构化 `decision_result`：
1. `recommendation`：最终建议，例如“今天休息 40 分钟后再做最重要的一件事”。
2. `rationale`：为什么这么建议。
3. `tradeoffs`：牺牲了什么、保留了什么。
4. `actions`：可执行动作，例如移动任务、添加休息、减少计划。
5. `handoff_target`：交给 Maxwell、Mira、Planner service 或只回复用户。

用户点“同意”后，圆桌本身不直接写库，而是把 `decision_result` 回交给总管/Committer。如果 actions 里包含日程改动，Maxwell 生成 pending action；如果只是建议，直接作为 agent 回复保存；如果涉及心理关怀，Mira 生成 care intervention；如果用户点“返回”，则把总结以一条普通消息回到原私聊，不做执行。

第六层是自由头脑风暴圆桌。这个模式的目标不是立刻执行，而是让用户和角色一起发散。它可以更有表演性：角色可以有风格、有比喻、有不同立场，但仍要控制轮次。输出不是 `decision_result`，而是 `brainstorm_result`：
1. `themes`：讨论出的主题。
2. `ideas`：具体想法列表。
3. `tensions`：有趣的矛盾或分歧。
4. `followup_questions`：值得继续追问的问题。
5. `save_as_memory`：用户是否想保存为灵感记录。

头脑风暴模式不默认交给总管执行，也不默认改日程。用户可以选择“保存灵感”“继续聊这个方向”“让 Maxwell 帮我把它变成计划”。只有用户主动选择转计划，才回到轻量管家团队链路。

第七层是表演性设计。圆桌的价值一半来自内容，一半来自角色感。每个角色发言都应该有明确姿态：Maxwell 冷静现实、Mira 温柔关注状态、Leo 轻快重体验、Athena 理性结构化。前端可以用座位、头像、短暂停顿、发言高亮、主持总结卡来增强舞台感。但表演性不能牺牲可读性：每个角色一段话，不要长篇独白；冲突要围绕决策点，不要空泛争吵；主持总结必须清楚。

第八层是性能策略。默认圆桌尽量一次 LLM 生成整轮，避免多次 API 调用。Prompt 里传入角色设定、用户问题、必要上下文、输出 JSON schema。禁止工具调用，除非进入策略决策圆桌前由总管预先读取了日程/心理状态/RAG 摘要。也就是说，圆桌讨论阶段只讨论，不查工具；上下文准备阶段由轻量管家完成。这样响应速度可控。

第九层是和总管团队的关系。圆桌不是默认管家链路的一部分，而是一个独立模式。默认管家可以推荐圆桌；圆桌结束后可以把结果回交默认管家。关系是：
1. `私聊 -> 推荐圆桌`：当前角色发现问题复杂，推荐进入。
2. `圆桌 -> 总管`：策略结果被用户同意后，交给 Committer/Maxwell 执行。
3. `圆桌 -> 私聊`：用户返回时，把总结作为一条消息带回原角色。
4. `头脑风暴 -> 计划`：用户要求把灵感落地时，再交给 Maxwell 或 Athena。

第十层是数据结构。圆桌只需要简单表，不需要复杂 team task：
1. `roundtable_sessions`：保存 `id, mode(decision|brainstorm), source_session_id, source_agent_id, title, user_prompt, status, created_at, updated_at`。
2. `roundtable_turns`：保存每个角色发言 `session_id, agent_id, speaker_name, turn_type(opening|opinion|crossfire|summary), content, sort_order, created_at`。
3. `roundtable_results`：保存结构化结果 `session_id, result_type, result_json, user_choice, handoff_status, created_at`。

如果现有项目已经有 `roundtable_sessions / roundtable_turns`，可以复用，只新增 `mode`、`result_json`、`handoff_status` 等字段。

第十一层是接口设计：
1. `POST /roundtable/start`：创建圆桌，传入 `mode`、用户问题、来源 agent/session。
2. `POST /roundtable/{id}/run`：生成一轮固定脚本讨论。
3. `POST /roundtable/{id}/continue`：用户继续追问，追加一轮短讨论。
4. `POST /roundtable/{id}/accept`：用户接受策略结果，回交总管/Maxwell。
5. `POST /roundtable/{id}/return`：用户返回私聊，把总结带回原会话。
6. `POST /roundtable/{id}/save`：头脑风暴结果保存为灵感或记忆。

第十二层是典型策略决策流程。用户和 Mira 聊天时说“我今天好累，但还有学习任务，要不要继续？”Mira 推荐圆桌。用户同意后，系统预先读取今天日程、任务压力和 mood snapshot，然后启动 `decision` 圆桌。Maxwell 说现实上还有任务但可压缩，Mira 说身体状态需要恢复，Athena 或 Leo 提出更低成本的替代方案。主持总结建议“先休息 40 分钟，再做 1 个最重要任务，其余挪到明天”。用户点同意后，结果交给 Maxwell，生成待确认日程调整卡。

第十三层是典型头脑风暴流程。用户主动进入圆桌说“我想设计一个陪伴型 AI，但不知道它应该有什么气质”。系统启动 `brainstorm` 圆桌。Athena 从产品结构说，Mira 从情感安全说，Leo 从体验趣味说。最后总结出几个方向和追问问题。用户可以继续讨论，也可以点“保存灵感”。除非用户说“帮我做成计划”，否则不会进入任务落盘。

这个圆桌模式的核心原则是“轻量固定轮次、角色感强、结论清楚、执行需确认、Brainstorm 独立”。策略圆桌帮助用户做选择，结果能回交总管执行；头脑风暴圆桌帮助用户发散，默认只保存灵感不落地。这样既能保留多角色表演和思维碰撞，又不会把默认系统拖成复杂慢速多 Agent 架构。
