4.整体架构描述：从“聊天”到“计划落盘”再到“秘书执行”的闭环

这个架构的核心目标不是再给前端补几个兜底按钮，而是把 Jarvis 的聊天、会话历史、日程计划、多智能体协作和秘书工作台整理成一条真实可运行的业务链路。用户看到的是“我和某个智能体说一句话”，但系统内部必须把这句话拆成几个明确阶段：先识别这是普通聊天还是任务/日程意图，再决定由哪个 agent 负责，再把用户消息和会话历史落盘，再让 LLM 生成回复或结构化计划，最后把计划写入数据库，并通过日历、任务清单、秘书工作台持续消费这些数据。

整体可以分成六层：前端交互层、会话层、Agent 编排层、计划领域层、日历投影层、后台调度层。前端交互层负责让用户进入私聊、发送消息、查看历史、查看日历和任务；会话层负责保证每一次用户发送都能形成可恢复的 conversation 和 session；Agent 编排层负责判断意图、路由 agent、调用 LLM 和工具；计划领域层负责把 LLM 生成的长期/短期计划变成数据库里的真实业务对象；日历投影层负责把这些业务对象展示到日/周/月视图；后台调度层负责在每天凌晨或用户启动时把当天任务推送给 Maxwell，并处理未完成任务的重排。

第一条主链路是会话链路。用户从主页点击某个 agent，只是进入私聊状态；真正创建历史记录的时机应该是用户第一次发送文字。前端发送时生成 `session_id` 和 `turn_id`，调用后端 `/chat`。后端在进入 LLM 之前必须先完成两个动作：写入 `conversation_history`，并把用户消息写入 `agent_chat_turns`。这样即使 LLM 还没回复、LLM 报错、用户立刻返回，历史列表里也已经有这条会话。之后 LLM 返回时，后端再用同一个 `turn_id` 写入唯一一条 agent 回复。历史列表点击某条 conversation 时，前端使用该 conversation 的 `session_id + agent_id` 拉取消息，而不是只按 agent 拉最近消息。这样“历史记录 -> 会话 -> 消息”才是闭环。

第二条主链路是单轮回复链路。每次用户输入都被视为一个 turn，这个 turn 有唯一 `turn_id`。前端可以先乐观显示用户消息，但 agent 回复必须按 `turn_id` upsert：如果该 turn 已经有 agent 消息，就更新它；没有才新增。后端的 `run_agent_turn` 可以继续采用两段式 LLM：第一段让模型判断是否需要工具，第二段在工具结果基础上生成自然语言回复。但是第一段草稿永远不能入库、不能返回给前端，只有第二段最终 `clean_final` 能成为 agent 气泡。工具结果、待确认卡片、日程卡片、任务卡片都挂在这条 agent 消息的 `actions` 字段下渲染，不再作为第二条 agent 文本追加。这样用户每发一句话，界面上最多只出现一条 agent 回复。

第三条主链路是意图路由链路。不是所有 agent 都应该直接改日程。用户可能正在和 Mira、Leo 或其他角色聊天，但只要输入内容是“帮我安排”“制定计划”“明天提醒我”“30 天雅思学习计划”这类日程/计划意图，后端就应该把它路由给 Maxwell。这里不能只靠关键词做最终判断，应该采用“规则快速命中 + LLM 结构化分类”的组合：规则先识别明显计划词，LLM 再输出 JSON，说明 `intent=schedule|none`、`horizon=short_term|long_term`、`confidence`、`missing_fields`、`reason`。如果置信度足够且信息完整，Maxwell 直接规划；如果缺少周期、截止日期、每日可用时间等关键字段，Maxwell 只追问，不落盘。这样既不会误把闲聊写成任务，也不会把真正的计划停留在自然语言回复里。

第四条主链路是计划落盘链路。计划不能只存在于 LLM 回复文本里，而要进入统一的计划领域模型。短期计划和长期计划都写入 `jarvis_plans`，区别在于 `plan_type`：今天之内的安排是 `short_term`，今天之外或跨多日的是 `long_term`。每个计划再拆成若干 `jarvis_plan_days`。短期计划通常只有一天，长期计划则按日期保存每天的标题、描述、开始时间、结束时间、状态和原始 LLM payload。这样一个“30 天雅思学习计划”不再只是 `background_tasks` 里一段大 JSON，而是一个 plan 加 30 个可执行 day。每个 day 都可以单独完成、修改、延期、重排，也可以投影到日历。

第五条主链路是确认与写入链路。涉及写操作时，系统仍然应该保留 pending action 机制，避免 agent 未经确认直接改用户日程。区别是 pending action 不能只是前端浮层，它必须保存到 `pending_actions` 表。用户确认短期日程时，后端同时写入 `calendar_events`、`jarvis_plans(plan_type=short_term)` 和对应的 `jarvis_plan_days`；用户确认长期计划时，后端写入 `jarvis_plans(plan_type=long_term)`，批量写入 `jarvis_plan_days`，并为有明确时间段的 day 生成对应 `calendar_events`。用户取消时，pending action 状态改为 `cancelled`，不写计划表。用户修改待确认卡片时，修改的是 pending action 的 arguments，再确认时用修改后的 arguments 落盘。

第六条主链路是展示链路。日历视图不能只读手工创建的 `calendar_events`，任务列表也不能只读 `background_tasks`。后端应该提供统一查询接口，比如 `GET /planner/calendar-items?start=...&end=...`，把普通日历事件和计划 day 合并成统一结构返回。前端 `CalendarPanel` 的日视图、周视图、月视图都只消费这个统一结构：格子里尽量展示标题，鼠标悬停或点击时展示完整描述、来源计划、状态、时间段。另一个接口 `GET /planner/tasks` 返回计划列表，包含短期日程、长期计划、当天待办和状态统计。这样用户打开“日历 -> 查看所有任务”时，看到的是同一套计划数据，而不是另一套和日历脱节的数据。

第七条主链路是秘书工作台链路。Maxwell 不只是聊天 agent，它还应该是计划执行协调者。每天 Asia/Shanghai 凌晨 1 点，后端调度器扫描 `jarvis_plan_days`：凡是今天的、状态还是 `pending` 的 day，都生成一条 `jarvis_workbench_items`，分配给 Maxwell，并把 day 状态改成 `pushed`。如果程序凌晨没开，下一次启动时执行补偿扫描：检查上次调度时间，如果错过了今天 1 点，就立即初始化今天的待处理项，并通过消息系统告诉用户“正在初始化今日安排”。这样长期计划不会只是躺在数据库里，而是会在每天来到时自动进入秘书待办。

第八条主链路是逾期与重排链路。每天 1 点调度器在推送今天任务之前，先检查昨天以及更早的 `plan_days`。如果某个 day 仍然是 `pending` 或 `pushed`，并且没有被用户确认完成，就标记为 `missed`。然后系统把原始计划、剩余周期、已错过的内容、用户未完成的事实发给 Maxwell/LLM，让它重新安排未来的 day。LLM 返回新的结构化计划后，后端只更新未来日期的 `jarvis_plan_days` 和关联 `calendar_events`，不覆盖已经完成或已经错过的历史记录。这样计划会随着用户真实执行情况动态变化，而不是生成一次就失效。

第九条主链路是用户操作链路。用户必须能自主删除、修改、确认完成任务。删除一个 plan 时，不应该物理删除历史，而是把 `jarvis_plans.status` 改为 `cancelled`，同时取消未来未执行的 `plan_days` 和关联 calendar events。修改某一天任务时，更新 `jarvis_plan_days`，如果它有 `calendar_event_id`，同步更新日历事件。确认完成时，把 day 状态改为 `completed`，对应 workbench item 改为 `done`，并写一条 agent event。手动延期时，更新 day 日期和日历投影，并记录重排原因。这样所有用户操作都经过 planner service，而不是前端分别改日历、改任务、改本地状态。

第十条主链路是多智能体信息交换链路。系统里可以有很多 agent，但不能让每个 agent 都直接写数据库。应该建立统一事件表 `jarvis_agent_events`，所有重要变化都写事件：`plan.created`、`day.pushed`、`day.completed`、`day.missed`、`plan.rescheduled`、`calendar.changed`。Maxwell 订阅计划和工作台事件，负责解释、协调和提醒；Mira 可以根据压力和睡眠状态提出调整建议；Leo 可以根据天气和空闲时间提出生活安排；Shadow 只观察偏好，不直接说话也不改计划。各 agent 之间交换的是事件和结构化上下文，不是互相改对方的数据表。真正能修改计划状态的只有 planner service。

最终用户体验应该是这样的：用户随便找一个 agent 说“帮我安排 30 天雅思学习计划，每天 2 小时”。系统识别为长期计划，路由 Maxwell；Maxwell 如果信息足够，就调用结构化规划工具生成 30 天安排，并返回一条回复，下面挂一个待确认计划卡。用户确认后，后端写入 plan、每天的 plan_day、必要的 calendar_event。用户打开日历，日/周/月视图能看到每天的雅思学习标题；打开“查看所有任务”，能看到这个 30 天计划及每天安排。第二天凌晨 1 点，今天的雅思任务进入 Maxwell 工作台。用户完成后点完成，状态更新；如果没完成，次日系统标记 missed，并让 Maxwell 重新安排剩余计划。整个过程从聊天到落盘、从落盘到展示、从展示到执行、从执行到重排，形成一个真实闭环。

这个架构里最重要的约束是：前端只负责展示和发起用户动作，不能承担真实业务落盘；LLM 只负责理解、生成和重排计划，不能绕过工具直接改数据；所有写操作都必须经过后端 planner service；所有可恢复状态都必须进 SQLite 数据库；所有跨 agent 协作都通过结构化事件和统一领域模型完成。只有这样，Jarvis 才不会停留在“看起来像安排了”的聊天效果，而是能真正成为会记住、会展示、会提醒、会调整的个人秘书系统。
