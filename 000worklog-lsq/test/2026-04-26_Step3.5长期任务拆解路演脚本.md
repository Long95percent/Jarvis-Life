# 路演脚本：Step 3.5 Maxwell 长短期任务拆解与后台任务清单

创建时间：2026-04-26（Asia/Shanghai）

## 1. 路演目标

展示 Jarvis 从“只会聊天/写日程”升级为“私人专业管家团队里的秘书中枢”：

- 用户可以用自然、不严格的语言提出长期目标或未来安排。
- Maxwell 会把复杂任务拆成后台任务计划，而不是一次性塞满日历。
- 用户确认后，计划进入后台任务清单，后续可以继续滚动安排日程。
- 用户即使不是直接找 Maxwell，而是找其它 Agent，系统也会尝试把任务规划需求转交 Maxwell。
- 所有 demo 从统一路演画像开始，便于复现、追溯和对外演示。

## 2. 演示前准备

### 2.1 启动服务

按项目当前启动方式启动：

1. Python Jarvis AI 服务。
2. Java Gateway 服务。
3. `shadowlink-web` 前端服务。

确保设置里已有可用 LLM Provider，并且 Maxwell 私聊能正常返回。

### 2.2 重置测试画像与记忆

路演前先进入“测试/路演模式”，目标是清掉平时测试产生的脏画像，并从今天开始重新记录。

当前功能尚未完全落地时，先按人工验收口径准备；后续应做成设置页按钮：

```text
[清空测试画像与记忆]
[加载路演画像：雅思备考 + 生活节律 + 暑假旅行]
[开始新的 Demo Run]
[导出本次 Demo Trace]
```

准备标准：

- 清空或归档测试产生的用户画像、偏好、协作记忆、测试日程、测试后台任务。
- 加载固定路演画像，参考 `worklog-lsq/test/2026-04-26_路演用户画像种子.md`。
- 记录一个本次演示标识，例如 `demo_run_id = demo-2026-04-26-step3-5`。
- 后续每一步 demo 都按 `demo_step_id` 记录输入、Agent、工具调用、确认动作和结果。

### 2.3 建议打开的页面

- Jarvis 私聊页面。
- 后台任务清单接口调试工具，可选。
- 浏览器开发者工具 Network 面板，可选。

## 3. Demo A：直接找 Maxwell 拆解雅思长期计划

### demo_step_id

```text
step3.5-A-maxwell-ielts-plan
```

### 用户台词

在 Maxwell 私聊框输入：

```text
我想准备雅思，目标是一个月后开始系统学习，你帮我拆一下长期计划。不用现在把每天都写进日历，先帮我形成一个后台任务计划。
```

### 预期前端效果

- Maxwell 会给出秘书视角的说明。
- 聊天区出现一张卡片：`待确认长期/后台任务计划`。
- 卡片里能看到计划标题、类型、里程碑或秘书还需要确认的问题。
- 卡片包含 `确认加入后台任务` 和 `取消` 按钮。

### 路演讲解词

```text
这里不是简单把“学雅思”变成一条日程，而是先让 Maxwell 识别它属于长期项目。
系统会生成一个可确认的后台任务计划，里面有阶段、子任务和可转成日历的候选安排。
这符合私人秘书的工作方式：先建项目，再滚动安排，不会把未来几十天的日历一次性污染掉。
```

### 用户操作

点击：

```text
确认加入后台任务
```

### 追溯检查

本步骤应能追溯到：

- 用户输入原文。
- 触发 Agent：Maxwell。
- 触发 skill：`jarvis_task_plan_decompose`。
- 生成动作类型：`task.plan`。
- 确认动作：用户点击确认。
- 写入结果：后台任务清单出现“雅思备考计划”或类似标题。

### 通过标准

- 卡片状态变成 `已加入后台任务清单`。
- 后端后台任务列表能查到该计划。

## 4. Demo B：找其它 Agent，但任务路由到 Maxwell

### demo_step_id

```text
step3.5-B-route-to-maxwell-travel
```

### 用户台词

在 Mira、Nora 或 Leo 私聊框输入：

```text
我暑假后想去日本玩，你帮我先规划一下准备事项，比如什么时候看机票酒店、什么时候做攻略。
```

### 预期前端效果

- 系统不应该让非 Maxwell Agent 直接乱写长期计划。
- 任务规划需求应转交给 Maxwell 处理。
- 最终仍出现 `待确认长期/后台任务计划` 卡片。
- 不应出现 `tool not allowed for mira` 之类错误。

### 路演讲解词

```text
这个演示展示团队分工：用户不一定知道该找谁，只要表达了任务规划需求，系统会尝试把它交给秘书 Maxwell。
其它 Agent 可以参与识别意图，但最终规划、冲突检查和后台任务登记由 Maxwell 管理。
这为后续多智能体协作打基础。
```

### 追溯检查

本步骤应能追溯到：

- 原始对话 Agent：Mira / Nora / Leo。
- 实际路由 Agent：Maxwell。
- 路由原因：长期/未来任务规划意图。
- 触发 skill：`jarvis_task_plan_decompose`。
- 写入结果：后台任务清单出现旅行准备计划。

### 通过标准

- 非 Maxwell 私聊也能触发 Maxwell 的任务规划能力。
- 出现 task plan 卡片。
- 确认后能保存后台任务。

## 5. Demo C：短期任务与长期任务的区别

### demo_step_id

```text
step3.5-C-short-vs-long-task
```

### 用户台词 1：短期任务

在 Maxwell 私聊框输入：

```text
明天下午帮我安排两个小时整理房间，时间你看日历帮我找个合适的空档。
```

### 预期效果 1

- 这类更接近明确短期日程，应优先走日程卡片。
- 用户确认后写入日历。

### 用户台词 2：长期任务

在 Maxwell 私聊框输入：

```text
我想从五月开始建立一个长期健身习惯，每周三次，每次不用太久，你先帮我拆成一个长期计划。
```

### 预期效果 2

- 这类应进入后台任务计划。
- 不应直接生成一整个月或几个月的所有日程。

### 路演讲解词

```text
Jarvis 现在开始区分“马上要进日历的具体事项”和“需要长期管理的项目”。
短期事项走日程卡片，长期目标走后台任务清单，之后再由 Maxwell 滚动安排近期日程。
```

## 7. 接口验证

### 查询后台任务

如果直接访问 Python Jarvis AI 服务：

```text
GET /api/v1/jarvis/background-tasks
```

如果通过 Java Gateway：

```text
GET /api/jarvis/background-tasks
```

### 预期返回字段

返回数组中至少应看到：

```json
{
  "id": "task_xxx",
  "title": "雅思备考计划",
  "task_type": "long_project",
  "status": "active",
  "original_user_request": "...",
  "milestones": [],
  "subtasks": [],
  "calendar_candidates": []
}
```

字段内容会根据 LLM 和用户输入有所不同，不要求逐字一致。

## 8. 异常排查口径

### 没有出现卡片，只出现普通文字

检查：

- Maxwell 是否调用了 `jarvis_task_plan_decompose`。
- LLM 是否按工具调用格式输出。
- Maxwell 工具白名单是否包含 `jarvis_task_plan_decompose`。

### 点击确认失败

检查：

- 前端请求是否打到 `/pending-actions/{id}/confirm`。
- 后端是否返回具体错误，而不是固定 `Internal server error`。
- `background_tasks` 表是否能正常写入。

### 非 Maxwell 没有路由

检查：

- 输入是否包含明显规划/日程/长期目标关键词。
- `_should_route_to_maxwell` 是否命中。
- 是否因为浏览器或接口传参导致 `agent_id` 没有正确进入后端。

### 路演画像不一致

检查：

- 是否已清空旧测试画像和旧测试记忆。
- 是否已加载 `worklog-lsq/test/2026-04-26_路演用户画像种子.md`。
- 是否已开始新的 `demo_run_id`。
- 当前 Agent prompt 是否注入了过多旧历史。

## 9. 本次路演结论

Step 3.5 的核心不是做一个“待办列表”，而是建立私人管家团队的任务管理底座：

- Maxwell 是秘书和任务规划中枢。
- 长期目标先进入后台任务清单。
- 日程只承接近期、具体、用户确认后的安排。
- 固定路演画像能让每次测试高一致、可复现、可追溯。
- 后续可继续扩展为：任务编辑页、滚动排程、冲突检查、团队协作、记忆提取和自动关心用户。
