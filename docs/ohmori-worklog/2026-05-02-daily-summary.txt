二、背景与问题判断
--------------------------------------------------------------------

本轮优化来自圆桌模块的几个核心问题：

1. 私聊链路已经有一套比较成熟的意图识别和工具调用机制。
   如果圆桌只靠角色 prompt 聊天，会出现“私聊智能，圆桌不智能”的
   割裂感。

2. 早期圆桌工具能力是 router-centered tool prefetch：
   用户启动或继续圆桌时，router 统一识别一次意图、执行一次工具，
   再把工具结果拼进所有角色共享上下文。

3. 这种方式稳定、快、可控，但不够像真实 agent：
   Athena 不是自己判断要读文件，Nora 不是自己判断要 meal_plan，
   而是 router 先帮她们取好结果。

4. 用户明确希望优化第 6 点：
   圆桌工具决策应该升级为 agent-turn-centered tool reasoning。

5. 用户同时要求六个预设圆桌场景全覆盖，不能只优化某一个场景。

本轮设计边界：

1. 保留圆桌解耦：圆桌负责讨论、判断、总结和生成待确认动作。
2. 不把圆桌变成直接执行层。
3. 写操作仍走 pending action。
4. 工具能力复用私聊工具运行时，不另写一套平行工具系统。
5. 六个预设场景全部走同一套共享 agent-turn 工具决策入口。

--------------------------------------------------------------------

3. shadowlink-ai/tests/unit/jarvis/test_intent_router.py

核心实现：

1. 在 intent_router 中新增 plan_roundtable_intent。
   - 圆桌不另造一套意图识别规则。
   - 内部复用已有 plan_agent_intent。
   - 按参与角色扫描意图。
   - 支持 document_read 特殊意图。

2. 圆桌 start / continue 都会跑共享意图识别。

3. intent 结果被转换成圆桌可消费的上下文：
   - document_read -> 文档读取上下文。
   - call_tool -> 工具结果上下文。
   - ask_missing_slots -> 缺槽位上下文。
   - pending_confirmation -> 待确认动作上下文。

4. 写操作不会直接执行最终业务写入。
   - calendar add / update / delete 等仍生成 pending action。
   - context update、mood journal 等写操作同样不直接落业务数据。

行为示例：

1. 圆桌中用户说：
   “今晚很累，吃什么比较撑得住？”

   系统会识别 Nora 的 meal_plan 意图，执行只读/分析型工具，把结果
   作为“圆桌意图识别与工具结果”拼进上下文。

2. 圆桌中用户说：
   “明天下午 3 点提醒我复习英语 1 小时”

   系统会识别 Maxwell 的 calendar_create 意图，但只生成 pending action，
   不直接写入日历。

3. 圆桌中用户说：
   “帮我安排复习英语”

   系统会识别 calendar_create，但发现缺少 start/end，后续圆桌会围绕
   缺少槽位继续讨论，而不是静默丢掉。

能力范围：

1. 当前只支持文本类文档。
2. 支持扩展名：
   - .txt
   - .md / .markdown
   - .py
   - .json
   - .csv
   - .log
   - .yaml / .yml
   - .ts / .tsx
   - .js / .jsx
   - .java

3. 默认搜索目录：
   - settings.data_dir
   - settings.file_processing.upload_dir
   - repo_root / docs

4. 文件定位策略：
   - 从用户消息中提取 filename_query。
   - 对文件名、stem 和关键词 token 做 score。
   - 单一最佳匹配则读取。
   - 多个同分最佳匹配则返回 ambiguous。
   - 未找到则返回 not_found。

5. 读取结果拼进圆桌 prompt：
   “## 临时文档上下文”

6. Athena 已加入 file_read 工具白名单。

设计边界：

1. 目前不处理 PDF / docx / pptx 等复杂格式。
2. 不做大文件完整 RAG。
3. 默认截断 max_chars，避免圆桌 prompt 膨胀。
4. 圆桌只声明读取了注入的文件，不允许声称读取其它未提供文件。

五、从 router-centered prefetch 到 agent-turn-centered tool reasoning
--------------------------------------------------------------------

这是今天最关键的结构性优化。

优化前：

用户启动/继续圆桌
-> roundtable router 识别意图
-> router 执行工具
-> 工具结果拼进圆桌上下文
-> 各 agent 被动消费上下文发言

问题：

1. 工具调用责任在 router，不在 agent。
2. agent 不够“活”，不像自己具备工具能力。
3. Nora、Athena、Maxwell 等角色的工具判断被前置集中处理。
4. 和私聊的 run_agent_turn 动态工具能力不一致。

优化后：

轮到某个 agent 发言
-> 进入共享 run_roundtable_agent_turn
-> Jarvis 角色使用自己的 tool_whitelist
-> 模型可输出工具调用
-> 工具执行
-> 工具结果回喂给 agent
-> agent 基于结果产出最终发言
-> 写操作生成 pending action

新增共享能力：

文件：
shadowlink-ai/app/jarvis/roundtable_graph.py

新增结构和函数：

1. RoundtableAgentTurnResult
   - content
   - tool_results
   - action_results

2. run_roundtable_agent_turn
   - 统一圆桌角色回合执行入口。
   - Jarvis 角色可启用工具。
   - 非 Jarvis / brainstorm 角色可禁用工具。
   - 内部复用私聊 tool_runtime.run_agent_turn。

3. _persist_roundtable_agent_actions
   - 将角色回合中产生的 pending_confirmation action 保存到 pending_actions。
   - 不直接写日历、计划或关怀数据。

4. roundtable_content_deltas
   - 保持前端仍能收到 role_delta。
   - 因 run_agent_turn 返回完整内容，为了不破坏圆桌舞台流式事件，
     将最终内容切成至少两个 delta 发出。

--------------------------------------------------------------------

新增核心结构：

1. RoundtableProtocolPhase
   - id
   - title
   - owner_agent
   - objective
   - role_instructions

2. ScenarioProtocol
   - scenario_id
   - mode
   - phases
   - tool_policy
   - result_contract
   - safety_rules
   - handoff_target

六个场景协议：

1. schedule_coord
   context_scan -> conflict_check -> role_proposal -> crossfire -> alfred_decision

2. study_energy_decision
   energy_gate -> task_value_check -> minimum_viable_study -> crossfire -> decision

3. local_lifestyle
   collect_constraints -> discover_candidates -> enrich_candidates -> feasibility_score
   -> energy_filter -> rank_options -> plan_candidate

4. emotional_care
   safety_check -> emotional_validation -> body_support -> low_stimulation_options -> care_summary

5. weekend_recharge
   recovery_goal -> available_blocks -> activity_rest_balance -> crossfire -> weekend_rhythm

6. work_brainstorm
   frame_problem -> ingest_context -> divergent_ideas -> cluster_ideas
   -> critic_review -> synthesis -> validation_plan

实现方式：

1. 六个 graph executor 仍保留各自文件和现有 SSE 事件。
2. 每个 agent turn prompt 会拼入对应 ScenarioProtocol。
3. prompt 中会明确：
   - protocol_id
   - protocol_mode
   - phase_sequence
   - current_phase
   - current_role_instruction
   - tool_policy
   - safety_rules
   - result_contract
4. summarize / finalize 也会读取协议的最终阶段和结果契约。
   result_contract 同时包含对外通用字段和场景语义字段，例如：
   - schedule_coord.calendar_adjustment_candidates
   - study_energy_decision.minimum_study_block
   - local_lifestyle.ranked_activities
   - emotional_care.low_barrier_actions
   - weekend_recharge.weekend_rhythm
   - work_brainstorm.minimum_validation_steps
5. final result context 现在会带可选追踪字段：
   - scenario_protocol_id
   - scenario_protocol_mode
   - scenario_protocol_phases
   - scenario_protocol_handoff_target
   - scenario_protocol_result_contract

边界说明：

1. schedule_coord、study_energy_decision、emotional_care、weekend_recharge
   当前是中度特化 B，不是四套完全独立状态机。
2. local_lifestyle 和 work_brainstorm 已升级为 C 方案：
   - local_lifestyle 会显式产生 collect_constraints / discover_candidates /
     enrich_candidates / feasibility_score / energy_filter / rank_options /
     plan_candidate 阶段事件。
   - work_brainstorm 会显式产生 frame_problem / ingest_context /
     divergent_ideas / cluster_ideas / critic_review / synthesis /
     validation_plan 阶段事件。
   - 两个 C 方案都会在本轮收敛后发 scenario_state，暴露场景专属中间产物。
3. C 方案仍不改变圆桌 URL、请求模型和确认边界；它只是让两个重点场景
   拥有更明确的内部状态和可展示产物。
4. crossfire 当前是轻量阶段约束：后发言角色必须回应前面至少一个
   共识、分歧、风险或约束。
5. crossfire 还不是完整多轮互相追问状态机。
6. 圆桌接口 URL、请求模型、SSE 事件名保持兼容；scenario_stage /
   scenario_state 是可选增强事件。
7. 写操作仍只走 pending action，不直接写业务数据。

十五、结论
--------------------------------------------------------------------

今天圆桌模块从“可用 MVP”向“更像真实多 Agent 协作”的方向推进了一步：

1. 圆桌不再只靠 router 前置识别工具。
2. 每个 Jarvis agent 的工具能力开始进入自己的发言回合。
3. 六个预设场景全部覆盖，并通过 ScenarioProtocol 拥有各自阶段协议。
4. work_brainstorm 和 local_lifestyle 已先行升级为 C 方案，并有前端同步展示。
5. 写操作仍保持 pending confirmation，符合解耦和安全边界。
6. 解耦接口说明已同步，后续前端或圆桌业务开发人员可以按文档理解新边界。

这次优化没有把圆桌变成复杂后台多 Agent 系统，而是在保持产品化舞台和
确认机制的前提下，让每个角色更接近“自己判断、自己使用工具、自己基于
结果发言”的体验。

附录：2026-05-03 对 5-2 圆桌改动的补充修正
--------------------------------------------------------------------

本附录记录对 5-2 圆桌前后端改动的审查后修复。修复目标是把前端展示、
SSE payload 消费方式和后端写入边界重新对齐，避免“界面接了字段，但
语义不符合后端设计”的问题。

一、修复内容

1. 圆桌 task plan 写入边界收紧。
   - 问题：`jarvis_task_plan_decompose` 在私聊链路中可能直接生成 plan /
     plan_day / calendar projection，不适合在圆桌讨论阶段直接执行。
   - 修复：圆桌 start / continue 识别到该工具时，只合成
     `pending_confirmation` action，并保存到 pending_actions。
   - 修复：角色回合内 agent-turn tool reasoning 也把
     `jarvis_task_plan_decompose` 作为 deferred confirmation tool 处理，
     不直接执行业务写入。

2. 文档读取候选仲裁修正。
   - 问题：文件候选先按 score 和路径长度排序，但 ambiguous 判断只看 score，
     导致本可用路径长度打破的同分候选仍被判为 ambiguous。
   - 修复：只有 score 和路径长度都相同的候选才算真正 ambiguous；否则选
     排序后的最优候选。

3. 圆桌前端初始模式修正。
   - 问题：前端只把 `work_brainstorm` 识别为 brainstorm，导致
     `local_lifestyle`、`emotional_care`、`weekend_recharge` 首屏按
     decision 视觉和文案渲染。
   - 修复：新增 `initialRoundtableModeForScenario`，与后端模式规则对齐：
     `schedule_coord`、`study_energy_decision` 为 decision，其余预设圆桌
     为 brainstorm。

4. `role_completed.action_results` 展示语义修正。
   - 问题：前端把 `action_results.length` 直接显示为“待确认”数量，但
     action_results 中也可能包含非 pending 的成功结果。
   - 修复：新增 `pendingActionCount`，只统计
     `pending_confirmation === true` 的动作。

5. `scenario_state.next_routes` 前端消费补齐。
   - 问题：后端已经在 `scenario_state` 中发出下一轮路线建议，但前端只保存
     payload，没有可点击入口。
   - 修复：新增 `routeButtonsFromScenarioState`，把 `next_routes` 渲染为
     “下一轮路线”按钮；点击后把后端提供的 prompt 填入用户输入框。
