三、私聊协作路由：从“切给 Maxwell”改成“当前角色私下咨询”
--------------------------------------------------------------------

背景问题：

早期私聊里，用户对某个角色提出跨领域问题时，系统容易把对话切向
Maxwell 或产生“只有 Maxwell 能路由/协调”的体验。这和用户想要的
真实感不一致：用户希望私聊 Nora、Mira、Leo、Alfred 时，也可以让
当前角色去“问一下别的角色”，但最终仍由当前角色对用户说话。

产品目标：

1. 私聊当前角色不被动切走。
2. 当前角色可以私下咨询别的角色。
3. 支持显式传话：
   - “你去问问心理师”
   - “让营养师问心理师，再让心理师问秘书”
4. 支持自动轻量判断：
   - 饮食问题可咨询 Nora
   - 情绪压力可咨询 Mira
   - 日程安排可咨询 Maxwell
   - 活动生活方式可咨询 Leo
5. 不能让协作内部动作污染用户可见聊天卡片。

实现要点：

1. app/jarvis/agent_consultation.py
   - 增加 ConsultEdge。
   - 增加 AgentConsultationResult。
   - 增加 parse_consult_edges。
   - 增加 plan_consult_edges。
   - 增加 run_agent_consultations。

2. 协作边规则：
   - 支持角色别名：
     Alfred / 管家
     Maxwell / 秘书 / 日程
     Nora / 营养师
     Mira / 心理师
     Leo / 生活顾问
   - 限制最多 2 条边，避免私聊里自动爆炸成小圆桌。
   - 禁止咨询自己。
   - 禁止咨询 Shadow。

3. jarvis_router.py
   - chat_with_agent 内调用 run_agent_consultations。
   - 咨询结果以 “## 私下咨询结果” 注入当前角色 prompt。
   - response.agent_id 保持 routed_agent_id，也就是当前私聊角色。
   - routing 不再用于把私聊切换到 Maxwell。

4. 前端 AgentChatPanel
   - 隐藏 agent.consult action 卡片。
   - 用户只看到自然回复，不看到内部咨询卡。

行为结果：

1. 私聊 Nora 问“你先去问问心理师我最近心理状态，再决定今天吃什么”
   - 后端会让 Nora 咨询 Mira。
   - 最终仍由 Nora 回复。
   - response.agent_id = nora。

2. 私聊 Mira 问“今晚很累但还要学 2 小时，帮我重新安排一下”
   - Mira 可以私下咨询 Maxwell。
   - 最终仍由 Mira 回复，不切到 Maxwell。

3. 私聊 Leo 让他安排散步，Leo 可以咨询 Maxwell 的排程意见，
   但用户仍留在 Leo 私聊体验里。

测试覆盖：

1. parse_consult_edges 支持角色别名。
2. parse_consult_edges 支持二跳传话。
3. plan_consult_edges 可按情绪、饮食、日程、活动关键词推断咨询对象。
4. 明确请求优先于自动推断。
5. 不咨询自己、不咨询 Shadow。
6. 小聊不触发咨询。
7. chat pipeline 确认咨询上下文注入最终 agent prompt。
8. 私聊 schedule intent 咨询 Maxwell 时不切换 agent。

四、私聊本地意图识别：轻量 Intent + Slot Router
--------------------------------------------------------------------

讨论背景：

用户指出，意图识别和槽位抽取应该结合使用。主流做法有：

1. Prompt + few-shot + CoT 一次性识别和抽槽。
2. 意图识别与槽位抽取拆成两步，用小模型先判意图，再用大模型抽槽。

本项目当前阶段选择了更轻量、可控、成本低的方案：

1. 不先引入复杂 intent 模型。
2. 在私聊主链路前加一层 deterministic local router。
3. 高置信、常见、工具 schema 明确的任务直接由本地规则识别。
4. 低置信或未覆盖任务仍交给角色 LLM 正常回答。

新增核心 API：

文件：
shadowlink-ai/app/jarvis/intent_router.py

核心结构：

AgentIntentDecision:

1. agent_id
2. intent
3. tool_name
4. confidence
5. slots
6. missing_slots
7. next_action
   - chat_only
   - ask_missing_slots
   - call_tool
   - pending_confirmation
8. reason

核心函数：

plan_agent_intent(agent_id, message, local_now=None)

覆盖角色与意图：

1. Maxwell
   - calendar_create -> jarvis_calendar_add
   - task_decompose -> jarvis_task_plan_decompose

2. Nora
   - meal_plan -> jarvis_meal_plan
   - caffeine_guard -> jarvis_caffeine_cutoff_guard
   - nutrition_lookup -> jarvis_nutrition_lookup

3. Mira
   - breathing_protocol -> jarvis_breathing_protocol
   - checkin_schedule -> jarvis_checkin_schedule

4. Leo
   - local_activities -> jarvis_local_activities
   - plan_activity_slot -> jarvis_plan_activity_slot

重要保护：

1. 只在当前角色工具白名单内计划工具。
2. Mira 收到“今晚吃什么比较好？”不会调用 Nora 的 meal_plan。
3. Nora 的小聊不会误触发工具。
4. 槽位不足时只让当前角色追问，不假装执行。

例子：

1. Maxwell：
   “明天下午 3 点提醒我复习英语 1 小时”
   - intent = calendar_create
   - tool = jarvis_calendar_add
   - slots.title = 复习英语
   - slots.start = 2026-05-01T15:00:00
   - slots.end = 2026-05-01T16:00:00
   - next_action = pending_confirmation

2. Nora：
   “今晚很累，吃什么比较撑得住？”
   - intent = meal_plan
   - tool = jarvis_meal_plan
   - slots.meals = ["dinner"]
   - slots.goal = stress_recovery
   - next_action = call_tool

3. Mira：
   “我焦虑得有点喘不过气”
   - intent = breathing_protocol
   - tool = jarvis_breathing_protocol
   - slots.goal = calm_down

4. Leo：
   “把散步安排进日程”
   - intent = plan_activity_slot
   - missing_slots includes start
   - next_action = ask_missing_slots

接入方式：

1. chat_with_agent 在 local_now 建好之后调用 plan_agent_intent。
2. 如果 next_action 是 call_tool / pending_confirmation：
   - 先 execute_tool_calls。
   - 把工具结果和抽取槽位注入 final LLM prompt。
   - 最终 LLM 只负责自然语言表达。
3. 如果 next_action 是 ask_missing_slots：
   - 不执行工具。
   - prompt 要求当前角色自然追问缺失字段。
4. 如果 chat_only：
   - 原流程继续。

价值：

1. 高置信工具任务少走一次 LLM。
2. 工具调度更稳定，不完全依赖模型输出 <jarvis-tool> 标签。
3. 角色工具边界更明确。
4. 为后续真正的小模型 intent router 留出接口。

测试覆盖：

1. test_intent_router.py
   - 12 passed
   - 覆盖 Maxwell / Nora / Mira / Leo / 白名单 guard / small talk。

2. test_agent_intent_pipeline.py
   - 覆盖 Nora 工具预执行。
   - 覆盖 Maxwell 待确认日程 action 保存。
   - 覆盖小聊不误调工具。

验证结果：

1. intent router + pipeline focused：
   - 15 passed
2. nearby backend regression：
   - 51 passed
3. frontend build：
   - passed

五、私聊 Agent Loop 状态判断
--------------------------------------------------------------------

今天检查了私聊执行链路，确认当前项目私聊不是完整 agent loop。

当前结构：

1. chat_with_agent 准备上下文。
2. 本地意图 router 可预执行高置信工具。
3. run_agent_turn 执行一次 LLM。
4. 如果 LLM 输出 <jarvis-tool>：
   - 执行工具。
   - 再做一次 LLM 汇总工具结果。
5. 不会循环到“模型继续决定下一步”。

因此准确描述是：

1. 不是完整 agent loop。
2. 是“单轮工具规划 + 最多一次工具执行 + 最多一次 follow-up LLM”。
3. 新增本地意图 router 后，部分任务变成：
   - 本地识别/抽槽
   - 工具执行
   - 一次 LLM 总结

延迟影响：

1. 未命中本地意图时：
   - 可能需要 2 次 LLM。
2. 命中本地意图时：
   - 通常只需要 1 次 LLM。
3. 如果还有私下咨询：
   - 每条咨询边额外一次 LLM。
   - 最多 2 条边。

工程判断：

现阶段不急于上完整 agent loop。完整 loop 会提升灵活性，但也会带来：

1. 延迟更高。
2. token 成本更高。
3. 工具重复调用风险更高。
4. 私聊体验更难控。

当前更合适的路径是：

1. 用轻量本地 router 覆盖高频稳定意图。
2. 保留单轮工具执行的可控性。
3. 后续再按工具链复杂度决定是否引入 bounded loop。

六、非圆桌私聊延迟优化：主模型 / 旁路模型 / 后台任务
--------------------------------------------------------------------

问题：

私聊端到端延迟不只来自主 LLM。原链路中还有一些非用户可见但阻塞
响应的工作：

1. 长期记忆提取。
2. 偏好学习。
3. 旧记忆压缩。
4. Shadow observe。
5. 上下文准备阶段的多份记忆/画像读取。

其中长期记忆提取本身会调用 LLM：

extract_and_save_chat_memories
-> extract_memory_candidates_with_llm
-> llm_client.chat(...)

这意味着用户明明已经拿到了可回复内容，接口仍可能因为后台记忆提取
继续等待主模型。

目标：

1. 主模型只服务用户可见对话、多 Agent、工具总结。
2. 长期记忆提取、偏好学习等后台维护任务使用旁路小模型。
3. 这些后台维护任务不阻塞用户回复。
4. 完成后如果有实际更新，通过现有 proactive message 机制通知用户。

实现：

1. 新增 app/llm/background_client.py
   - SidecarLLMClient
   - build_background_llm_client
   - get_background_llm_client

2. settings_router.py
   - llm_providers.json 增加 background_id。
   - /v1/settings/providers 返回 background_id。
   - 新增：
     POST /v1/settings/providers/{provider_id}/activate-background
   - _apply_background_provider 会刷新 background_llm_client。

3. lifespan.py
   - 启动时读取 background_id。
   - 初始化 background_llm_client。
   - PreferenceLearner 优先使用 background LLM client。

4. preference_learner.py
   - 增加 set_llm_client。
   - observe 返回 bool，表示本次是否提取并保存了新偏好。
   - 方便后台任务决定是否发完成通知。

5. jarvis_router.py
   - 新增 _run_chat_background_tasks。
   - memory_job：
     * remember_user_constraint
     * remember_tool_actions
     * extract_and_save_chat_memories(llm_client=get_background_llm_client())
     * maybe_compact_old_raw_memories
   - preference_job：
     * PreferenceLearner.observe
     * 确保 learner 使用 background LLM client。
   - 两个 job 用 asyncio.gather 并发执行。
   - chat_with_agent 中只 create_task，不 await。

6. 完成通知：
   - 如果记忆提取保存了新记忆，发 proactive message：
     “长期记忆提取已完成：本轮更新 N 条后台资料。”
   - 如果偏好学习有实际更新，发 proactive message：
     “偏好学习已完成：本轮更新 1 条后台资料。”
   - trigger 分别为：
     memory_background_complete
     preference_background_complete

前端设置入口：

1. SettingsLLM 新增“后台旁路模型”区块。
2. 用户可以从已有 Provider 中选择 Sidecar Provider。
3. ACTIVE 主模型仍用于私聊、圆桌、多 Agent 回复。
4. Sidecar Provider 用于长期记忆提取和偏好学习。
5. settings-store 增加：
   - backgroundLlmId
   - setBackgroundLlmId
6. 调用后端：
   /v1/settings/providers/{id}/activate-background

七、上下文准备并行化
--------------------------------------------------------------------

原流程：

在 chat_with_agent 内，以下三个上下文源串行 await：

1. build_collaboration_memory_prefix
2. build_bounded_memory_recall_prefix
3. build_preference_profile_prefix

问题：

这三个函数大多是数据库读取和文本拼接，互相没有依赖。串行执行会把
耗时直接相加。

改造：

使用 asyncio.gather 并行执行：

1. collaboration_text
2. memory_text
3. preference_text

保留原 prompt 拼接顺序，不改变最终语义。

受控测试数据：

测试文件：
shadowlink-ai/tests/unit/jarvis/test_chat_latency_optimization.py

测试方式：

1. monkeypatch 三个上下文函数。
2. 每个函数固定 await asyncio.sleep(0.05)。
3. 串行理论耗时约 150ms。
4. 并行理论耗时约 50ms。

实测输出：

1. memory_context = 51.2ms
2. total_ms = 62.1ms

解释：

这个测试不是线上真实延迟，而是受控结构测试。它证明：

1. 代码结构已从串行等待变成并行等待。
2. 三个独立上下文源的等待时间不再累加。
3. 实际线上收益取决于数据库大小、磁盘状态、连接池、系统负载。

--------------------------------------------------------------------
八、后台任务不阻塞响应：受控延迟测试
--------------------------------------------------------------------

测试目标：

证明长期记忆提取 / 偏好学习不再阻塞 chat_with_agent 返回。

测试方式：

1. monkeypatch extract_and_save_chat_memories。
2. 让它固定 await asyncio.sleep(0.08)，模拟 80ms 后台任务。
3. 在 chat_with_agent 返回时检查后台任务尚未完成。
4. 再等待后台任务完成，确认它确实在后台继续跑。

实测输出：

1. total_ms = 9.4ms
2. background_scheduled = 0.0ms
3. persist_final_turns = 2.7ms
4. memory_context = 2.8ms

意义：

旧结构中，这 80ms 会出现在用户响应主链路里。
新结构中，主链路只负责调度 create_task，不等待后台记忆提取完成。

边界说明：

1. 这是受控测试，不代表线上只有 9.4ms。
2. 真实线上仍会受主 LLM、工具 API、咨询 LLM、数据库和网络影响。
3. 但它可靠证明：后台任务不再结构性阻塞返回。

九、私聊流式接口
--------------------------------------------------------------------

新增：

POST /api/v1/jarvis/chat/stream

事件：

1. chat_status
   - 请求已接收。
   - 包含 agent_id / session_id。

2. chat_result
   - 完整 AgentChatResponse。

3. chat_done
   - ok。
   - elapsed_ms。
   - response_ready_ms。

4. chat_error
   - 失败时返回错误信息。

前端：

1. jarvisApi 新增 chatStream。
2. jarvisStore.sendMessage 优先调用 jarvisApi.chatStream。
3. 如果 SSE 不可用或没有 body，回退旧 chat。

当前边界：

1. 现在不是 token-by-token 流式。
2. 当前版本先把私聊接口从普通阻塞响应迁移到 SSE 生命周期响应。
3. 动态工具链路仍复用原可靠 chat_with_agent。
4. 后续如要 token 级流式，需要进一步拆分 run_agent_turn：
   - 本地意图已命中且工具已预执行时，可直接 chat_stream 输出最终回复。
   - LLM 自己决定工具的场景，仍需要先拿到完整 first_reply 才能解析工具块。

工程判断：

今天先做低风险版本是合理的：

1. 不破坏现有工具执行可靠性。
2. 前端私聊已经可以统一走 SSE。
3. 为后续 token streaming 留出接口。

十一、重要工程判断与设计取舍
--------------------------------------------------------------------

1. 这次没有把私聊做成完整 agent loop。

原因：

完整 loop 会使延迟、成本、工具重复调用和调试复杂度都上升。
当前生活管家场景更需要稳定、可预测、低延迟的常用能力。

采用方案：

1. 本地 intent router 覆盖高频确定性任务。
2. LLM 保留自然语言表达和低置信场景处理。
3. 工具执行保持最多一轮。

2. 旁路模型不是替代主模型。

主模型用途：

1. 用户可见私聊回复。
2. 圆桌发言。
3. 多 Agent 讨论。
4. 工具结果自然语言总结。

旁路模型用途：

1. 长期记忆提取。
2. 偏好学习。
3. 后续可扩展到摘要、标签、轻量分类。

3. 延迟测试是真实执行代码，但不是线上性能 benchmark。

本次测试可靠证明结构变化：

1. 串行 -> 并行。
2. 主链路 await -> create_task 后台执行。
3. 主模型调用 -> background_llm_client 调用。

但不宣称线上绝对延迟，因为真实环境还受：

1. 模型服务网络。
2. Provider 响应时间。
3. SQLite 数据量。
4. 工具 API。
5. 系统负载。
6. 是否触发私下咨询。

4. 通知策略保持克制。

只有实际保存记忆或偏好时才通知。

原因：

如果每轮都通知“后台分析完成”，会变成噪音。
本轮只在 count > 0 时发 proactive message。

A + 轻量 C。

1. jarvis_tools.py 已经很大，继续塞业务逻辑会恶化边界。

Review 验收发现的问题与二次修复：

使用 review 模式验收后，先后确认了三个需要补强的逻辑问题。
这些问题都集中在一个核心点：本地生活结果一旦进入缓存，后续所有
读取路径仍然必须尊重当前请求的半径、时间窗口和使用场景，不能因为
“缓存已经存在”就绕过业务约束。

问题 1：搜索失败 fallback 路径可能返回未按当前请求过滤的缓存结果。

原始风险：

1. 用户先用较大半径搜索，缓存了较远的活动。
2. 后续用户用较小半径搜索同类活动。
3. 如果实时 web search 失败，旧 fallback 可能直接返回缓存路径中的结果。
4. 结果可能把不符合 radius_m 的活动暴露给用户。

修复方式：

1. LocalLifeSearchService.search 内部不再把缓存行直接作为返回值。
2. 缓存行先转换为 LocalLifeSearchItem。
3. cached items 统一进入 _dedupe_and_rank。
4. web search 失败时只返回已经过滤后的 cached_items[:limit]。

新增回归测试：

1. test_local_life_service_filters_cached_fallback_when_web_search_fails
   - 先写入一个 8000m 外、仍在有效期内的缓存活动。
   - 注入会抛错的 web_search。
   - 使用 radius_m=2000 搜索。
   - 期望返回空，证明 fallback 不会绕过半径过滤。

问题 2：上下文和 proactive 缓存读取只过滤“今天以后”，没有统一应用
radius_m 和 window_days。

原始风险：

1. 私聊或圆桌只想使用未来 14 天、3km 内的机会。
2. 缓存里可能已有 20 天后或 9km 外的活动。
3. 旧的上下文注入路径直接读 list_local_life_items。
4. 结果可能把超窗口、超半径的活动放进 prompt，被角色误用。

修复方式：

1. 新增 list_cached_local_life_opportunities。
2. 该 helper 内部复用 LocalLifeSearchQuery 与 _dedupe_and_rank。
3. jarvis_router._build_local_life_context_prefix 改为调用该 helper。
4. proactive_routines.build_local_life_opportunity_hint 也改为调用该 helper。
5. cached context 和 proactive hint 现在都支持：
   - radius_m
   - window_days
   - category
   - limit

新增回归测试：

1. test_context_prefix_filters_cached_local_life_by_window_and_radius
   - 缓存近期近距离活动、远距离活动、超窗口活动。
   - 构建 roundtable/private context prefix。
   - 期望只注入近期近距离活动。

2. test_proactive_hint_filters_cached_local_life_by_window_and_radius

   - 同时写入远距离和超窗口干扰项。
   - 期望 proactive hint 只选择符合窗口和半径的机会。

问题 3：主动提醒曾经无条件拼接本地生活机会。

原始风险：

1. 早晨 brief、周复盘或低压力状态下也可能附带活动推荐。
2. 这会让 proactive message 变得像广告插入，而不是上下文相关的关怀。
3. 场景越多，用户越容易觉得 Jarvis 在“硬塞附近活动”。

1. 新增 _should_attach_local_life_hint(rule, ctx, active)。
2. 只有用户近期活跃时才考虑附加本地机会。
3. midday_appetite 仅在 stress_level 或 schedule_density >= 6.5 时附加。
4. evening_checkin 仅在 stress_level >= 8、sleep_quality <= 4 或

--------------------------------------------------------------------
十五、预设圆桌 LangGraph 深度编排迁移
--------------------------------------------------------------------

背景：

用户希望圆桌会议不只是“几个 Agent 各说一段”，而是按预设场景做
更有质量的多轮讨论。前端已有固定的圆桌入口，用户会选择某一种预设
场景进入讨论，因此改造方向不是自由生成会议流程，而是在每个预设
场景下建立清晰的会议协议。

设计目标：

1. 用户自己可以作为会议成员提出需求。
2. 每一轮都先让角色依次发言。
3. 前端在每个角色发言时流式显示气泡。
4. 一轮结束后显示纪要、共识、分歧和待确认问题。
5. 用户可以选择继续讨论，也可以直接收敛。
6. 不同预设场景保留不同参与者、目标和产物类型。
7. 后端事件协议统一，前端只按 event type 渲染，不为每个场景堆分支。

新增共享模块：

1. shadowlink-ai/app/jarvis/roundtable_graph.py
   - 定义 RoundtableGraphState。
   - 定义 RoundtableTurn。
   - 定义 RoundtableSummary。
   - 定义 RoleDirective。
   - 统一 round_started / role_started / role_delta / role_completed /
     round_summary / user_checkpoint / final_result 事件语义。

新增场景图：

   - 场景：今日日程协调。
   - 类型：decision。
   - 角色：Maxwell / Nora / Mira / Alfred。

   - 场景：本地生活方式建议。
   - 类型：brainstorm。
   - 角色：Leo / Maxwell / Nora / Alfred。

3. shadowlink-ai/app/jarvis/emotional_care_graph.py
   - 场景：情绪照护。
   - 类型：brainstorm。

   - 场景：工作头脑风暴。
   - 类型：brainstorm。

   - 目标：保留原有 brainstorm 专用角色语义，让发散、质疑、收敛
     都有明确职责。

后端接入：

1. shadowlink-ai/app/api/v1/jarvis_router.py
   - start_roundtable 对 graph-backed 场景走新图执行器。
   - continue / finalize 走同一套 graph context builder。
   - 保留 legacy path，避免一次性切掉所有旧逻辑。
   - 图执行器输出 SSE 事件，前端可边收边显示角色发言。
   - decision 场景落 decision_result。
   - brainstorm 场景落 brainstorm_result。

2. enabled agent 过滤：
   - start_roundtable 先根据用户设置过滤可用角色。
   - graph executor 不再硬编码完整角色列表。
   - 这样用户禁用某个角色后，graph 圆桌不会再偷偷调用它。

3. prior discussion 保留：
   - continue / finalize 时通过 _build_graph_round_context 带上
     session.format_for_prompt(max_chars=6000)。
   - previous_discussion 包含上一轮 agent 发言、用户反馈和 round summary。
   - 这避免“继续讨论”或“直接收敛”忘掉刚刚讨论过的内容。

   - 处理 role_completed，角色气泡完成后沉淀为 transcript。
   - 处理 round_summary，显示本轮纪要、共识、分歧和问题。
   - 处理 user_checkpoint，显示“继续讨论 / 直接收敛”控制。
   - 处理 decision_result / brainstorm_result 后清理 checkpoint 状态。

Review 验收发现的问题与修复：

1. 问题：continue / finalize 丢失上一轮 agent 讨论。
   - 风险：用户点继续或直接收敛后，图只能看到用户反馈，看不到刚刚
     各角色说了什么。
   - 修复：统一 _build_graph_round_context，加入 previous_discussion。
   - 覆盖：decision graph 和 brainstorm graph 两条路径。

2. 问题：graph executor 不尊重 disabled agents。
   - 风险：用户禁用了 Nora / Mira 等角色，legacy path 已过滤，但 graph
     场景仍使用硬编码 participant 列表。
   - 修复：所有场景 execute_round / continue_round / finalize 都接受
     participants 参数，并在 state 中传播。

3. 问题：brainstorm_result 后前端 checkpoint 未清理。
   - 风险：结果已经生成，底部仍显示“继续讨论 / 直接收敛”。
   - 修复：brainstorm_result handler 和 decision_result 一样调用
     setWaitingForCheckpoint(false)。

十八、今日总体评价
--------------------------------------------------------------------

今天的工作不是单个功能点，而是一次私聊系统的架构收敛：

1. 角色体验上，从“找 Maxwell 解决所有事”变成“当前角色可以传话、
   咨询和吸收其它角色意见”。
2. 工具调度上，从“完全依赖 LLM 自己输出工具标签”增加了可测试、
   可控的本地 intent + slot 层。
3. 延迟结构上，从“所有维护工作塞在用户响应前”改成“用户可见路径
   和后台维护路径分离”。
4. 模型资源上，从“一个主模型包办全部”变成“主模型负责体验，旁路
   小模型负责维护任务”。
5. 本地生活上，从“附近 POI 查询”扩展成“可缓存、可过滤、可跨角色
   使用的本地生活机会雷达”。
6. 圆桌上，从“角色轮流回复”升级为“按预设场景执行的图编排会议”，
   每轮有流式发言、纪要、共识、分歧、问题和用户 checkpoint。
7. 角色体系上，Athena 已经补齐为一等角色，不再只是某个场景里的
   临时名字。

这为后续继续做真实低延迟 Jarvis 私聊打了基础。下一步真正有价值的
不是继续凭感觉优化，而是补真实 benchmark，把不同私聊场景的
timing.spans、LLM call count、tool count 和 p50/p95 固化下来。
