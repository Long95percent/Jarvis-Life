四、修复一：私聊历史按 session 精确恢复
--------------------------------------------------------------------

问题：

agent_chat_turns 原来主要按 agent_id 拉历史。同一个 Agent 如果有多个
私聊 session，点击历史恢复时会把这些会话混在一起，无法精确恢复
某一条历史对话。

修复思路：

1. 历史记录不能只靠 agent_id 作为隔离边界。
2. 每次私聊必须带 session_id。
3. 保存、查询、清空都要支持 session_id + agent_id。
4. 不传 session_id 时保留旧行为，避免破坏已有调用。

实现过程：

1. 数据库层：
   - agent_chat_turns 增加 session_id。
   - 初始化时兼容旧库迁移。
   - 新增 idx_chat_session_agent 索引。

2. 持久层：
   - save_chat_turn 写入 session_id。
   - get_chat_history 支持 session_id。
   - clear_chat_history 支持 session_id。

3. API 层：
   - /api/v1/jarvis/chat 保存 user/agent turn 时传入 req.session_id。
   - /chat/{agent_id}/history 增加 session_id 查询参数。

4. 前端层：
   - jarvisApi.getChatHistory 支持 sessionId。
   - jarvisStore.loadChatHistory 按当前 session 加载。
   - AgentChatPanel 加载和清空时传当前 sessionId。
   - 从主页新开私聊时生成 private-{agentId}-{timestamp} session。

测试：

1. test_private_chat_history_can_be_scoped_by_session
2. test_clearing_private_chat_history_is_session_scoped

验收标准：

1. 同一 Agent 的两个私聊 session 不再串话。
2. 清空当前 session 不影响同 Agent 的其它 session 历史。

五、修复二：跨 Agent 路由本地 history 重复显示
--------------------------------------------------------------------

问题：

例如用户在 Mira 私聊里说“明天下午帮我安排散步”，后端会路由给
Maxwell。旧前端逻辑会把同一轮 Maxwell 回复同时写进：

1. 原 Agent：Mira 的本地 chatHistory。
2. routed Agent：Maxwell 的本地 chatHistory。

风险：

1. 同一回复重复显示。
2. 刷新前后状态不一致。
3. 原 Agent 面板短暂残留 routed Agent 的回复。

修复思路：

1. 路由后的最终归属应该是 routedAgentId。
2. 原 Agent 不应保留这轮 routed 回复。
3. routed Agent 不应沿用其它 session 的旧本地 history。
4. 切换 activeAgent 后，由组件按 sessionId 重新加载后端真实历史。

实现过程：

1. 修改 shadowlink-web/src/stores/jarvisStore.ts 的 sendMessage。
2. 普通未路由场景保持原逻辑。
3. 路由场景中：
   - 移除原 Agent 刚乐观追加的 user turn。
   - 只把当前轮 user + agent 写入最终 routedAgentId。
   - 不再把 routed Agent 回复追加到原 Agent history。
   - 切换 activeAgent 后重新加载对应 session 历史。

验收脚本结果：

1. old_mira_turns: 3
2. old_maxwell_turns: 3
3. fixed_mira_turns: 1
4. fixed_maxwell_turns: 2

验收标准：

跨 Agent 路由后，只在目标 Agent 当前 session 中显示当前轮消息；
原 Agent 不再残留同一条 routed 回复。

八、Agent Consult 私下咨询层
--------------------------------------------------------------------

目标：

实现非圆桌场景下的受控 Agent 私下咨询：

1. 用户可以明确要求一个角色去问另一个角色。
2. 支持最多两跳链式咨询。
3. Shadow 不参与可见角色咨询。
4. 最终仍由当前私聊角色回复用户，不自动进入圆桌。

示例需求：

让营养师去问问心理师最近的心理状况，再决定吃什么。

修复/实现思路：

1. 不做无限 Agent 社交网络。
2. 只处理用户明确触发的咨询。
3. 用规则解析别名和“问/咨询”类触发词。
4. 最多两跳，避免 token、延迟和行为不可控。
5. 咨询结果作为上下文注入最终主 Agent prompt。

实现过程：

1. 新增：
   - shadowlink-ai/app/jarvis/agent_consultation.py

   - Mira / 心理师 / 心理医生 / 心理咨询师 / 心理
   - Leo / 生活顾问 / 生活

3. 支持触发词：
   - 问
   - 问问
   - 问一下
   - 咨询

4. 执行策略：
   - 最多执行 2 条咨询边。

4. Prompt 回流：

4. Shadow LLM 输出 global + per-agent 偏好后能持久化。

目标：

把长期记忆从“全局池按重要性取前 N 条”升级为更接近人脑记忆的机制：

1. 有边界共享：
   - 全局稳定记忆可跨 Agent 共享。
   - Agent 私有原文不跨角色泄露。
   - 敏感内容跨角色只走摘要。
2. 有生命周期：
   - 最近记忆保留 raw。
   - 久远 raw 记忆压缩为 condensed。
   - 原 raw 归档，默认不再召回。
3. 兼顾速度和精确性：
   - 第一版不引入向量库。
   - 使用 SQLite + 规则打分。
   - 召回条数固定，不随记忆总量线性增长。

修复/实现思路：

1. 先在 SQLite schema 中补生命周期字段。
2. 主召回链路替换为 bounded memory recall。
3. 召回前做 visibility 过滤，避免串长期记忆。
4. 压缩器先做确定性摘要，保持速度快、测试稳定。
5. 旧 raw 不物理删除，改为 archived，方便恢复和审计。

实现过程：

1. 持久层：
   - jarvis_memories 增加 memory_tier、visibility、owner_agent_id、
     allowed_agent_ids、compressed_from_ids、expires_at、decay_score、
     last_accessed_at、access_count。
   - save_jarvis_memory 支持写入生命周期元数据。
   - mark_jarvis_memories_used 更新 last_accessed_at 和 access_count。
   - 新增 archive_jarvis_memories。

2. 有边界召回：
   - 新增 shadowlink-ai/app/jarvis/memory_recall.py。
   - 新增 build_bounded_memory_recall_prefix(agent_id, user_message, limit=6)。
   - 召回过滤：
       global：所有可见 Agent 可用。
       private_raw：仅 owner / allowed agent 可用。
       agent_scoped：owner / allowed agent 可用。
       sensitive_summary：只共享摘要，按 allowed_agent_ids 控制。
   - 召回打分考虑 importance、recency、agent relevance、memory_kind relevance、
     keyword overlap、tier/visibility、decay_score。
   - /jarvis/chat 从旧 build_memory_recall_prefix 切换到 bounded recall。

3. 记忆压缩：
   - 新增 shadowlink-ai/app/jarvis/memory_compactor.py。
   - compact_old_raw_memories(cutoff_days=7) 找到超过 cutoff 的 active raw 记忆。
   - 按 owner/sensitivity 分组生成 condensed summary。
   - 原 raw 标记为 archived。
   - maybe_compact_old_raw_memories 按 min_interval_seconds 节流。
   - /jarvis/chat 保存记忆后按节流触发压缩检查。

测试：

1. save_jarvis_memory 能保存生命周期元数据。
2. Nora 能召回 global 记忆和允许共享的 sensitive_summary。
3. Nora 不能召回 Mira 的 private_raw 心理原文。
4. Nora / Maxwell 召回会偏向各自角色相关记忆。
5. 旧 raw 记忆会压缩成 condensed，并归档原始 raw。
6. maybe_compact_old_raw_memories 有节流。
7. /jarvis/chat 最终 prompt 使用 bounded memory recall。

十一、异常触发主动打扰可靠性升级
--------------------------------------------------------------------

范围：

本轮只处理异常触发主动打扰。日常节律 / 固定时间 routine 当时暂不实现，
后续在第十二节单独完成。

问题：

1. 原 proactive message 只存在 ProactiveTriggerEngine._pending_messages 内存队列。
2. GET /jarvis/messages 和 SSE 都通过 pop_pending_messages() 取出即清空。
3. 服务重启、页面刷新、多客户端订阅都可能导致消息丢失或不可见。
4. 前端有 ProactiveMessageFeed 组件，但 JarvisHome 没有真正渲染。

修复思路：

1. 主动消息必须持久化。
2. GET /jarvis/messages 必须非破坏式读取。
3. SSE 只能标记 delivered，不能清空全局消息。
4. 前端本地 store 需要按 id upsert，防止 SSE / demo / 页面加载重复插入。
5. feed 要真正显示在 JarvisHome。

实现过程：

1. 后端模型：
   - ProactiveMessage 增加 priority、status、delivered_at、read_at、dismissed_at。

2. 数据库：
   - proactive_messages 表增加 id、agent_id、agent_name、content、trigger、
     priority、status、created_at、delivered_at、read_at、dismissed_at。

3. 持久化 API：
   - save_proactive_message。
   - list_proactive_messages。
   - mark_proactive_messages_delivered。
   - mark_proactive_message_read。
   - dismiss_proactive_message。

4. 引擎：
   - ProactiveTriggerEngine 命中 send_message 后写入 SQLite。

5. API：
   - GET /jarvis/messages 改为非破坏式读取。
   - POST /jarvis/messages/{message_id}/read。
   - POST /jarvis/messages/{message_id}/dismiss。
   - SSE /jarvis/messages/stream 从 SQLite 读取未读消息，并在单个 SSE 连接内按 id 去重。
   - /jarvis/proactive/fire 从持久化消息里筛本次新增消息返回。

6. 前端：
   - JarvisPage mount 时调用 loadProactiveMessages。
   - jarvisStore.addProactiveMessage 按 id upsert。
   - markMessageRead 调用后端 read endpoint 并合并返回状态。
   - JarvisHome 右侧栏渲染 ProactiveMessageFeed。

测试：

1. test_proactive_messages_are_persistent_and_non_destructive
2. test_proactive_message_status_flow_read_and_dismissed
3. test_check_triggers_persists_proactive_messages
4. test_get_proactive_messages_is_non_destructive
5. test_mark_proactive_message_read_updates_state

十二、日常节律定时打扰方案 B
--------------------------------------------------------------------

目标：

实现用户选择的方案 B：不要做过重的完整 scheduler 系统，但要比纯固定
时间更聪明一些，具备活跃判断、quiet hours、优先级调节和去重。

设计取舍：

1. 不做独立调度服务。
2. 不做配置界面。
3. 复用现有 ProactiveTriggerEngine 的 5 分钟轮询。
4. 复用 proactive_messages、SSE、feed、read/dismiss 状态流。
5. routine 固定写在代码里，后续再由偏好学习动态调整。

实现过程：

1. 新增 routine scheduler：
   - shadowlink-ai/app/jarvis/proactive_routines.py

2. 固定 4 类节律：
   - morning_brief：08:00-10:30，Maxwell 晨间提醒 / 日程检查。
   - midday_appetite：11:30-13:30，Nora 午餐 / 胃口询问。
   - evening_checkin：21:30-23:30，Mira 晚间状态确认。
   - weekly_review：周日 19:00-21:30，Alfred 周复盘。

3. 轻量策略层：
   - quiet hours：00:00-06:30 不触发 routine。
   - 用户活跃判断：最近 120 分钟内由 user / user_chat / user_ui 更新过上下文。
   - 活跃用户：normal priority，压力/日程/睡眠异常时可升为 high。
   - 非活跃用户：仍落库，但降为 low priority，等用户回来后在 feed 中看到。
   - 晚间高压力或情绪下降时，evening_checkin 升为 high priority。
   - 睡眠差或日程密集时，morning_brief 文案更温和/更偏日程保护。

4. 去重持久化：
   - 新增 proactive_routine_runs。
   - 以 routine_id + run_date 唯一约束保证同一天同一 routine 只触发一次。
   - 新增 has_proactive_routine_run。
   - 新增 save_proactive_routine_run。

5. 引擎接入：
   - ProactiveTriggerEngine 新增 routine_scheduler。
   - start() 循环中在 check_triggers() 后调用 check_routines()。
   - routine 生成的仍是 ProactiveMessage。

6. 用户活跃记录：
   - /jarvis/chat 开始时调用 LifeContextBus.update_fields({}, source="user_chat")。
   - 不改变压力、日程、睡眠等业务字段，只更新时间戳和 source_agent。
   - routine scheduler 可把真实聊天行为识别为“用户最近活跃”。

7. 旧库兼容修复：
   - 旧库 agent_chat_turns / conversation_history / collaboration_memories 可能缺
     session_id 等新列。
   - 将相关索引创建移动到列迁移之后，避免初始化时报 no such column。
   - 为 conversation_history 补 session_id / route_payload / last_opened_at。
   - 为 collaboration_memories 补 session_id。

测试：

1. test_morning_brief_fires_once_when_user_recently_active
2. test_midday_appetite_inactive_user_is_low_priority_but_persisted
3. test_evening_checkin_becomes_high_priority_when_stress_is_high
4. test_routines_do_not_fire_during_quiet_hours
5. test_proactive_routine_runs_are_recorded_once_per_day
6. test_existing_conversation_and_collaboration_tables_get_session_columns
7. test_chat_with_agent 追加断言：聊天会刷新 source_agent=user_chat

验证记录：

2. 最新 Jarvis unit tests：
   - 76 passed
3. focused integration tests：
   - 3 passed

   - passed

   - passed

1. 节律配置暂时写死。
