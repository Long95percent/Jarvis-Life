# 2026-04-26 Step3.6 主私聊记忆真实闭环 Worklog

## 本次原则

用户明确要求继续 Step 3.6，但不要再做“只服务路演的最小闭环”，也不要在功能未真实落地前写路演脚本。因此本次只做真实私聊链路：对话前召回、对话后提取、持久化保存、下一轮注入、用户可查看/删除。

## 本次完成

- 新增统一长期记忆表 `jarvis_memories`，字段包含：记忆类型、内容、来源 Agent、session_id、source_text、structured_payload、敏感等级、置信度、重要性、last_used_at、status。
- 新增持久层能力：
  - `save_jarvis_memory`
  - `list_jarvis_memories`
  - `mark_jarvis_memories_used`
  - `delete_jarvis_memory`
- 新增 `app/jarvis/memory_extractor.py`：
  - 规则型提取偏好/约束、情绪信号、生活节律信号、长期目标、人际关系信息。
  - 保存时带 sensitivity、confidence、importance 和结构化 payload。
  - 构造 `build_memory_recall_prefix`，在下一轮私聊前注入长期记忆，并更新 last_used_at。
- 接入真实私聊 `/api/v1/jarvis/chat`：
  - 对话准备阶段加入长期记忆召回前缀。
  - 对话结束后，从用户输入和 Agent 回复提取候选记忆并写入 `jarvis_memories`。
- 新增记忆治理 API：
  - `GET /api/v1/jarvis/memories`
  - `DELETE /api/v1/jarvis/memories/{memory_id}`
- 新增测试 `tests/unit/jarvis/test_memory_extractor.py`，覆盖规则提取和“提取→保存→召回→标记使用”的真实闭环。
- 更新 `4-26plan.md` 的 Step 3.6 进度，不写路演脚本。

## 当前闭环状态

已经形成真实闭环：

1. 用户在私聊里表达偏好、压力、熬夜、长期目标或关系信息。
2. `/api/v1/jarvis/chat` 生成回复后自动提取候选记忆。
3. 候选记忆写入 `jarvis_memories`，带隐私等级、置信度、重要性。
4. 下一轮私聊准备上下文时召回长期记忆并注入 prompt。
5. 用户或前端可通过 API 查看和删除记忆。

## 剩余问题

- 目前提取器是规则型，只能覆盖明确表达；下一步应接 LLM JSON 提取器，提高事实/关系/隐含偏好识别能力。
- 目前召回按重要性和更新时间排序，还没有按当前 query 做语义相关性检索。
- 情绪/节律信号已保存，但还没有趋势统计、异常检测和低打扰关心策略。
- 前端还没有记忆管理页，需要接 `GET /memories` 和 `DELETE /memories/{id}`。
