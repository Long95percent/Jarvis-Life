# 2026-04-26 刷新保持页面与历史对话模块 Worklog

## 用户反馈

用户不希望用接口命令验证产品，而是要在前端真实使用。进一步发现两个体验问题：

1. 浏览器刷新后当前聊天界面会消失并回到主页。
2. Brainstorm 模式返回/刷新后之前的聊天也会消失。
3. 右下角「主动消息」不如改成可落盘的「历史对话」，让用户能恢复、打开、删除历史。
4. 历史记录如果超过 7 天用户还没点开过，应自动删除。

## 本次完成

### 前端状态保持

- Jarvis 当前 UI 状态写入 `localStorage`：
  - 当前模式：主页 / 私聊 / 圆桌。
  - 当前 Agent。
  - 当前 roundtable scenario。
  - 当前 roundtable 输入。
  - 当前 sessionId。
- 刷新浏览器后不再强制回主页，而是恢复刷新前的私聊或圆桌界面。

### 历史对话后端落盘

- 新增 `conversation_history` 表。
- 支持历史类型：`private_chat`、`roundtable`、`brainstorm`。
- 支持字段：标题、agent_id、scenario_id、session_id、route_payload、last_opened_at、status。
- 新增后端 API：
  - `GET /api/v1/jarvis/conversation-history`
  - `POST /api/v1/jarvis/conversation-history`
  - `POST /api/v1/jarvis/conversation-history/{conversation_id}/open`
  - `DELETE /api/v1/jarvis/conversation-history/{conversation_id}`
- 列表查询时自动软删除“创建超过 7 天且从未打开过”的历史。

### 历史对话前端入口

- 右侧栏下半部分从「主动消息」改为「历史对话」。
- 支持：
  - 查看历史对话。
  - 打开历史对话并恢复到对应界面。
  - 删除历史对话。
  - 显示上次更新时间和上次打开时间。
- 私聊发送消息后后端会保存历史。
- 圆桌/Brainstorm 开始时前端也会先登记历史，避免过程异常导致入口丢失。

### 圆桌/Brainstorm 刷新恢复

- `RoundtableStage` 初始化时会先读取已有 session turns。
- 如果已有历史 transcript，则直接恢复展示，不重新发起 `/roundtable/start`，避免刷新后重复跑一遍。
- 如果没有历史 transcript，才启动新一轮 stream。

## 当前体验方式

1. 打开 `/jarvis`。
2. 进入私聊或 Brainstorm。
3. 刷新浏览器，页面应保持在当前模式，不回主页。
4. 右下角查看「历史对话」。
5. 点击某条历史，可恢复对应私聊/圆桌界面。
6. 点击删除，可自主删除历史。

## 剩余问题

- Brainstorm executor 的完整流式 transcript 是否完全持久化，取决于后端事件是否写入 roundtable/session turns；当前已至少保证历史入口落盘和页面不回主页。
- 历史对话还没有重命名功能。
- 主动消息模块被替换后，如后续仍需要，可放到顶部通知或单独抽屉。
