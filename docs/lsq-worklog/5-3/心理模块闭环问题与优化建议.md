# 心理模块闭环问题与优化建议

> 2026-05-03 检查结论：当前心理模块的基础链路已经跑通，但更像“情绪识别 + 指标快照 + 关怀提醒 MVP”。如果要体现 Jarvis-Life 的差异化能力，还需要把它升级成“可解释指标趋势 + 合理回访时机 + 日程协同减负 + 用户反馈降频”的完整闭环。

## 一、当前已经实现的闭环

目前已经存在这些能力：

1. **聊天中识别心理状态**
   - Mira 聊天时会识别疲惫、压力、睡眠、高风险关键词等信号。
   - 入口主要在 `shadowlink-ai/app/jarvis/mood_care.py`。
   - 如果文本较复杂，会尝试调用 LLM 做结构化情绪分析。

2. **写入情绪观察和每日快照**
   - 聊天识别出的情绪会写入 emotion observation。
   - 每日会聚合成 mood snapshot。
   - 入口主要在 `shadowlink-ai/app/jarvis/mood_snapshot.py`。

3. **生成心理中心趋势数据**
   - 后端接口：`GET /api/v1/jarvis/care/trends`。
   - 前端接口层：`shadowlink-web/src/services/jarvisCareApi.ts`。
   - 前端展示面板：`shadowlink-web/src/components/jarvis/CareTrendsPanel.tsx`。

4. **触发 Mira 主动关怀**
   - 会根据高风险、连续高压、熬夜连续、任务过载等情况生成关怀候选。
   - 入口主要在 `shadowlink-ai/app/jarvis/care_triggers.py`。
   - 定时 routine 会在非安静时间检查关怀触发，入口在 `shadowlink-ai/app/jarvis/proactive_routines.py`。

5. **用户可以反馈关怀是否有用**
   - 前端关怀卡片有“有帮助 / 我已处理 / 稍后提醒 / 太频繁 / 不需要这类”等按钮。
   - 后端接口：`POST /api/v1/jarvis/messages/{message_id}/care-feedback`。
   - 反馈会影响后续 daily budget 和同类触发 cooldown。

## 二、现在不够合理的地方

### 1. 指标折线图还没有真正做出来

现在心理中心虽然有趋势数据，但前端展示方式主要是：

- week/month：单指标竖条图。
- year：热力小方块。

所以用户直观看不到真正的“心情、压力、能量、睡眠、日程压力随时间变化的折线趋势”。

建议后续改成：

- 默认展示最近 7 天或 30 天的多指标折线图。
- 允许单独开关：心情、压力、能量、睡眠风险、日程压力。
- 鼠标悬停某一天时显示该日解释：数据来源、风险标签、触发过什么关怀。
- 对小白用户加一句解释：这些不是医学诊断，只是 Jarvis 根据聊天、日程、行为记录估算的生活状态信号。

### 2. 各种指标现在偏 MVP，计算还比较粗

当前指标大致是这样来的：

- **心情**：根据情绪 valence 映射到 0-10。
- **压力**：取当天情绪观察里的 stress_score 平均值。
- **能量**：约等于 `10 - fatigue_score`。
- **睡眠风险**：如果有熬夜或睡眠问题信号，分数会偏高；否则偏低。
- **日程压力**：取日程压力信号里的较高值。

问题是：

- 睡眠风险现在更像 2/8 这种跳变，不够细。
- 日程压力更多依赖规则信号，和真实日程密度、任务难度、连续工作时长的结合还不够。
- “压力”在生活状态卡片和心理中心里可能含义不同，前端文案需要区分清楚。
- 每个指标缺少“为什么今天是这个分数”的稳定说明。

建议后续把指标分成两层：

- **展示分数**：给用户看的 0-10 分。
- **解释证据**：告诉用户这个分数来自哪些聊天、日程、睡眠、任务过载、正负事件。

### 3. Mira 回访时机不合理，可能“刚聊完就回访”

目前 `mood_care.py` 里会计算 `next_checkin_at`，例如中风险约 2 小时后、高风险约 1 小时后。

但是关键问题是：

- `next_checkin_at` 目前没有真正作为“未来展示时间”被保存和消费。
- `persist_mood_care()` 会直接保存一条 `status=pending` 的 proactive message。
- 前端消息流看到 pending 消息后，就可能立刻展示出来。

所以用户体感可能是：刚和 Mira 说完，Mira 马上又来回访，这不符合真实关怀逻辑。

建议后续区分两种东西：

1. **当前对话陪伴**
   - 用户刚说完压力大，Mira 可以马上在聊天里安抚。
   - 这不是“回访”。

2. **延迟回访任务**
   - Mira 判断稍后需要回访，就创建一个未来任务。
   - 到时间后再进入主动消息流。
   - 卡片应该显示“计划在今晚 21:30 回访”，而不是马上弹出。

### 4. Mira 没有充分利用日程挑空档

用户提到的想法是对的：Mira 应该能看到日程，就不应该在用户有日程、开会、学习、深度工作时打断。

当前关怀触发会参考压力信号和日程压力，但还没有形成完整的“回访排程器”。

建议后续新增一个逻辑层：**Mira 回访排程器**。

它负责根据这些条件选时间：

- 当前风险等级。
- 用户是否刚刚聊过。
- 今天和明天的日程空档。
- 晚间和睡眠时间，避免太晚打扰。
- 用户对上次关怀的反馈，比如“太频繁”“不需要这类”。
- 是否有低优先级任务可以交给 Maxwell 调整。

建议规则示例：

- 高风险：不等待空档，优先安全提示，但仍不做医疗诊断。
- 中风险：优先选 1-3 小时后的第一个 15-30 分钟空档。
- 低风险：放到晚上固定温和 check-in，或者次日早上。
- 如果接下来 2 小时内日程很满：只记录，不弹出；等空档再回访。

### 5. 关怀和日程减负还没有形成强闭环

现在关怀触发里已经有 `suggested_action`，例如：

- `reduce_today_scope`
- `move_low_priority_task`
- `safety_check`

但目前它更像建议字段，还没有稳定变成“可确认的协作动作”。

理想闭环应该是：

1. Mira 发现连续高压或任务过载。
2. Mira 解释为什么认为压力高。
3. Mira 询问是否需要 Maxwell 调整今天安排。
4. Maxwell 生成减负方案。
5. 用户确认后才改日程。
6. Mira 在合适空档回访效果。

这样心理模块才不是普通聊天机器人，而是“心理状态识别 + 日程协同干预 + 后续追踪”的生活系统。

## 三、建议后续优先级

### P0：先修正回访时间逻辑

优先解决“刚聊完就回访”的问题。

建议做法：

- 给 proactive message 或 care intervention 增加可用的 `scheduled_at` / `visible_after` 字段。
- 前端消息流只展示已经到展示时间的消息。
- `next_checkin_at` 不再只是 action 参数，而要真正进入持久化和查询过滤。
- 关怀卡片文案改成“计划回访 / 已稍后提醒 / 已降频”。

### P1：补真正的指标折线图

建议不改后端接口，继续使用 `jarvisCareApi.getCareTrends()`。

前端在 `CareTrendsPanel.tsx` 内新增折线图展示即可。

重点是：

- 一张图看多条指标趋势。
- 点击某一天看解释。
- 不要只显示固定柱状图。

### P2：增强指标解释和计算颗粒度

建议优先改：

- 睡眠风险：从“有/无信号”升级为结合熬夜次数、连续天数、睡眠表达、日程恢复窗口。
- 日程压力：结合日程密度、任务数量、连续无休息时间、逾期任务、改期次数。
- 心情和能量：增加置信度和数据不足提示，避免假装很准。

### P3：做 Mira + Maxwell 协同减负闭环

建议让 Mira 不直接改日程，而是生成 pending action：

- Mira 提出“你今天可能过载”。
- Maxwell 生成“可挪动/可缩小/必须保留”的候选方案。
- 用户确认后才写入日程。

这符合前后端解耦和安全写入原则。

## 四、给开发人员的注意点

1. **不要把心理关怀写成固定文案弹窗**
   - 要区分“即时聊天回复”和“延迟回访”。

2. **不要绕过接口层**
   - 前端继续走 `shadowlink-web/src/services/jarvisCareApi.ts`。
   - 后端继续走 `/api/v1/jarvis/care/*` 和 `/messages/{id}/care-feedback`。

3. **不要让 Mira 直接改日程**
   - 涉及日程调整时，应该进入 pending confirmation。
   - 用户确认后再写入。

4. **不要过度医学化**
   - 文案要强调“生活状态信号”和“关怀建议”。
   - 高风险只做安全提示和求助建议，不做诊断。

5. **不要让用户感觉被打扰**
   - 低中风险回访应该尊重日程空档、安静时间、用户反馈和最近交互时间。

## 五、建议下一步实施顺序

1. 先设计并实现 Mira 回访排程协议。
2. 再让 proactive message 支持未来可见时间。
3. 然后补心理指标折线图。
4. 再逐步增强睡眠、日程压力、能量等指标算法。
5. 最后做 Mira 发现压力后联动 Maxwell 减负的完整 demo。
