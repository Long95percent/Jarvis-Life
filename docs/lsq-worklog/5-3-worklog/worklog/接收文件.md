# 接收文件

## 1. 先把“邮件确认日程”做成秘书侧异步工具
- 不新增一套平行的调度架构，优先复用现有秘书 / 任务 / 主动消息链路。
- 邮件能力作为秘书工具接入：负责发送邮件、收邮件、识别对方回复、把回复转换成可执行的日程变更建议。
- 这一步只增加一个“邮件往返确认”的中间层，不改变现有日程数据模型和主写入入口。

## 2. 把邮件往返做成可追踪的后台流程
- 每次发邮件都生成一个会话/线程标识，绑定到当前日程、计划或待确认动作。
- 线程状态至少区分：`drafted`、`sent`、`awaiting_reply`、`accepted`、`needs_reschedule`、`unavailable`、`closed`。
- 收到心理咨询师回复后，由后端解析为结构化结果，再决定是维持日程、改期，还是重新安排。

## 3. 多智能体在后台自主协调
- 由秘书智能体作为入口，必要时自动路由到心理相关智能体、排期智能体、关怀智能体做后台协商。
- 心理相关任务优先让擅长该领域的智能体参与，不要求前端显式介入每一次协调过程。
- 协调结果统一回到秘书侧，由秘书把最终结论写入日程系统，并生成用户可读的摘要。

## 4. 日程变更必须回写后端数据库
- 无论是对方说“没空”还是“需要改日程”，最终都要落到后端数据库的日程/计划日/投影更新。
- 不能只改前端展示，也不能只改内存状态。
- 如果影响到投影视图，必须触发后端重新投影或同步刷新。

## 5. 主动消息要作为用户通知的一部分
- 在邮件发出后，先给用户一条主动消息，说明“邮件已发送、正在等待回复”。
- 如果对方回复不能按原时间安排，再发一条主动消息，说明“对方没空/需要改期，已经替你安排了新的方案”。
- 主动消息的内容应该说明处理进展，而不是只说内部状态。

## 6. 对现有架构的最小改动建议
- 复用现有 `agent_router` / `jarvis_tools` / 主动消息机制，不新造一套单独的跨智能体框架。
- 新增一个“邮件秘书工具”即可，工具内部负责发信、收信轮询、回复解析、触发协商。
- 多智能体协同尽量走现有 agent roster 和 tool whitelist 机制，只补充心理协商所需的路由规则。
- 对外 API 尽量只暴露一个“发起邮件确认日程”的入口，内部步骤由智能体自动完成。

## 7. 实现顺序建议
- 第一步：先打通邮件发送和线程记录。
- 第二步：接入收信解析和结构化回复。
- 第三步：接入多智能体后台协商。
- 第四步：把协商结论写回日程数据库并同步投影。
- 第五步：补主动消息，让用户能看到全过程。


## 8. 第二点：系统要具备可扩展 MCP / Skill / Tool / API 闭环

### 8.1 目标
- 系统不能只支持“邮件确认日程”这一种能力，而要允许后续持续接入新的 API、MCP 服务、skill 和 tool。
- 不要求第一版做成完整插件市场，但至少要能在一个管理界面里新增、启用、禁用和测试基础能力。
- 智能体可以在后台根据任务需要调用这些能力，并把调用结果继续交给其它智能体协同处理。

### 8.2 最小闭环
- 管理界面新增 skill/tool/API 配置。
- 后端保存配置到数据库或配置表。
- 工具注册器加载这些配置，生成可被智能体调用的工具。
- 智能体执行任务时通过白名单和权限策略选择工具。
- 工具调用结果写回工具执行日志。
- 如果工具产生业务影响，比如改日程、发邮件、写记忆，必须走对应模块的后端写入口。
- 用户能在界面看到“这个工具已添加、可测试、最近调用是否成功”。

### 8.3 MCP 接入方式
- 把 MCP 当成一种工具来源，不直接让前端绕过后端调用外部 MCP。
- 后端维护 MCP Server 配置，例如名称、启动方式、连接地址、可用 tools、权限范围。
- 智能体只看到经过注册器筛选后的工具，不直接看到所有外部能力。
- MCP tool 的执行结果要统一包装成系统内部 ToolResult，方便日志、错误处理和多智能体继续协商。

### 8.4 Skill / Tool 管理界面最小能力
- 新增 tool：填写名称、描述、参数 schema、调用方式、权限级别。
- 新增 MCP：填写 MCP server 名称、连接方式、可暴露工具列表。
- 新增 skill：填写 skill 名称、触发说明、适用智能体、可调用工具范围。
- 测试按钮：允许用一组示例参数执行一次，展示返回结果和错误。
- 启用/禁用：可以临时关闭某个 tool / skill，避免坏工具影响智能体。

### 8.5 权限与安全边界
- 工具要分成只读、需确认写入、可自动写入三类。
- 涉及心理、医疗、财务、外部通信、日程写入的工具，默认至少需要可追踪日志。
- 外部发邮件、改日程、发消息这类动作，即使允许智能体自动执行，也必须记录原因、发起 agent、上下文和结果。
- 不允许前端直接把任意 API 暴露给智能体；必须经过后端注册、schema 校验和权限控制。

### 8.6 多智能体联通方式
- 每个智能体保留自己的角色定义和 tool whitelist。
- 新增 tool / skill 时，可以指定哪些智能体可用。
- 如果一个任务需要跨智能体协作，入口智能体只发起协商任务，具体工具调用仍由拥有权限的智能体执行。
- 例如邮件确认心理咨询日程：秘书智能体发起，心理师智能体判断沟通语气和心理风险，日程智能体负责修改日程，最终秘书智能体给用户主动消息。

### 8.7 与第一点的关系
- 邮件 API 是这个扩展框架里的第一个实际样例。
- 邮件 tool 通过同一套注册、权限、日志、智能体白名单机制接入。
- 后续接入 Google Calendar、Notion、微信、飞书、支付、健康数据等 API 时，不需要重新设计智能体协同链路。

### 8.8 第一版可以不做很完善，但必须具备的能力
- 能新增一个外部 API tool。
- 能给 tool 定义参数 schema。
- 能把 tool 分配给某个智能体。
- 能执行一次测试调用。
- 能让智能体在任务中调用该 tool。
- 能记录 tool 调用日志。
- 能把写操作落到对应后端模块，而不是只返回文本。


## 9. 第二点当前完成度：已做到什么、还缺什么

### 9.1 现在已经做到的部分
- 后端已经有 MCP 基础目录和组件：`app/mcp/client.py`、`app/mcp/server.py`、`app/mcp/registry.py`。
- 后端已经有统一 `ToolRegistry`，可以注册、注销、列出和按名称获取工具。
- 应用启动时已经会初始化 tool registry，并把内置工具注册进去。
- 后端已经有 `/v1/mcp/tools` 查询工具列表接口。
- 后端已经有 `/v1/mcp/tools/call` 调用工具接口，可按工具名和参数执行工具。
- MCP client 已经具备连接外部 MCP server、发现 tools、调用 tools 的基础代码结构。
- Jarvis 智能体定义里已经有 `tool_whitelist`，说明“不同智能体拥有不同工具权限”的方向已经存在。
- 前端 ambient / agent 面板已有一些 tool 展示和 enabled tools 配置影子结构，但还不是完整 tool/skill 管理后台。
- 已有 pending action / 主动消息 / 日程写入链路，可以作为未来“工具执行后闭环落库和通知用户”的基础。

### 9.2 现在只算半成品的部分
- MCP 连接能力存在，但还没有完整的“在界面新增 MCP server 并保存配置”的闭环。
- ToolRegistry 能注册工具，但主要还是启动时注册，缺少运行时从数据库加载用户新增工具。
- `/v1/mcp/tools/call` 能调用工具，但还缺权限校验、agent 白名单校验、写操作确认策略。
- 插件接口目前更像占位：`/v1/mcp/plugins` 返回空列表，reload 也只是返回请求成功。
- 前端能看到一些工具调用过程，但还没有“新增 tool / 新增 skill / 测试 tool / 启用禁用”的正式管理页面。
- 工具调用日志目前主要靠运行日志，不是面向用户或审计的结构化数据库记录。
- 多智能体协作已经有一些咨询/编排工具雏形，但还没有形成“后台 agent-to-agent 协商任务队列”。

### 9.3 还需要做的核心闭环
- 新增 tool 配置表：保存名称、描述、参数 schema、调用方式、权限等级、启用状态、可用 agent。
- 新增 MCP server 配置表：保存 server 名称、transport、endpoint/command、暴露工具、启用状态。
- 后端启动和配置变更时，从数据库加载 tool/MCP 配置并注册到 ToolRegistry。
- 给 `/v1/mcp/tools/call` 增加调用者上下文：agent_id、session_id、permission_mode、是否允许写操作。
- 增加 tool 调用日志表：记录 tool_name、agent_id、arguments、result、success、error、latency、业务影响、创建时间。
- 增加工具权限分级：只读、需要用户确认、允许后台自动执行。
- 增加 skill 配置：skill 名称、触发条件、说明、可用 agents、可调用 tools。
- 增加前端管理页：新增/编辑/启用禁用 tool、MCP、skill，并支持测试调用。
- 增加后台多智能体协商任务：入口 agent 可以创建协商任务，由相关 agent 产出建议，最后回到 owner agent 执行。
- 写操作工具必须落到业务模块，例如日程改动走日程 API，邮件发送走邮件工具，记忆写入走记忆 API。

### 9.4 最小可交付版本建议
- 第一阶段只做“可配置外部 API tool”：不先做复杂插件市场。
- 支持在界面新增一个 HTTP API tool，填写 URL、method、headers、参数 schema、权限等级。
- 支持测试调用，并把调用结果写入 tool 调用日志。
- 支持把这个 tool 分配给某个 agent 的 tool whitelist。
- 支持 agent 在后台任务中调用该 tool。
- 支持写操作后的业务闭环：工具结果必须交给对应服务处理，不能只停留在 LLM 文本。

### 9.5 与邮件秘书功能的优先关系
- 邮件 API 可以作为第一个“可配置外部 API tool”样例。
- 第一版可以先手写邮件工具，保证业务闭环跑通；随后再迁移成可配置 tool。
- 如果一开始就做完整 MCP/skill 管理后台，范围会明显变大，建议先做邮件工具闭环，再抽象为通用扩展能力。
