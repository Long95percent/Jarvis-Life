# 2026-05-04 工作总结

## 1. MCP 外接工具闭环方案
- 重新梳理了外接 MCP 的定位：MCP 不只是“能接工具、能调用 API”，而是必须进入 Jarvis-Life 的业务闭环。
- 明确新闭环：外部 MCP 工具结果 -> 结果归一化 -> ActionProposal -> 内部业务模块执行 -> 数据库落库 -> 投影/上下文刷新 -> 主动消息反馈用户。
- 覆盖更新了 `docs/lsq-worklog/5-3/MCP协议轻量接入方案.md`。
- 原先“只接 MCP / discover tools / 调用 API”的方案已标注为弃用。
- 新方案增加了详细实施计划，包括：Tool 调用日志、ActionProposal 存储、MCP 结果归一化、Proposal Executor、Agent 内部工具 `jarvis_external_life_arrange`、真实或模拟 MCP 样例接入。
- 明确最小验收标准：必须证明外部工具结果能最终写入日程数据库，并让用户收到可见反馈。

## 3. 延续记录
- 今天的主要工作偏方案和接口边界整理，没有继续改核心代码。
- 当前重点结论：
  - MCP 要形成闭环，不能停在工具调用层。
  - 圆桌要继续保持接口稳定，前后端通过契约和 service 边界协作。
  - 外部工具、圆桌、多智能体协作都必须最终回到后端业务模块落库和用户可见反馈。

## 4. 后续建议
- 如果继续实现 MCP 闭环，建议优先做最小样例：外部地点/网页检索 -> 生成 ActionProposal -> 用户确认 -> 写入日程 -> 主动消息。
- 如果继续做圆桌解耦，建议先拆 `roundtableApi.ts`，不要一开始大改 `JarvisHome.tsx` 或 `jarvisStore.ts`。
