6.轻量管家团队架构：以成熟 ReAct / LangGraph 思路为原型的可运行版本

这一版管家团队不再追求一开始就做复杂的多 Agent 公司式组织，而是先做一个轻量、稳定、响应快、容易落地的“管家内核”。它的目标是保留基础能力：角色聊天、MCP 接口、skill 使用、ReAct 工具调用、记忆读取、日程/关怀/任务落盘；同时避免过多 agent 互相讨论导致外接 API 慢、成本高、链路难 debug。复杂圆桌和 Brainstorm 先独立出去，不放进默认管家链路。默认管家模式应该像一个安静高效的单入口系统：用户说一句话，系统快速判断是否需要工具，需要就调用，不需要就直接回答。

这个架构可以参考 LangGraph 的轻量 StateGraph 和 ReAct Agent，而不是采用全自由多 Agent 群聊。核心图非常简单：`Input -> Router -> Context Loader -> Role Agent(ReAct) -> Tool/MCP/Skill Executor -> Committer -> Response`。其中 Router 只做轻量路由，Role Agent 只选一个主要角色执行，Tool Executor 统一管理 MCP 和 skill，Committer 负责把会话、计划、日程、心理观测等真实写入数据库。整个链路最多一次主 LLM 调用，只有在需要工具结果二次总结时才做第二次调用，避免外接 API 响应过慢。

第一层是统一入口。用户仍然可以从任意角色私聊、日历、任务、心理关怀卡片进入，但后台统一封装成 `jarvis_request`：`session_id, turn_id, user_message, active_agent_id, source_surface, user_confirmed_action, context_refs`。前端不需要知道管家团队怎么协作，只负责传入当前用户在哪个界面、正在和谁说话、消息是什么。后端收到请求后，第一步仍然是立即保存 conversation 和 user turn，保证历史记录可见。

第二层是轻量 Router。Router 不做复杂规划，只做四类判断：`chat` 普通聊天、`schedule` 日程/任务、`care` 心理关怀、`knowledge_or_tool` 知识/工具需求。Router 可以先用本地规则判断，只有模糊时才调用 LLM 做 JSON 分类。这样大部分简单消息不需要额外 API 调用。Router 输出字段控制在很少几个：`intent`、`target_agent`、`needs_tool`、`needs_confirmation`、`reason`。例如“你好”直接路由到当前 agent；“帮我安排明天学习”路由 Maxwell；“我最近好累”路由 Mira；“帮我总结这个文件”路由 Athena/Hermes。

第三层是角色 Agent。默认每轮只激活一个主 agent，不做多 agent 互相讨论。建议保留少量稳定角色：
1. `Maxwell`：秘书和计划执行，负责日程、任务、工作台、重排。
2. `Mira`：心理关怀，负责情绪理解、压力建议、作息提醒。
3. `Leo`：生活建议，负责活动、休息、生活安排。
4. `Athena`：知识和学习，负责 RAG、资料总结、学习辅助。
5. `Hermes`：工具执行身份，不直接聊天，只作为 Tool Executor 的内部能力名。
6. `Shadow`：偏好学习器，异步静默运行，不进入主回复链路。

这里不再设置 Auditor、复杂黑板、团队任务分派等机制。质量约束写进 system prompt 和工具 schema，由 Committer 做硬校验。比如 Maxwell 可以生成计划，但真正写库前必须经过 planner service 校验；Mira 可以生成关怀建议，但不能诊断；Hermes 可以执行工具，但所有危险操作必须确认。

第四层是 Agent 基础能力层。这里不应该只保留 ReAct，而是保留现有 agent 基础能力，只是在默认管家链路里按需选择，避免每轮都跑复杂流程。基础能力包括：ReAct 工具调用、Plan-and-Execute 分步规划、RAG 检索、长期/短期记忆读取、MCP 工具、内置 skill、结构化 action 输出、流式回复、取消执行和错误恢复。默认聊天优先走轻量 ReAct：先看用户消息和必要上下文，再判断是否需要工具；不需要工具就直接回答，需要工具就输出结构化 tool call。遇到长期计划、文件处理、复杂资料整理时，可以切换到 Plan-and-Execute，但必须由 Router 明确触发，不能所有请求默认进入复杂规划。Brainstorm 能力保留，但不进入默认管家链路，只作为用户确认后的独立圆桌模式。这里必须限制单轮工具数，例如最多 3 个，避免慢；也要限制二次 LLM，只有工具成功返回且需要自然语言解释时才进行二次总结。对于日程、任务、心理观测这类写操作，agent 只能生成 pending action 或结构化候选，不直接越权写库。

第五层是 MCP 和 Skill 接口。Tool Executor 统一管理两类工具：MCP 工具和本地 skill。MCP 用来接外部或标准化能力，例如日历、文件、搜索、浏览器、本地系统能力；skill 用来封装项目内部稳定能力，例如 `create_plan`、`analyze_emotion`、`summarize_file`、`update_calendar_event`。角色 agent 不直接知道具体实现，只看到工具 schema。这样以后替换 MCP server 或新增 skill，不需要重写 agent。工具返回必须统一成：`ok, data, error, requires_confirmation, pending_action_id, affected_resources`。

第六层是 Committer。Committer 是轻量架构里最关键的业务安全层。它不负责聊天，只负责把工具候选结果变成真实业务状态：保存会话、保存 agent 回复、保存 pending action、确认后写 planner/calendar/care 数据库、同步事件。所有写操作都从 Committer 进入，不允许 agent 直接写 SQLite。这样即使后续换成 LangGraph、AutoGen 或别的框架，业务落盘规则仍然稳定。

第七层是异步观察能力。为了速度，Shadow 偏好学习、心理长期趋势、月度统计、年度统计不应该阻塞用户当轮聊天。主回复完成后，后台异步任务再做：抽取偏好、分析情绪、更新 mood snapshot、更新日程压力信号。用户需要立即看到的只保留轻量结果，比如本轮 Mira 的关怀回复或 Maxwell 的待确认卡。长期趋势图可以由后台定时聚合，不要每次聊天都实时重算全年数据。

第八层是 Brainstorm / 圆桌模式边界。Brainstorm 不放进默认管家团队架构。它是一个独立产品模式：当用户主动点击圆桌，或者某个角色在聊天中判断“这个问题复杂、涉及多视角、需要讨论”时，才推荐用户进入圆桌。推荐方式可以是一个 escalation hint：“这个问题可能适合圆桌模式，要不要让 Maxwell、Mira、Athena 一起讨论？”用户确认后才启动 Brainstorm。默认私聊链路不自动拉多个 agent 发言，避免响应慢和回复混乱。

第八点五层是和其它模块的协作关系。轻量管家团队不是替代日程、心理关怀、历史会话、圆桌模式，而是它们的统一入口和调度层。和会话模块的关系是：所有请求先保存 conversation、session、turn，再进入 Router，保证历史可恢复。和日程模块的关系是：Maxwell 通过 planner skill 生成计划候选，Committer/Planner service 负责确认后写入 `jarvis_plans`、`jarvis_plan_days` 和日历投影。和心理关怀模块的关系是：Mira 可以读取 mood snapshot、作息偏移和日程压力信号，主链路只给即时建议，长期情绪统计交给异步 care service。和 MCP/Skill 模块的关系是：所有外部能力都挂在 Tool Executor 下，agent 只看 schema，不直接碰实现。和 Brainstorm 的关系是：默认管家只推荐圆桌，不自动启动；圆桌结束后把结论回交给 Maxwell、Mira 或 planner service 落盘。这样各模块各司其职，管家团队只负责“选谁处理、调什么能力、把结果交给哪个业务 service”。

第九层是性能策略。由于外接 API 慢，默认链路要尽量少调用模型：
1. 普通聊天：一次 LLM。
2. 明确工具请求：一次 LLM 生成工具调用 + 工具执行 + 必要时一次 LLM 总结。
3. 明确日程/任务：规则先路由 Maxwell，能用本地模板补全的先生成 pending action，复杂长期计划再调用 LLM。
4. 情绪/偏好分析：主链路不阻塞，后台异步。
5. 历史上下文：只取当前 session 最近少量 turn + 必要摘要，不把所有历史塞进 prompt。
6. 工具结果缓存：天气、日历、RAG 检索结果短时间缓存。

第十层是最小数据表。轻量版不需要复杂 team task 表，只保留必要扩展：
1. `agent_chat_turns`：增加 `session_id`、`turn_id`，保证会话恢复和单轮去重。
2. `pending_actions`：所有写操作先进入待确认。
3. `jarvis_agent_events`：保存关键事件，例如 `chat.routed`、`tool.called`、`plan.created`、`care.triggered`。
4. `tool_call_records`：记录工具调用审计。
5. 计划和心理模块继续使用前面设计的 `jarvis_plans / jarvis_plan_days / jarvis_mood_snapshots` 等领域表。

第十一层是接口设计。可以先做很少几个接口：
1. `POST /jarvis/chat`：统一聊天入口，内部完成 Router + Role Agent + ReAct + Committer。
2. `GET /jarvis/chat/{agent_id}/history?session_id=...`：按 session 恢复历史。
3. `GET /jarvis/tools`：返回当前可用 MCP/skill 工具和 schema，便于调试。
4. `POST /jarvis/pending-actions/{id}/confirm`：确认写操作。
5. `POST /jarvis/pending-actions/{id}/cancel`：取消写操作。
6. `GET /jarvis/events?session_id=...`：查看本轮路由、工具和落盘事件，便于 debug。

第十二层是技术实现建议。当前项目 `shadowlink-ai/app/agent/` 已有 ReAct、Plan-and-Execute、MultiAgent、Brainstorm 等目录，轻量管家团队可以优先实现为一个新的 `jarvis_orchestrator`，内部用很简单的状态对象：`request -> route -> context -> tool_calls -> action_results -> response`。如果后续要接成熟框架，最适合的是 LangGraph StateGraph，因为它可以把 Router、Agent、ToolExecutor、Committer 做成节点，同时保留状态可控和可测试。AutoGen/CrewAI 暂时只借鉴思想，不建议作为默认执行框架，因为默认群聊/多角色协作会增加 API 次数和不确定性。

第十三层是典型流程。用户说“你好”，Router 判断 `chat`，目标 agent 是当前私聊角色，Context Loader 取少量历史，Role Agent 一次 LLM 回复，Committer 保存 agent turn，结束。用户说“帮我安排 30 天雅思学习计划”，Router 判断 `schedule`，目标 Maxwell，Maxwell ReAct 调用 `create_plan` skill，生成待确认计划卡，Committer 保存 pending action，用户确认后 planner service 写入计划表和日历投影。用户说“最近好累还睡得晚”，Router 判断 `care`，目标 Mira，Mira 读取近期 mood 和作息信号后给出一条关怀建议，后台异步更新心理趋势。

第十四层是和圆桌模式的关系。默认管家团队解决“快速、稳定、可落盘”的日常问题；圆桌/Brainstorm 解决“复杂、开放、需要多视角讨论”的问题。二者不混在一起。默认管家可以推荐圆桌，但不能擅自启动圆桌；圆桌讨论结束后，可以把最终结论交给 Maxwell 或 planner service 落盘。这种边界能保证日常体验快，复杂讨论也有独立空间。

这个简化版管家团队的核心原则是“单轮一个主 agent、工具统一执行、写操作统一提交、重分析异步、圆桌独立”。它仍然具备成熟 Agent 架构的基础能力：Router、ReAct、Plan-and-Execute、RAG、MCP、Skill、记忆、工具审计、确认机制和异步观察；但不会引入过重的多 Agent 协商。这样更适合当前外接 API 较慢的现实情况，也更容易先把历史、日程、心理关怀这些真实业务链路跑通。
