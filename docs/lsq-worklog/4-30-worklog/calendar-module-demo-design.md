# 日程模块评委演示 Demo 设计

> 日期：2026-05-01  
> 模块：Jarvis / Maxwell 日程与长期计划模块  
> 目标：用 2-3 个高密度 demo 展示日程模块的核心功能、设计理念、系统链路、技术实现和可量化评测指标。

## 1. 演示总目标

本次演示不只展示“能创建日程”，而是重点展示日程模块的三个设计理念：

1. **从自然语言到结构化长期计划**
   - 用户只需说出目标，例如“我要考雅思”。
   - 系统识别这是长期目标，而不是普通聊天或单次日历事件。
   - Maxwell 负责拆解任务、生成计划主记录和每日计划日。

2. **计划是可维护的，而不是一次性生成物**
   - 用户可以延期、重排、批量操作、合并/拆分、投影到日历。
   - 系统保留计划主模型、计划日、工作台、变更事件，支持后续持续维护。

3. **日程不是孤立列表，而是有冲突检测和执行工作流**
   - 后端统一聚合 calendar events、plan days、background task days。
   - 计算冲突和空闲窗口。
   - 前端提供冲突处理、每日维护、Maxwell 工作台推送等执行闭环。

## 2. 推荐 Demo 结构

建议现场演示 3 个测试场景，每个场景 3-5 分钟，总时长控制在 12-15 分钟。

| Demo | 核心展示 | 适合评委看到的亮点 |
|---|---|---|
| Demo 1：自然语言长期目标到可执行计划 | 意图识别、任务拆解、计划主模型、计划日、日历投影 | “不是 ToDo List，而是 Agent 驱动的计划系统” |
| Demo 2：计划维护与异常处理 | 逾期扫描、每日维护、延期、批量操作、重排 | “计划会随着现实变化被维护” |
| Demo 3：冲突处理与执行工作台 | 冲突检测、空闲窗口、Maxwell Workbench、合并/拆分 | “系统能发现问题并支持行动闭环” |
| Demo 4：跨 Agent 路由与秘书专属 Skill | 非 Maxwell Agent 接收请求、意图识别、路由 Maxwell、秘书 Skill 调用、操作留痕 | “项目核心不是单 Agent 聊天，而是多 Agent 分工协作” |

## 3. Demo 1：自然语言长期目标到可执行计划

### 3.1 演示目标

展示用户从一句自然语言目标出发，系统自动完成：

- 意图识别。
- 路由给 Maxwell。
- 生成长期计划主记录。
- 拆解每日计划日。
- 展示任务清单。
- 将计划投影到日历。

### 3.2 建议用户输入

```text
我 3 个月后要考雅思，目标 7 分。帮我安排一个每天都能执行的学习计划，平时晚上有空，周末可以多学一点。
```

### 3.3 系统运行链路

#### Step 1：用户输入

- 用户在 Jarvis 聊天窗口输入长期目标。
- 输入中包含：
  - 时间跨度：3 个月后。
  - 目标结果：雅思 7 分。
  - 约束：平时晚上、周末更多时间。

#### Step 2：意图判断

系统从语义上判断该请求不是普通聊天，而是一个 **长期规划 / 日程安排需求**。

关键判断点：

- 包含长期目标词：`3 个月后`、`目标 7 分`。
- 包含执行计划意图：`安排`、`每天都能执行`。
- 包含时间资源约束：`平时晚上`、`周末`。

预期路由结果：

```text
target_agent = maxwell
planning_scope = background_task_plan / long_project
```

#### Step 3：Maxwell 任务拆解

Maxwell 将目标拆解成结构化计划：

- 计划主记录：`jarvis_plans`
  - title：雅思备考计划。
  - goal：3 个月后达到 7 分。
  - time_horizon：开始日期、目标日期。
  - source_agent：maxwell。

- 计划日：`jarvis_plan_days`
  - 每日听力 / 阅读 / 写作 / 口语训练。
  - 计划日期。
  - 建议开始时间、结束时间。
  - estimated_minutes。
  - status：pending / scheduled。

#### Step 4：前端展示

在 `打开日历 -> 查看所有任务` 中展示：

- 顶层只显示一个“雅思备考计划”。
- 不应出现无数个重复“雅思任务”。
- 点开计划后，右侧显示每日计划、计划详情、变更历史。

#### Step 5：写入日历

用户点击计划详情中的：

```text
写入日历
```

系统调用：

```text
POST /plans/{plan_id}/project-calendar
```

预期结果：

- 将计划日投影为日历项。
- 返回写入数量。
- 前端显示：`已写入 N 个日历项`。

### 3.4 展示重点

向评委强调：

- 系统不是简单把用户输入保存成一条 Todo。
- 系统有清晰模型边界：
  - `jarvis_plans` 是长期计划主模型。
  - `jarvis_plan_days` 是每日执行单元。
  - `calendar_events` 是真正写入日历的时间块。
- 前端任务清单已经统一到 planner task API，避免重复长期任务污染。

### 3.5 可观察验收点

- 任务清单只出现一个长期计划。
- 计划详情能看到多天计划日。
- 点击“写入日历”后日历视图出现对应时间块。
- 计划变更历史能看到相关事件。

## 4. Demo 2：计划维护、延期与批量操作

### 4.1 演示目标

展示计划不是生成后就结束，而是能被持续维护：

- 逾期扫描。
- 今日维护。
- 单个计划日延期。
- 计划整体顺延。
- 批量完成 / 批量延期 / 批量取消。

### 4.2 建议用户输入

```text
我今天太忙了，雅思学习没完成。把没做完的内容帮我往后顺延，并把今天还能做的推给 Maxwell。
```

### 4.3 系统运行链路

#### Step 1：用户输入

用户说明现实变化：今天太忙，任务没完成。

#### Step 2：意图判断

系统识别这是 **计划维护 / 重排请求**，不是新建计划。

核心意图：

```text
intent = planner_maintenance / reschedule
target_agent = maxwell
```

#### Step 3：逾期扫描

用户在任务清单中点击：

```text
逾期计划扫描
```

系统调用：

```text
POST /planner/mark-overdue-missed
```

处理逻辑：

- 找出今天之前仍未完成的 `background_task_days` 和 `jarvis_plan_days`。
- 将状态更新为 `missed`。
- 前端展示：

```text
扫描完成：已标记 X 个逾期项（长期任务 Y 个，计划日 Z 个）
```

#### Step 4：每日维护

用户点击：

```text
今日计划维护
```

系统调用：

```text
POST /planner/daily-maintenance/once
```

处理逻辑：

- 标记 missed。
- 根据需要自动重排。
- 将今日任务推送到 Maxwell Workbench。

前端展示：

```text
维护完成：逾期 X 项，重排 Y 项，推送 Z 项
```

#### Step 5：单个延期 / 整体顺延

如果用户只想延期某一天：

```text
计划日 -> 延期
```

系统调用：

```text
POST /plan-days/{day_id}/move
```

如果用户想整个计划顺延：

```text
整体顺延一天
```

系统调用：

```text
POST /plans/{plan_id}/reschedule
```

处理逻辑：

- 只移动未完成计划日。
- 已完成历史不被改动。
- 记录 `plan.rescheduled` 事件。

#### Step 6：批量操作

用户在计划详情中多选计划日，然后点击：

```text
批量延期 / 批量完成 / 批量取消
```

系统调用：

```text
POST /plan-days/bulk-update
```

处理逻辑：

- 批量更新状态或日期。
- 同步关联日历事件。
- 记录 `plan_day.bulk_updated`。

### 4.4 展示重点

向评委强调：

- 真实计划一定会变化，所以系统重点不是“生成一次”，而是“持续维护”。
- 状态模型清晰：pending、scheduled、pushed、completed、missed、rescheduled、cancelled。
- 高级能力包括：逾期识别、重排、批量处理、事件留痕。

### 4.5 可观察验收点

- 逾期计划日状态从 pending 变为 missed。
- 整体顺延不影响 completed 的历史计划日。
- 批量完成后多个计划日同时变为 completed。
- Maxwell 工作台出现今日推送项。

## 5. Demo 3：冲突处理、工作台执行与计划结构调整

### 5.1 演示目标

展示系统不是静态日程表，而是能处理复杂日程冲突和任务结构变化：

- 冲突检测。
- 空闲窗口推荐。
- 冲突处理。
- 推送到 Maxwell Workbench。
- 合并重复计划。
- 拆分计划日为新计划。
- 归档 legacy background task。

### 5.2 建议用户输入

```text
我明天晚上已经有会议，但雅思学习也安排在那个时间。帮我找冲突，把学习任务挪到合适的空闲时间。另一个“雅思口语练习”其实属于同一个备考计划，也帮我整理一下。
```

### 5.3 系统运行链路

#### Step 1：用户输入

用户描述两个问题：

1. 时间冲突：会议和雅思学习重叠。
2. 结构混乱：雅思口语练习和雅思备考应合并或拆分。

#### Step 2：意图判断

系统识别为复合型日程管理请求：

```text
intent = calendar_conflict_resolution + planner_structure_cleanup
target_agent = maxwell
```

#### Step 3：冲突检测

前端打开日历后调用：

```text
GET /planner/calendar-items?start=...&end=...
```

后端统一聚合：

- `calendar_events`
- `jarvis_plan_days`
- `background_task_days`

然后计算：

- `conflicts`：时间重叠。
- `free_windows`：可用空闲窗口。

前端展示：

```text
冲突处理：N 个
冲突原因：时间重叠
冲突项：会议 / 雅思学习
```

#### Step 4：移动到空闲窗口

用户点击：

```text
移动到空闲窗口
```

处理逻辑：

- 前端选择冲突中的 `plan_day`。
- 根据冲突时长选择足够长的 `free_window`。
- 调用：

```text
POST /plan-days/{day_id}/move
```

- 后端更新计划日日期/时间，状态变为 `rescheduled`。
- 同步关联日历事件。

#### Step 5：推送到 Maxwell Workbench

用户点击：

```text
Maxwell 工作台 -> 推送今日任务
```

系统调用：

```text
POST /maxwell/workbench/push-daily-tasks
```

处理逻辑：

- 找出当天 pending 的计划日。
- 创建工作台项。
- 计划日状态更新为 pushed。

前端展示：

- 工作台 item 标题。
- 来源：计划日 / 长期任务日。
- 截止时间。
- 状态。

#### Step 6：合并 / 拆分计划

如果用户发现两个长期计划重复：

```text
任务合并/拆分 -> 选择目标计划 -> 合并到当前计划
```

系统调用：

```text
POST /plans/merge
```

处理逻辑：

- 源计划日移动到目标计划。
- 源计划状态标记为 merged。
- 不物理删除源计划。
- 记录 `plan.merged`。

如果用户想把某个计划日拆成专项计划：

```text
选中计划日 -> 拆分选中计划日
```

系统调用：

```text
POST /plans/{plan_id}/split
```

处理逻辑：

- 创建新计划。
- 移动指定计划日到新计划。
- 记录 `plan.split`。

#### Step 7：归档 legacy task

如果任务清单中仍有历史 orphan background task：

```text
归档历史任务 / 删除历史任务
```

系统调用：

```text
PATCH /background-tasks/{task_id}
```

处理逻辑：

- 软状态更新为 archived / deleted。
- 默认任务清单隐藏。
- 历史仍可按状态查询。

### 5.4 展示重点

向评委强调：

- 系统有统一日程聚合层，不是多个列表各管各的。
- 冲突处理基于真实时间窗口计算。
- Maxwell Workbench 是从“计划”到“今日执行”的桥梁。
- 合并/拆分体现长期计划模型可维护。

### 5.5 可观察验收点

- 冲突详情能展示冲突项，而不是只显示数量。
- 点击“移动到空闲窗口”后冲突数量下降或消失。
- 工作台显示今日推送项。
- 合并后源计划状态为 merged，目标计划包含迁移后的计划日。
- 拆分后生成新计划。
- 归档 legacy task 后默认任务清单不再展示。

## 6. Demo 4：跨 Agent 路由与秘书专属 Skill

### 6.1 演示目标

这个 Demo 用来突出项目的核心卖点：系统不是一个单体聊天机器人，而是一个有分工、有路由、有专属能力边界的多 Agent 生活操作系统。

重点展示：

- 用户可以对任意 Agent 发起日程/长期任务请求。
- 系统会先做意图判断，而不是让当前 Agent 直接处理所有事情。
- 当请求属于日程、提醒、长期计划、任务拆解、延期维护等秘书职责时，系统会路由到 Maxwell。
- Maxwell 使用秘书专属 Skill 完成结构化计划、日历投影、逾期维护、顺延、冲突处理和工作台推送。
- 路由、工具调用、计划变更和用户可见结果之间有清晰链路，便于评委观察和验证。

### 6.2 建议用户输入

建议现场不要直接选择 Maxwell，而是先选择一个非秘书 Agent，然后输入：

```text
我准备三个月后考雅思，目标 7 分。帮我安排每天学习计划，如果有几天没完成，要能自动顺延并提醒我。
```

这个输入故意包含多层意图：

- 长期目标：三个月后考雅思。
- 目标约束：目标 7 分。
- 日程生成：安排每天学习计划。
- 维护要求：没完成要自动顺延。
- 主动提醒：需要秘书式跟进。

### 6.3 系统运行链路

#### Step 1：非秘书 Agent 接收请求

用户先和普通 Agent 对话，例如陪伴、讨论、建议类 Agent。

系统不能让该 Agent 直接写日程，因为这样会导致：

- 每个 Agent 都能改任务仓库，职责混乱。
- 长期计划、日历、工作台之间容易重复写入。
- 后续延期和维护不知道由谁接管。

因此第一步不是生成计划，而是进入意图判断。

#### Step 2：意图判断与路由决策

系统从输入中识别出这不是普通闲聊，而是秘书类计划请求。

预期识别结果：

```text
intent = long_term_planning + schedule_management + reminder_followup
source_agent = 当前对话 Agent
target_agent = maxwell
route_reason = 用户表达了长期计划、日程安排、延期维护和提醒需求，应由秘书 Maxwell 统一处理
```

评委可观察点：

- 当前 Agent 不直接创建一堆普通任务。
- 对话响应中能说明“该请求将交由 Maxwell 规划/管理”。
- 后端路由 payload 或事件记录中能看到 `source_agent`、`target_agent`、`intent`、`reason`。

#### Step 3：Maxwell 接管并选择秘书 Skill


Maxwell 接管后，不只是换一个角色继续聊天，而是根据意图选择专属 Skill。

本场景应触发的秘书 Skill 能力包括：

- 长期目标拆解 Skill：把“雅思 7 分”拆成阶段目标和每日训练项。
- 日程投影 Skill：把计划日映射到日历时间块。
- 计划维护 Skill：当某天未完成时，将任务标记为 missed 并支持顺延。
- 提醒/今日工作台 Skill：把当天可执行内容推送到 Maxwell Workbench。

预期内部链路：

```text
Maxwell
  -> classify_secretary_skill
  -> create_or_update_plan
  -> generate_plan_days
  -> project_calendar_when_confirmed
  -> register_maintenance_policy
  -> emit_agent_events
```

#### Step 4：结构化落库

Maxwell 生成结果后，应落到统一计划模型，而不是写成零散任务。

核心数据边界：

- `jarvis_plans`：长期计划主记录，只出现一个“雅思备考计划”。
- `jarvis_plan_days`：每日执行单元，记录每天要做什么、预计耗时、计划日期和状态。
- `calendar_events`：用户确认写入日历后才生成真实日历时间块。
- `jarvis_agent_events`：记录路由、Skill 调用、计划创建、计划变更等关键事件。

这个边界可以向评委强调：Agent 的能力不是直接堆数据，而是通过统一业务模型完成可维护的执行闭环。

#### Step 5：前端展示

前端应展示三类结果：

- 对话区：显示请求已由 Maxwell 接管，并总结生成的计划。
- 日程区：任务清单只出现一个长期计划，点开后显示每日计划日。
- 工作台/维护区：可以看到今日任务、逾期扫描、顺延、推送 Maxwell Workbench 等动作。

如果项目开启 demo trace 展示，可以额外展示：

```text
用户输入 -> source_agent -> intent_router -> Maxwell -> secretary_skill -> persistence -> calendar/workbench
```

### 6.4 展示重点

向评委强调：

- 这是 Agent 路由，不是按钮跳转。
- Maxwell 是秘书 Agent，有清晰职责边界。
- 秘书 Skill 是可组合能力，不是写死的单个流程。
- 日程模块通过统一计划模型避免“无数个雅思任务”的混乱问题。
- 后续延期、逾期、提醒都由 Maxwell 继续维护，体现 Agent 的持续服务能力。

### 6.5 可观察验收点


- 对非 Maxwell Agent 输入日程请求后，系统能路由到 Maxwell。
- 生成结果只创建一个长期计划主记录，不重复污染任务仓库。
- 每日计划日可展开查看，并能投影到日历。
- 未完成计划可以被逾期扫描和顺延逻辑维护。
- 路由和 Skill 调用有事件记录，可用于演示可追踪性。

## 7. 客观评测指标

### 7.1 Agent 路由与秘书 Skill 指标

| 指标 | 评测方式 | 目标值 |
|---|---|---|
| Agent 路由准确率 | 对非 Maxwell Agent 输入 20 条日程/长期任务请求，检查是否路由 Maxwell | ≥ 90% |
| 错误路由率 | 输入闲聊、情绪陪伴、知识问答等非日程请求，检查是否误路由 Maxwell | ≤ 10% |
| 秘书 Skill 选择准确率 | 检查长期计划、延期、提醒、冲突处理是否触发对应 Skill 链路 | ≥ 85% |

## 8. 建议现场评测脚本

### 8.1 准备数据

演示前建议准备：

1. 一个空白或清理后的用户数据库。
2. 一个已有日历事件，例如“明晚 20:00-21:00 会议”。
3. 一个雅思长期计划，包含至少 5 个计划日。
4. 一个和会议冲突的计划日。
5. 一个 orphan background task，用于展示归档/删除。

### 8.2 演示顺序

1. 先选择非 Maxwell Agent，输入长期目标，展示系统路由到 Maxwell。
2. 展示 Maxwell 选择秘书 Skill，生成长期计划和每日计划日。
3. 打开日历，展示任务清单只有一个主计划。
4. 打开计划详情，展示每日计划。
5. 点击写入日历。
6. 制造/展示冲突，点击移动到空闲窗口。
7. 运行今日维护，打开 Maxwell 工作台。
8. 多选计划日，批量延期或完成。
9. 展示合并/拆分计划。
10. 归档 legacy task。

## 9. 评委讲解话术

可以用以下方式总结：

```text
这个日程模块的核心不是“保存一个日程”，而是由多 Agent 协作把用户的自然语言目标转成可维护的计划系统。
它有四个层次：第一层是普通 Agent 接收请求并做意图判断，第二层是路由到秘书 Agent Maxwell，第三层是 Maxwell 调用秘书专属 Skill 生成计划主模型和计划日，第四层是日历冲突、每日维护和工作台执行。
所以它能处理真实生活中的变化：没完成可以逾期扫描，计划变了可以整体重排，冲突了可以移动到空闲窗口，任务混乱了可以合并拆分，旧任务可以归档。
```

## 10. 风险与演示注意事项

1. **不要只演示聊天生成**
   - 只演示聊天会让评委以为这是普通 LLM todo。
   - 必须展示计划日、投影、冲突、维护、工作台。

2. **提前准备冲突数据**
   - 冲突展示依赖时间重叠。
   - 建议提前放一个固定会议，再安排同时间学习计划。

3. **明确说明软删除**
   - background task 的删除是软状态 `deleted`。
   - 这是为了保留历史和审计，不是物理删除。

4. **强调边界而非堆功能**
   - 后端负责业务规则。
   - 前端负责触发动作和展示结果。
   - 数据模型负责可追溯和可维护。

5. **必须展示 Agent 路由**
   - 不要一开始就直接选择 Maxwell，否则评委看不到多 Agent 路由能力。
   - 建议先从非秘书 Agent 输入日程请求，再展示系统如何把请求交给 Maxwell。

6. **必须说明秘书 Skill 边界**
   - Maxwell 负责计划、提醒、延期、冲突和工作台。
   - 普通 Agent 可以理解用户和解释结果，但不应直接绕过 Maxwell 写计划仓库。
