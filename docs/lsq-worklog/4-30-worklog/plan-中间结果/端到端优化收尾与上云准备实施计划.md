# Jarvis 端到端优化收尾与上云准备实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development`（推荐）或 `superpowers:executing-plans` 按任务逐项执行。本文是执行计划，不是最终代码；实现时必须遵守 TDD、先小范围验证、再扩大验证。

**Goal:** 补齐 Jarvis 私聊端到端优化的最后工程闭环，让系统从“本地 CPU 可演示”进入“云上可压测、可观测、可回滚、可继续优化”的状态。

**Architecture:** 保持现有 FastAPI + React + Docker Compose 架构，不做大重构。后端把私聊主链路拆成可复用阶段：上下文准备、本地 intent、工具预执行、最终回复、后台维护；前端把已有 `/chat/stream` SSE 从“结果级流式”升级为“状态 + token delta + 最终结果”的渐进式体验；后台维护任务从裸 `asyncio.create_task` 收敛到轻量 runner，便于上云后限流、追踪和 graceful shutdown。

**Tech Stack:** Python 3.10、FastAPI、sse-starlette、pytest/pytest-asyncio、React 18、Zustand、TypeScript、Vite、Docker Compose、OpenAI-compatible LLM provider、SQLite/MySQL/Redis。

## 0. 当前判断

### 0.1 为什么还要补端到端优化

本地 CPU 跑模型肯定慢，继续只围绕本地模型做微调收益有限。现在真正有价值的不是“让本地 CPU 变快”，而是把架构收口到适合上云：

1. 用户可见路径只做必要工作。
2. LLM 调用次数可数、可测、可对比。
3. 慢任务后台化且可观测。
4. 前端尽快显示状态和 token，降低体感等待。
5. 上云后能用同一套 benchmark 比较本地 CPU、本地 Ollama、云 API、不同 provider。

### 0.2 从日志确认的现状

来源：`docs/ohmori-worklog/2026-04-30-daily-summary.txt`

已完成：

1. 私聊角色可以稳定持有对话主权。
2. 私聊角色可以私下咨询其它角色。
3. 已有角色本地 intent + slot 层，覆盖 Maxwell / Nora / Mira / Leo / Athena 的部分高频任务。
4. 长期记忆提取和偏好学习已从主响应链路拆出，改为 `create_task` 后台执行。
5. 背景旁路模型 `background_client` 已存在，主模型和后台维护模型可分离。
6. 上下文准备已有并行读取。
7. `/chat/stream` SSE 入口已存在，前端 `jarvisApi.chatStream` 已能读取 `chat_result`。
8. `timing.spans` 已返回并记录。

未完成但最影响上云闭环：

1. 真实 benchmark 脚本和指标落盘。
2. 私聊 token 级流式输出。
3. Alfred 本地 intent router。
4. sidecar provider probe。
5. 后台任务 runner 化。
6. LifeContextBus 异步生命周期治理。
7. 上云配置、provider 配置、secret 和压测流程固化。

### 0.3 本计划的优先级

P0 必须先做：

1. 基准指标：没有 benchmark，不允许宣称优化完成。
2. token 流式：上云后也可能慢，体感优化优先。
3. 后台 runner：上云后裸 `create_task` 风险放大。

P1 紧随其后：

1. Alfred intent router：减少总管家常用任务多余 LLM。
2. sidecar probe：云 provider 抖动时可诊断。
3. 生命周期治理：减少 pending task warning 和关闭风险。

P2 上云前收口：

1. Docker/env/README/cloud checklist。
2. benchmark 对比报告。
3. 前端 debug 面板或日志可见性。

## 4. Task 3：把 `/chat/stream` 升级为 token 级流式

**目标：** 降低体感等待。即使云 API 总耗时仍受 provider 影响，用户能更早看到状态和回复 token。

**Files:**

- Modify/Create: `shadowlink-ai/app/jarvis/chat_pipeline.py`
- Modify: `shadowlink-ai/app/api/v1/jarvis_router.py`
- Modify: `shadowlink-ai/tests/unit/jarvis/test_chat_token_streaming.py`
- Modify: `shadowlink-web/src/services/jarvisApi.ts`
- Modify: `shadowlink-web/src/stores/jarvisStore.ts`
- Modify: `shadowlink-web/src/components/jarvis/AgentChatPanel.tsx`

### Backend event contract

`/chat/stream` 建议输出事件顺序：

1. `chat_status`

```json
{"phase":"route","message":"正在理解你的请求"}
```

2. `chat_status`

```json
{"phase":"context","message":"正在读取上下文"}
```

3. `chat_status`

```json
{"phase":"tool","message":"正在执行相关工具","tool_name":"jarvis_calendar_create"}
```

4. `chat_delta`

```json
{"delta":"好的，","agent_id":"maxwell"}
```

5. `chat_result`

完整 `AgentChatResponse`，与 `/chat` 兼容。

6. `chat_done`

```json
{"response_ready_ms": 1234.5}
```

7. `chat_error`

```json
{"error":"...","stage":"..."}
```

### Steps

- [ ] Step 1: 写后端流式事件测试

测试必须覆盖：

1. 成功时包含 `chat_status`。
2. 成功时至少一个 `chat_delta`。
3. 最终有 `chat_result`。
4. 最终有 `chat_done`。
5. `chat_result.response` 等于所有 delta 拼接或最终落盘回复。

- [ ] Step 2: 拆出可复用 prompt/context 阶段

不要在 `_chat_stream_events` 里简单调用 `chat_with_agent`，因为那只能等完整结果。

建议拆成：

```python
prepared = await prepare_private_chat(req, llm_client)
async for delta in llm_client.chat_stream(...):
    yield chat_delta(delta)
final = await finalize_private_chat(prepared, full_response)
yield chat_result(final)
```

保留 `/chat`：

```python
response = await run_private_chat_non_stream(req, llm_client)
```

原则：`/chat` 和 `/chat/stream` 共享准备、工具、保存、后台任务逻辑，不复制两套业务。

- [ ] Step 3: 支持工具场景两阶段流

对于本地 intent 命中并预执行工具：

1. 先发 `chat_status phase=tool`。
2. 工具执行完成后，把工具结果放入 prompt。
3. 最终回复用 `llm_client.chat_stream` 输出 token。

对于动态工具块场景：

1. 初期可保留非 token 流，仍发 status。
2. 如果 `run_agent_turn` 内部暂时无法流 token，先输出 `chat_status phase=agent_turn`，最后发一次 `chat_result`。
3. 不要为了这一步大改 `tool_runtime.py`；先覆盖本地 intent + 无工具两类高频路径。

- [ ] Step 4: 前端 `jarvisApi.chatStream` 改为事件回调

建议签名：

```ts
async chatStream(agentId, message, sessionId, browserTimezone, handlers?: {
  onStatus?: (payload) => void;
  onDelta?: (delta: string) => void;
  onResult?: (response: ChatResponse) => void;
}): Promise<ChatResponse>
```

保留旧调用兼容：没有 handlers 时仍返回最终 `ChatResponse`。

- [ ] Step 5: Zustand store 增量更新临时回复

发送用户消息后立即插入：

```ts
{ role: "assistant", content: "", streaming: true, status: "正在理解你的请求" }
```

收到 delta：追加 content。

收到 result：用最终 `AgentChatResponse` 生成完整 turn，覆盖 streaming turn。

出错：把 streaming turn 标记为 error，不吞掉用户消息。

- [ ] Step 6: UI 展示状态

`AgentChatPanel.tsx`：

1. assistant turn 为空但 streaming 时显示状态 chip。
2. 有 delta 后显示逐步增长内容。
3. 工具执行时显示“正在执行工具：xxx”。
4. 错误时显示重试提示。

### 注意点

- 不要破坏圆桌 graph 的 `role_delta` 事件；私聊事件使用 `chat_delta`，命名分离。
- `chat_result` 必须保持旧 `ChatResponse` 格式，避免 store 大改。
- 如果 provider 不支持流式，fallback 到非 token 流：status -> result -> done。
- token 流式完成后仍要保存历史、pending action、background task，不能因为 stream 中断导致状态不一致。
- 客户端断开时后端应停止继续流式生成；后台记忆任务只有在最终回复完成后才调度。

## 6. Task 5：后台任务 runner 化

**目标：** 把当前私聊后台记忆/偏好任务从裸 `asyncio.create_task` 变成可观测、可关闭、有限流的后台 runner，为上云稳定性做准备。

**Files:**

- Create: `shadowlink-ai/app/jarvis/background_runner.py`
- Modify: `shadowlink-ai/app/api/v1/jarvis_router.py`
- Modify: `shadowlink-ai/app/core/lifespan.py`
- Create: `shadowlink-ai/tests/unit/jarvis/test_background_runner.py`
- Modify: `shadowlink-ai/tests/unit/jarvis/test_chat_latency_optimization.py`

### Design

`BackgroundTaskRunner` 最小接口：

```python
class BackgroundTaskRunner:
    def submit(self, name: str, coro_factory: Callable[[], Awaitable[Any]], *, metadata: dict[str, Any] | None = None) -> str: ...
    def snapshot(self) -> dict[str, Any]: ...
    async def shutdown(self, timeout_seconds: float = 5.0) -> None: ...
```

状态字段：

1. submitted。
2. running。
3. completed。
4. failed。
5. cancelled。
6. last_error。
7. max_concurrency。

### Steps

- [ ] Step 1: 写 runner 单测

覆盖：

1. submit 后返回 task id。
2. 成功任务计入 completed。
3. 异常任务计入 failed，不向 event loop 泄漏 exception。
4. max concurrency 生效。
5. shutdown 等待短任务完成。
6. shutdown 超时取消长任务。

- [ ] Step 2: 实现 runner

约束：

1. 不引入 Celery/RQ，当前先轻量实现。
2. 默认 max concurrency 可配置，例如 `JARVIS_BACKGROUND_MAX_CONCURRENCY=3`。
3. 所有异常必须 logger.warning/exception，并进入 snapshot。

- [ ] Step 3: lifespan 初始化

`app/core/lifespan.py`：

1. 启动时创建 runner。
2. 提供 `get_background_task_runner()`。
3. shutdown 时调用 runner.shutdown。

- [ ] Step 4: 替换 chat 后台调度

`jarvis_router.py`：

```python
runner = get_background_task_runner()
runner.submit("chat_memory_and_preference", lambda: _run_chat_background_tasks(...), metadata={...})
```

如果 runner 不可用，fallback 到 `asyncio.create_task`，但要记录 warning。

### 注意点

- 后台任务不能阻塞用户响应。
- runner 不是持久任务队列；服务重启丢任务可以接受，但要在文档中说明。
- 上云后如果需要强可靠，再升级 Redis Queue/Celery；本阶段 YAGNI。

## 12. 风险与注意点

### 12.1 不要大重构

`jarvis_router.py` 已经很大，但本轮目标是收尾，不是重写。只拆出私聊 pipeline 必要阶段，避免把日历、计划、圆桌、心理中心一起重构。

### 12.2 不要伪造性能数字

所有 p50/p95 必须来自 benchmark 输出。日志可以写“mock benchmark 验证格式”，但不能把 mock 数字当真实性能。

### 12.3 本地 CPU 慢不是 bug

本地 CPU 模型慢是预期。验收关注：

1. 本地是否能跑通。
2. 云上是否明显改善。
3. 慢任务是否不阻塞主响应。
4. 首 status / 首 token 是否足够早。

### 12.4 SSE 代理风险

上云后 Nginx/网关可能缓冲 SSE，导致后端已流式但前端仍一次性收到。必须检查：

1. Nginx `proxy_buffering off`。
2. 响应头 `Cache-Control: no-cache`。
3. 必要时 `X-Accel-Buffering: no`。
4. 前端 fetch stream 在目标浏览器可用。

### 12.5 后台任务不能无限增长

后台记忆、偏好、心理快照、旧记忆压缩都可能堆积。runner 必须有限流和 snapshot，否则上云后高并发会隐性拖垮服务。

### 12.6 provider 成本

主模型可用较强模型，后台模型应便宜快速。不要用同一个昂贵模型处理所有后台维护任务。

### 12.7 隐私和日志

benchmark 和 probe 不得输出：

1. API key。
2. 完整用户隐私历史。
3. 过长 prompt。
4. 敏感地理位置原文，除非用户明确用于测试。
