# 前后端协同解耦与止血实施计划

**目标：** 在不冻结后续迭代能力的前提下，先把项目从“单人维持的耦合大工程”整理成“前后端可以并行协作、接口可变更但可快速修正”的状态，并为两天后进入前后端协同开发建立可执行边界。

**定位：** 这不是全面重构计划，也不是功能清单，而是项目治理计划。优先解决：任务过多、职责交织、前后端耦合、接口路径混乱、demo bug 穿插、端到端优化与云部署互相干扰的问题。

**原则：**
1. 先保 demo，再保架构。
2. 先收口接口，再拆内部实现。
3. 先建立协作边界，再允许接口变化。
4. 允许后续接口调整，但每次调整必须有契约和回归测试兜底。
5. 不做一次性大重构，只做可验证的小步解耦。

---

## 0. 当前判断

### 0.1 为什么现在必须解耦

当前项目同时存在：

- 端到端优化收尾
- 云端部署准备
- 多个 demo / 功能 bug
- 私聊、圆桌、计划、日历、本地生活、设置、记忆等多条链路

如果继续所有事情在同一批文件里堆叠，单人维护会很快超过可承受范围。解耦的目的不是为了“好看”，而是为了让：

- 前端可以独立做 UI 和交互调整；
- 后端可以独立改实现；
- 接口调整后双方能按契约快速修正；
- 任何一个 bug 都能定位到明确责任层，而不是全栈联动排查。

### 0.2 当前项目的现状摘要

已观察到的现状：

- 前端已经有服务层：`shadowlink-web/src/services/jarvisApi.ts`，也有通用 `api.ts`。`CareCard.tsx`、`CareTrendsPanel.tsx` 已基本通过 service 调用，但 `AgentChatPanel.tsx` 仍同时承载私聊、日程/计划、待确认动作和 UI 状态编排。`KnowledgePage.tsx` 是 RAG/知识库入口，目前仍直接调用 `/rag/indices` 和 `/file/upload-and-ingest`，需要作为独立 RAG 检索模块解耦出来。`RoundtableStage.tsx` 是圆桌独立入口，圆桌应作为一个前后端整包交给同一个全栈开发者维护，不参与本轮前后端拆工。
- 后端有路由分层，但 `shadowlink-ai/app/api/v1/jarvis_router.py` 已经超过 4k 行，还是巨石式接口文件；其中圆桌端点集中在 `/roundtable/*`，适合整体挖出成为 roundtable 包，但不拆成前端/后端两个协作面。
- RAG 后端已经有相对独立的 `shadowlink-ai/app/api/v1/rag_router.py`、`shadowlink-ai/app/rag/engine.py`、`shadowlink-ai/app/rag/retrieval/*` 和 Java `RagBridgeService`，但前端知识库管理、文件上传摄取、索引管理还没有形成独立 service 契约。
- Java 网关存在 Jarvis 透明代理，但只显式处理了部分 SSE 路径，私聊流式链路尚未完全统一。
- 生产环境下前端、Java 网关、Python AI 三层路径不完全一致，dev/prod 行为可能不同。

### 0.3 这次解耦的成功标准

两天后如果要开始前后端协同开发，最低应该达到：

1. 前端大部分业务请求都通过 service 层，不直接散落 `fetch`。
2. 后端核心接口路径有明确契约文档或类型定义。
3. 圆桌作为独立全栈包有清晰入口；私聊主链路、设置、日程安排、心机关怀、RAG 检索/知识库这些模块有清晰前后端边界。
4. 接口改动能通过类型/测试/Mock 快速修正，而不是全项目手改。
5. Demo 的关键路径有 P0 兜底，不会因为解耦重构导致完全不可用。

## 1. 任务分级

### P0：止血与协作底座

必须先做的事：

1. 先明确圆桌整包边界：圆桌由同一个全栈开发者负责，主界面只保留进入圆桌和接收返回结果的入口。
2. 第二步立刻做前端解耦：收口 Jarvis 主界面的 service/store/component 边界，减少 `AgentChatPanel.tsx` 对 URL、payload、业务动作的直接感知。
3. 明确私聊、设置、日程安排、心机关怀、RAG 检索/知识库的前后端接口归属。
4. 给前后端建立最小接口契约清单。
5. 确保 demo 关键路径不被解耦拖死。

### P1：端到端优化收尾

在 P0 稳定后推进：

1. benchmark。
2. token 级流式。
3. 后台任务 runner。
4. sidecar probe。
5. Alfred intent router。

### P2：结构性解耦

在接口稳定后推进：

1. 先把圆桌识别为独立全栈包：后端 `/roundtable/*`、前端 `RoundtableStage.tsx` 和进入/返回入口归同一负责人。
2. 第二步做前端解耦：拆 `jarvisApi.ts` 中的 chat、schedule、care、settings、memory 分组，并为 RAG/知识库单独建立 `ragApi`。
3. 再推进其它模块后端服务解耦：私聊、日程安排、心机关怀、RAG 内部子路由分别拆。
4. 统一 Java / Nginx / Python 的代理路径。
5. 清理公共类型和接口定义。

## 2. 前端协同解耦计划

### 2.1 前端当前问题

现状不是“没有接口层”，而是“接口层有，但不统一”：

- `shadowlink-web/src/services/jarvisApi.ts` 已经是 Jarvis 主要 API 门面。
- `shadowlink-web/src/services/api.ts` 是通用 HTTP wrapper。
- 但部分页面和组件仍直接 `fetch`。
- 一些页面把 API 路径写死在组件内部，导致路径变更需要逐个搜。

### 2.2 前端第一阶段目标

第一阶段只做四件事：

1. 所有 Jarvis 主界面相关请求统一走 service 层。
2. 页面组件不再直接拼后端路径。
3. `jarvisApi` 里把“请求定义”和“UI 状态更新”分离。
4. 圆桌不参与前后端拆工，只在主界面保留进入入口和返回结果接口；圆桌内部前后端由同一个全栈开发者整体维护。

### 2.3 前端推荐切分方式

建议把现有 Jarvis 服务层拆成下面几类，先按业务边界分组，再决定是否落成多个文件：

- `jarvisChatApi.ts`
  - 私聊、流式、历史、会话。
- `roundtable/` 或 `jarvisRoundtableApi.ts`
  - 圆桌 start/continue/accept/save/return/plan。这个模块作为独立全栈包交给同一个开发者，不作为前后端分离协作的拆分对象；主界面只依赖进入圆桌和接收返回结果的最小接口。
- `jarvisScheduleApi.ts`
  - 日程安排、计划、日历、后台任务、待确认动作。这个模块需要从 `AgentChatPanel.tsx` 中逐步剥离，是前端解耦第二步的重点之一。
- `jarvisCareApi.ts`
  - 心机关怀、心理追踪开关、关怀趋势、关怀反馈、压力/情绪/行为观测。现有关怀组件已基本经由 `jarvisApi`，下一步是独立类型和 service 文件。
- `ragApi.ts`
  - RAG 检索、知识库索引列表、文件上传并摄取、索引删除、后续检索调试。这个模块不挂在 `jarvisApi` 下面，避免 Jarvis 主界面和知识库管理继续互相牵连。
- `jarvisSettingsApi.ts`
  - 个人资料、LLM provider、后台模型 probe、设置保存。
- `jarvisMemoryApi.ts`
  - 长期记忆、协作记忆、记忆删除与查看。

如果短期不拆文件，也可以先在 `jarvisApi.ts` 内部按命名空间分组，但必须保证：

- UI 不直接访问 `fetch`。
- store 只依赖 service 返回值。
- 组件只消费 store 状态，不自己拼协议。

### 2.4 前端解耦优先清单

优先收口这些位置：

1. `shadowlink-web/src/components/jarvis/AgentChatPanel.tsx`：优先抽出日程安排/计划相关 action handler、payload 构造和状态刷新逻辑。
2. `shadowlink-web/src/services/jarvisApi.ts`：先按 `chat / roundtable / schedule / care / settings / memory` 分区整理，再拆文件。
3. `shadowlink-web/src/components/jarvis/CalendarPanel.tsx`：作为日程安排模块的展示边界，避免继续把日程逻辑塞回聊天面板。
4. `shadowlink-web/src/components/jarvis/CareCard.tsx` 与 `CareTrendsPanel.tsx`：保留 UI，后续只切换到 `jarvisCareApi`。
5. `shadowlink-web/src/components/jarvis/RoundtableStage.tsx`：归入圆桌整包，不作为前端解耦重点；主界面只保留进入圆桌和接收返回结果的边界。
6. `shadowlink-web/src/pages/KnowledgePage.tsx`：抽出 RAG/知识库 service，页面只负责上传交互、索引列表展示和错误态，不直接拼 `/rag/*` 或 `/file/*` 路径。

`KnowledgePage.tsx` 属于 RAG/知识库链路，不纳入 Jarvis 主界面解耦主线，但必须作为独立模块并行解耦，避免后续知识库、RAG 检索和 Jarvis 聊天接口混在一起。

### 2.5 前端协同开发规则

从后天开始前后端协同开发时，前端必须遵守：

- 新接口先在 service 层声明。
- 新状态先在 store 层接入。
- UI 组件只关心显示，不关心 URL。
- 接口变动后，只改 service + 类型 + 少量调用点，不在组件里全局搜索硬改。

## 3. 后端协同解耦计划

### 3.1 后端当前问题

后端不是没分层，而是层太大：

- `shadowlink-ai/app/api/v1/jarvis_router.py` 已经 4k+ 行。
- 路由、schema、业务编排、持久化入口都在一个大文件里。
- 这会导致：接口改动时牵扯太多，单人维护很难保持心智清晰。

### 3.2 后端第一阶段目标

第一阶段不拆大系统，只先拆“最容易把其它功能拖下水”的边界：

1. 圆桌整包：优先确认进入方式、返回主线方式和独立文件夹边界，但不拆成前端/后端两个协作任务。
2. 前端解耦：第二步立刻做，先把 `jarvisApi.ts`、`AgentChatPanel.tsx`、`KnowledgePage.tsx` 的 service 边界理清。
3. 私聊主链路和流式输出：保持 demo 稳定，后端后续拆 `chat_router.py` / `chat_pipeline.py`。
4. 日程安排模块：计划、日历、后台任务、待确认动作必须前后端解耦。
5. 心机关怀模块：care settings、trends、snapshots、feedback 必须前后端解耦。
6. RAG 检索/知识库模块：后端已有 `rag_router.py`，本轮重点是固定 `/rag/*`、`/file/upload-and-ingest`、Java bridge 和前端 `ragApi` 的边界。

### 3.3 建议拆分的后端边界

建议后续从 `jarvis_router.py` 拆出：

- `app/api/v1/jarvis/roundtable/router.py`：圆桌 start/continue/decision-result/brainstorm-result/accept/save/plan/return。
- `app/api/v1/jarvis/roundtable/schemas.py`：圆桌 request/response Pydantic model。
- `app/api/v1/jarvis/roundtable/service.py`：圆桌 session、turn、决策收敛、保存和返回主线的业务编排。
- `app/api/v1/jarvis/chat_router.py`：私聊与流式，等圆桌整包边界确认后再拆。
- `app/api/v1/jarvis/schedule_router.py`：日程安排、计划、日历、后台任务、待确认动作，优先级低于圆桌。
- `app/api/v1/jarvis/care_router.py`：心机关怀、心理追踪、关怀趋势、行为/压力/情绪观测，优先级低于圆桌。
- `app/api/v1/rag_router.py`：保留为 RAG/知识库独立后端入口，不并入 Jarvis router；必要时只在 RAG 包内部继续拆 query、ingest、indices 子模块。
- `app/rag/`：保持 engine、retrieval、embedding、index、chunking 的后端领域边界；Jarvis 工具只能通过 `knowledge_search` 或明确 service 调用进入 RAG，不直接操作索引细节。
- `settings_router.py` 保留，但内部可继续拆 provider 操作。
- `chat_pipeline.py`：私聊上下文准备、intent、工具预执行、最终回复。

### 3.4 后端协同开发规则

从后天开始后端协同开发时，必须遵守：

- 所有新增接口先定义 request/response schema。
- 路由只做参数校验和转发，不堆业务。
- 业务逻辑要落到可测试函数里。
- 新增接口要附单测，至少覆盖成功和失败路径。
- 接口变更必须同步更新前端 service 类型。

## 4. 接口契约策略

### 4.1 为什么要有契约

你担心“接口有改后续能不能修正”，答案是：**能修正，但前提是变更有边界。**

如果没有契约，接口改动会变成隐性炸弹；如果有契约，即使接口改了，也只是：

1. 改后端 schema；
2. 改前端 service 类型；
3. 改少量 UI 适配；
4. 跑测试确认；
5. 继续开发。

### 4.2 最小契约清单

建议先把这些契约固定下来：

- 私聊：`/chat`、`/chat/stream`、`/chat/{agent_id}/history`
- 设置：`/providers`、`/providers/{id}/activate`、`/providers/{id}/activate-background`、`/providers/test`
- 圆桌：`/roundtable/start`、`/roundtable/continue`、`/roundtable/{session_id}/decision-result`、`/roundtable/{session_id}/brainstorm-result`、`/roundtable/{session_id}/accept`、`/roundtable/{session_id}/save`、`/roundtable/{session_id}/plan`、`/roundtable/{session_id}/return`
- 日程安排：`/calendar/events`、`/plans`、`/plan-days`、`/background-tasks`、`/background-task-days`、`/planner/*`、`/pending-actions/*`
- 心机关怀：`/care/settings`、`/care/settings/tracking`、`/care/trends`、`/care/days/{day}`、`/care/snapshots`、`/care/behavior-observations`、`/care/behavior-events`、`/care/stress-signals`、`/messages/{message_id}/care-feedback`
- RAG/知识库：`/rag/indices`、`/rag/indices/{mode_id}`、`/rag/query`、`/file/upload-and-ingest`，以及 Java `RagBridgeService` 对 Python RAG 的转发契约
- 记忆：`/memories`、`/collaboration-memory` 相关端点

### 4.3 契约的落地方式

优先级从高到低：

1. TypeScript interface/type。
2. Python Pydantic model。
3. 单元测试。
4. 接口文档注释。

如果三者冲突，以测试和真实响应为准。

## 5. Demo 与 bug 的处理策略

### 5.1 先别把所有 bug 混到一起

现在 bug 太多，不要按“看到一个修一个”来做。建议分为：

- P0：会让 demo 崩溃、打不开、直接报 500 的。
- P1：影响体验、但不致命的。
- P2：可延后到协同开发后再修的。

### 5.2 Demo 优先顺序

建议优先保住这几条链路：

1. 私聊能正常发消息和回复。
2. 设置页能正常打开、保存基本配置。
3. 圆桌能起、能继续、能收敛，并且迁移到独立后端模块后路径不变。
4. 日程/计划能查看，不黑屏。
5. 端到端优化相关流式/后台任务不报错。

### 5.3 修 bug 时的原则

- 不要顺手重构无关模块。
- 不要把一个 demo bug 修成全局重写。
- 先定位是哪一层坏了：UI、service、路由、持久化、模型、部署。
- 每修一类 bug，就记录一次接口/契约缺口，避免重复踩坑。

## 6. 两天后协同开发的建议节奏

### 第 1 天：收口与分工

1. 冻结新功能。
2. 列出 P0 bug。
3. 确认后端私聊 / 设置 / 圆桌 / 日程安排 / 心机关怀的接口归属。
4. 先给圆桌建立独立后端模块边界，并保证 `/roundtable/*` 对前端路径不变。
5. 收口 Jarvis 主界面 service 分组，尤其是 `AgentChatPanel.tsx` 中的日程安排逻辑。
6. 形成接口契约清单。

### 第 2 天：并行开发起步

1. 前端解耦放第二个做：按 service/store/component 三层推进，圆桌不参与前后端拆工。
2. 圆桌作为独立全栈包单独交付，只暴露进入和返回主线边界。
3. 后端按 chat / schedule / care / rag 的 router / pipeline / service 推进。
4. 每天至少一次同步契约变更。
5. 每个模块都有单测或最小回归验证。
6. demo 路径保持可运行。

### 协同规则

- 接口变更必须先同步到文档或类型声明，再改调用方。
- 如果接口改了，允许修正，但必须可追踪、可回退。
- 不靠口头约定，靠 contract + test。

## 8. 接下来要做的具体实施顺序

### 8.1 第一步：圆桌整包独立交付

目标：把圆桌识别为独立全栈包，由同一个开发者同时维护圆桌前端和后端；主界面只保留进入圆桌和接收返回结果的接口。

要做：

1. 新建 `shadowlink-ai/app/api/v1/jarvis/roundtable/` 文件夹。
2. 新建 `router.py`，迁移这些端点：
   - `POST /roundtable/start`
   - `POST /roundtable/continue`
   - `GET /roundtable/{session_id}/decision-result`
   - `GET /roundtable/{session_id}/brainstorm-result`
   - `POST /roundtable/{session_id}/accept`
   - `POST /roundtable/{session_id}/save`
   - `POST /roundtable/{session_id}/plan`
   - `POST /roundtable/{session_id}/return`
3. 新建 `schemas.py`，迁移圆桌相关 Pydantic model。
4. 新建 `service.py`，承接圆桌 session、turn、决策结果、保存、返回主线等业务编排。
5. 在 `jarvis_router.py` 或 `app/main.py` 中 include 新 router，保持原 URL 不变。
6. 补最小回归测试：至少覆盖 start、continue、return 三条路径。

验收：

- 前端无需改 URL。
- 圆桌 demo 能进入、继续、收敛、返回主线。
- `jarvis_router.py` 行数明显下降，圆桌逻辑不再和日程/关怀/私聊混在同一个文件里。

### 8.2 第二步：前端解耦优先推进

目标：除圆桌整包外，其它模块先把前端 service/store/component 边界理清，让后端接口后续变化时只改 service 和类型。

要做：

1. 拆或整理 `shadowlink-web/src/services/jarvisApi.ts`，至少形成 `chat / schedule / care / settings / memory` 分区。
2. 新建或整理 `shadowlink-web/src/services/ragApi.ts`，让 `KnowledgePage.tsx` 不再直连 `fetch`。
3. 从 `AgentChatPanel.tsx` 中优先剥离日程/计划 payload 构造、保存、编辑、完成、删除、刷新 handler。
4. `CareCard.tsx`、`CareTrendsPanel.tsx` 继续保留 UI，只切换到 care service 分区。
5. 圆桌前端不纳入本步骤，只保留主界面进入圆桌和接收返回结果的接口。

验收：

- Jarvis 主界面组件不直接拼业务 URL。
- 日程、关怀、RAG 的前端调用集中在各自 service。
- 后端接口改动时优先只改 service/type，不全局搜索 UI 组件。

### 8.3 第三步：日程安排模块前后端解耦

目标：在前端边界稳定后，把日程/计划的前后端接口契约固定下来，再逐步拆后端。

要做：

1. 在 `jarvisApi.ts` 里先整理出 `schedule` 分区，覆盖 calendar、plans、plan-days、background-tasks、pending-actions、planner 相关方法。
2. 从 `AgentChatPanel.tsx` 中抽出日程 payload 构造、保存、编辑、完成、删除、刷新这些 handler。
3. `CalendarPanel.tsx` 作为日程展示组件，不直接承载聊天逻辑。
4. 后端后续拆 `schedule_router.py`，覆盖 calendar、plans、plan-days、background-tasks、pending-actions、planner 相关端点。

验收：

- 聊天面板不再直接散落日程接口细节。
- 日程 UI 可以独立调用 schedule service。
- 后端日程相关端点有清单，后续可以按清单迁移。

### 8.4 第四步：心机关怀模块前后端解耦

目标：把关怀模块从通用 `jarvisApi` 里独立成清晰分组，并固定前后端契约。

要做：

1. 在 service 层整理 `care` 分区，覆盖 care settings、tracking、trends、day detail、data clear、feedback、behavior/stress/emotion 相关方法。
2. `CareCard.tsx` 和 `CareTrendsPanel.tsx` 继续只负责展示和用户操作，不直接感知 URL。
3. 后端后续拆 `care_router.py`，但路径和 response shape 先以契约清单固定。

验收：

- 心机关怀前端调用边界清楚。
- 以后拆后端 `care_router.py` 时，前端只需要改 service，不需要改 UI 组件。

### 8.5 第五步：RAG 检索/知识库模块解耦

目标：把 RAG 检索和知识库管理从通用页面直连中解耦出来，形成独立前端 service + 独立后端入口 + 明确 Java bridge 的边界。

要做：

1. 新建或整理 `shadowlink-web/src/services/ragApi.ts`，覆盖索引列表、索引删除、文件上传并摄取、后续 query/debug 方法。
2. `KnowledgePage.tsx` 改为只依赖 `ragApi`，不直接调用 `fetch`，不直接拼 `/rag/*` 和 `/file/*` 路径。
3. 后端保留 `shadowlink-ai/app/api/v1/rag_router.py` 作为 RAG 独立入口，不并入 `jarvis_router.py`。
4. 如果 `rag_router.py` 后续继续变大，只在 `shadowlink-ai/app/api/v1/rag/` 内拆 `query_router.py`、`ingest_router.py`、`indices_router.py`，不要放进 Jarvis 模块。
5. 明确 `shadowlink-ai/app/tools/knowledge_search.py` 是 Jarvis 调用 RAG 的工具入口，Jarvis 私聊/圆桌不得直接操作 RAG index manager。
6. 明确 Java `RagBridgeService` 只做转发和鉴权/错误包装，不承载 RAG 检索算法或索引实现。

验收：

- `KnowledgePage.tsx` 没有直接 `fetch`。
- RAG 前端类型和方法集中在 `ragApi.ts`。
- RAG 后端入口仍是 `/v1/rag` 或当前部署约定路径，Jarvis router 不新增 RAG 管理端点。
- Jarvis 需要知识检索时只通过 `knowledge_search` 或明确的 RAG service 契约进入。

### 8.6 第六步：后端服务继续分层

目标：圆桌整包边界确认后，再逐步拆私聊、日程、关怀、RAG 内部子路由，不把所有拆分压在同一轮。

顺序：

1. `chat_router.py` / `chat_pipeline.py`：私聊、流式、历史。
2. `schedule_router.py`：日程安排、计划、后台任务、待确认动作。
3. `care_router.py`：心机关怀、心理追踪、趋势、观测、反馈。
4. `rag/query_router.py`、`rag/ingest_router.py`、`rag/indices_router.py`：只在 RAG 模块内部继续细拆。
5. 保留 settings/provider 现有可用链路，只在必要时内部细拆。

验收：

- 每拆一个 router，都保持原前端路径不变。
- 每拆一个模块，都补一个最小接口测试。
- 圆桌整包单独交付；其它模块不在同一个 PR/提交里同时拆日程、关怀、RAG 三大块。

## 9. 小白也能用的前后端解耦初步检测方法

### 9.1 先理解什么叫“解耦做得好”

前后端解耦不是“前端和后端完全不联系”，而是：

- 前端知道“调用哪个 service 方法”，但不关心后端内部怎么实现。
- 后端知道“提供什么接口和数据格式”，但不关心页面内部怎么展示。
- 接口变化时，主要改 service/type/schema/test，不需要到处翻 UI 组件和业务实现。
- 一个模块出 bug 时，能先判断是 UI、前端 service、后端 router、后端 service、数据库/模型哪一层的问题。

简单说：**前端和后端通过契约合作，不通过互相猜代码合作。**

### 9.2 前端侧自查清单

每拆一个模块，先做这些检查：

1. 页面组件里是否还直接写 `fetch('/api/xxx')`。
   - 好：`AgentChatPanel.tsx` 调 `jarvisScheduleApi.createEvent(...)`。
   - 差：`AgentChatPanel.tsx` 里直接 `fetch('/api/v1/calendar/events', ...)`。
2. 页面组件里是否还直接拼 URL、query string、后端路径。
   - 好：URL 只在 `services/*.ts` 里出现。
   - 差：多个组件里都出现同一个 `/calendar/events`。
3. TypeScript 类型是否集中。
   - 好：`CalendarEvent`、`CareTrend`、`RagIndex` 这类类型在 service/types 中统一定义。
   - 差：每个组件自己写一份差不多的对象结构。
4. UI 组件是否只负责显示和交互。
   - 好：按钮点击后调用 `scheduleService.completePlanDay(id)`。
   - 差：按钮里直接组 payload、拼 URL、处理后端字段兼容。
5. 一个接口字段改名时，是否只需要改 1-3 个地方。
   - 好：改 service type + service adapter + 少量调用点。
   - 差：需要全项目搜索字段名，到处改 UI。

### 9.3 后端侧自查清单

每拆一个后端模块，先做这些检查：

1. router 是否只做三件事：接收参数、调用 service、返回 response。
   - 好：`router.py` 很薄，业务逻辑在 `service.py`。
   - 差：router 里几百行，又查库、又拼 prompt、又改状态。
2. request/response schema 是否清楚。
   - 好：有 Pydantic model，比如 `CreateCalendarEventRequest`、`CareTrendsResponse`。
   - 差：到处传 `dict[str, Any]`，前端不知道会返回什么。
3. 模块之间是否通过明确 service 调用。
   - 好：Jarvis 需要知识检索，只调 `knowledge_search` 或 RAG service。
   - 差：聊天逻辑直接操作 RAG index manager。
4. 原接口路径是否保持稳定。
   - 好：内部拆文件后，前端 URL 不变。
   - 差：拆一次后端，前端所有路径都要跟着改。
5. 是否有最小接口测试。
   - 好：至少测成功、失败/空数据、权限或参数错误中的一种。
   - 差：只能靠手点页面发现接口挂了。

### 9.4 最快的人工检测办法

如果你不懂代码，可以用这几个“土办法”初筛：

1. 搜 `fetch(`。
   - 命令：`rg "fetch\(" shadowlink-web/src`
   - 目标：业务页面里越少越好，最好集中在 `src/services/`。
2. 搜后端路径。
   - 命令：`rg "/calendar|/care|/rag|/plans|/roundtable" shadowlink-web/src`
   - 目标：路径主要出现在 service 文件，而不是组件文件。
3. 看大文件是否继续膨胀。
   - 重点看 `jarvis_router.py`、`jarvisApi.ts`、`AgentChatPanel.tsx`。
   - 如果越拆越大，说明只是搬代码，不是真解耦。
4. 改一个接口字段做演练。
   - 例如把测试环境 response 里的 `plan_date` 改名，看需要改多少地方。
   - 如果只改 service/type，说明解耦不错；如果 UI 到处炸，说明还耦合。
5. 断开某个后端接口，看前端错误态。
   - 好的解耦应该能显示“加载失败/重试”，而不是整页白屏。

### 9.5 模块验收标准

一个模块可以算“初步解耦完成”，至少满足：

- 前端页面不直接拼该模块后端 URL。
- 前端有独立 service 方法和类型定义。
- 后端有明确 router/schema/service 边界，或者至少有清晰的拆分计划。
- 接口契约写在文档、TypeScript type、Pydantic model 或测试里。
- 改接口时有固定流程，不靠口头通知。

## 10. 协作开发中接口要改怎么办

### 10.1 先记住一个原则

接口可以改，但不能偷偷改。

规范合作软件开发里，接口不是某个人随手改完再让别人猜，而是走一套小流程：

1. 先提出变更原因。
2. 再改契约文档/type/schema。
3. 前后端确认影响范围。
4. 写兼容或迁移方案。
5. 最后改代码和测试。

### 10.2 接口变更的标准流程

每次要改接口，按这个顺序走：

1. 写一句“为什么要改”。
   - 例：`/plan-days` 需要增加 `source_agent`，因为前端要区分 Maxwell 生成和用户手动创建。
2. 写清楚“改前/改后”。
   - 改前：`{ id, title, plan_date }`
   - 改后：`{ id, title, plan_date, source_agent }`
3. 判断是不是破坏性变更。
   - 非破坏性：只新增可选字段，旧前端还能跑。
   - 破坏性：删除字段、字段改名、类型改变、路径改变。
4. 优先做兼容。
   - 新增字段优于改名。
   - 新接口优于直接破坏旧接口。
   - 后端临时同时返回新旧字段，比立刻删字段更安全。
5. 前后端同时改契约。
   - 前端改 TypeScript type/service。
   - 后端改 Pydantic schema/router test。
6. 跑最小验证。
   - 前端至少能编译或打开对应页面。
   - 后端至少跑对应接口测试。
7. 在工作记录里写一句结论。
   - 例：`2026-05-02 /plan-days 新增 source_agent，可选字段，前端已适配，旧调用不受影响。`

### 10.3 接口变更分级

接口变更可以按风险分三级：

#### A 类：安全变更

通常可以当天合入：

- 新增可选字段。
- 新增接口。
- response 增加不影响旧前端的字段。
- 错误信息更清楚，但 status code 不变。

要求：更新 type/schema，简单测一下。

#### B 类：中风险变更

需要前后端同步：

- 字段含义变化。
- request 新增必填字段。
- response 结构多包了一层，比如从 `{ items: [] }` 变成 `{ data: { items: [] } }`。
- status code 或错误格式变化。

要求：先写兼容期，前后端一起改，至少保留旧逻辑一段时间。

#### C 类：高风险变更

不要随便做，必须单独排期：

- 删除接口。
- 改 URL 路径。
- 字段改名且不保留旧字段。
- 大改鉴权、分页、流式协议、SSE event 名称。
- 数据库结构变化导致旧数据不可读。

要求：写迁移说明、回滚方案、测试清单，并约定合入时间。

### 10.4 推荐的接口契约模板

每个模块至少维护这样的表：

| 模块 | 接口 | 方法 | 前端 service | 后端 router | 变更风险 | 负责人 |
| --- | --- | --- | --- | --- | --- | --- |
| 日程 | `/calendar/events` | GET/POST/PUT/DELETE | `jarvisScheduleApi` | `schedule_router.py` | B | 前端A/后端B |
| 关怀 | `/care/trends` | GET | `jarvisCareApi` | `care_router.py` | A | 前端A/后端B |
| RAG | `/rag/indices` | GET/DELETE | `ragApi` | `rag_router.py` | A | 前端A/后端B |
| 圆桌 | `/roundtable/*` | 多个 | 圆桌整包内部 | roundtable 包内部 | 整包内处理 | 圆桌全栈负责人 |

### 10.5 分支和提交怎么协作

规范开发一般不是大家直接往同一个文件乱改，而是：

1. 每个任务开一个小分支。
   - 例：`feat/frontend-service-decouple`
   - 例：`feat/schedule-contract`
2. 一个分支只做一类事。
   - 好：只拆日程 service。
   - 差：同一个分支里同时改日程、RAG、圆桌、样式、部署。
3. 小步提交。
   - 好：`refactor(web): extract schedule api service`
   - 好：`test(ai): add calendar event contract test`
4. 合并前做自查。
   - 有没有直接 `fetch` 残留？
   - 有没有改接口但没改 type/schema？
   - 有没有影响圆桌整包边界？
5. 合并后同步一条变更记录。
   - 写清楚改了什么接口、谁需要适配、怎么验证。

### 10.6 前后端联调时怎么沟通

推荐每个接口问题都按这个格式说，避免“它坏了”这种无效沟通：

```text
模块：日程安排
接口：POST /calendar/events
现象：前端创建日程返回 422
前端发送：`{ title, start_time, end_time }`
后端期望：`{ title, starts_at, ends_at }`
建议：后端兼容 start_time/end_time，或前端改为 starts_at/ends_at
影响：CalendarPanel 创建日程
优先级：P0，因为 demo 主链路会失败
```

### 10.7 什么情况要停下来同步

遇到这些情况，不要继续各改各的，要先同步：

- 要改 URL。
- 要删字段。
- 要把可选字段改成必填。
- 要改 SSE / stream event 名称。
- 要改登录、鉴权、provider 配置。
- 要改数据库表结构。
- 一个改动会同时影响 2 个以上模块。
- 前端为了适配接口写了很多临时判断。

### 10.8 这次项目的具体注意点

结合当前项目，特别注意：

1. 圆桌是独立入口，交给同一个全栈负责人，不把它拆成前后端协作战场。
2. 前端解耦第二步必须先做，否则后端再拆 router 时前端会到处跟着炸。
3. `AgentChatPanel.tsx` 不要继续塞日程、关怀、RAG 逻辑。
4. `jarvisApi.ts` 可以先分区，再拆文件，不要一上来全量重写。
5. `KnowledgePage.tsx` 的 RAG 直连 `fetch` 要尽快收口到 `ragApi.ts`。
6. 后端拆 router 时，优先保持 URL 不变，让前端少改。
7. 接口字段尽量新增可选字段，不要直接删除或改名。
8. 每个模块至少留一个“最小回归路径”：能打开页面、能发请求、能处理失败。
