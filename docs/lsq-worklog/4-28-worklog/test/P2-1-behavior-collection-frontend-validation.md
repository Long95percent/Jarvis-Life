# P2-1 行为采集可靠性与桌面端策略前端验证方法

## 验证目标

验证行为采集不是“每 30 秒堆一条 heartbeat”，而是能在正常使用中形成 activity session window，并能处理刷新、关闭、隐藏、idle、恢复和 Electron 桌面窗口事件。

## 路演 Demo 准备

1. 打开 Jarvis 前端正常聊天页面，不打开后台调试页。
2. 确认心理趋势追踪处于开启状态。
3. 如果演示桌面端，使用 Electron 启动应用；如果只演示 Web，也能验证浏览器生命周期。

## Demo 操作脚本

### 场景 1：正常使用合并为 activity window

1. 打开 Mira 私聊页面，停留 1 分钟以上。
2. 期间发送一条普通消息，或移动鼠标/输入文字。
3. 等待至少一次 heartbeat。
4. 打开心理趋势中心，点击当天。

预期表现：

- 行为证据中能看到 `activity_window`，而不是大量重复 heartbeat。
- activity window 有首次活跃、最后活跃和持续分钟数。
- 这些行为证据会进入当天心理趋势详情解释。

### 场景 2：刷新、切后台、关闭兜底

1. 在 Mira 页面刷新浏览器。
2. 切到其他标签页或最小化窗口，再切回来。
3. 关闭当前聊天面板或关闭页面。
4. 再打开心理趋势中心点击当天。

预期表现：

- `visibility_hidden` / `visibility_visible` / `closed` 等生命周期事件被记录。
- 关闭时优先使用 `sendBeacon` 或 `keepalive` 兜底，不依赖普通异步请求一定完成。
- 没有出现用户可见的调试面板。

### 场景 3：idle 与恢复

1. 打开 Mira 页面后不要操作超过 5 分钟。
2. 之后移动鼠标或按键恢复操作。
3. 点击心理趋势当天详情。

预期表现：

- 行为证据中能看到 `idle_start`，恢复后 activity window 被继续更新。
- 恢复事件不会把用户强制打断，也不会主动弹窗，只作为疲劳/作息风险输入。

### 场景 4：Electron 桌面端窗口事件

1. 使用 Electron 桌面端打开 Jarvis。
2. 最小化、恢复、重新激活主窗口。
3. 关闭窗口到托盘或退出应用。
4. 回到心理趋势中心点击当天。

预期表现：

- 前端通过 preload 收到 `app_opened`、`app_minimized`、`app_activated`、`app_restored`、`app_closed` 等事件并上报。
- 主进程不直接写心理数据，只把桌面生命周期安全转发给渲染层。

### 场景 5：关闭心理追踪后停止写入

1. 在心理趋势中心关闭心理追踪。
2. 重复刷新、切后台、聊天页停留等操作。
3. 再点击当天详情。

预期表现：

- 不再写入新的 behavior observation。
- 前端不会展示伪造的新行为数据。

## 验收口径

- 验证必须通过前端正常使用流程完成，不把后台日志当作唯一验收。
- 行为采集只作为心理趋势与疲劳风险输入，不做医疗诊断或强制干预。
- 本步骤只代表 P2-1 完成；P2-2 日程压力完整来源与计划联动、P2-3 心理中心产品化仍需继续。
