# StepC2 圆桌 Decision 路演使用手册

## 路演故事

用户晚上已经很累，但还有学习任务。Jarvis 不替用户做决定，也不擅自改日程，而是召集 Mira、Maxwell、Athena、Alfred 做一次决策圆桌：先判断“继续还是调整”，再把用户接受的建议交给 Maxwell 生成待确认动作。

## 演示前准备

为了让效果更明显，建议提前准备：

1. 在日历中放 1-2 个晚间学习或工作事件。
2. 在 Maxwell 工作台保留 2 个今日待办。
3. 打开 Mira 私聊，输入一句：`我今天很累，还有点焦虑，但晚上还有学习任务。`

这些准备不是为了展示后台数据，而是让用户能看到 Jarvis 会结合“疲惫 + 学习任务 + 日程压力”给出更像生活管家的建议。

## 路演步骤

### 1. 引出问题

打开 Jarvis 首页，对观众说：

> “我现在不是想要一个普通聊天回复，而是希望几个专业角色一起帮我判断今晚要不要继续学。”

进入圆桌场景，选择“疲惫学习决策”，输入：

```text
我很累但还有学习任务，要不要继续？
```

### 2. 展示角色轮流发言

观察舞台：

- Mira 关注情绪、疲劳和恢复边界。
- Maxwell 关注日程可行性和任务安排。
- Athena 关注学习收益、优先级和认知负荷。
- Alfred 做综合判断。

讲解重点：

> “这里不是一次性生成总结，而是保留圆桌感：每个角色轮流发言，用户能看到观点如何形成。”

### 3. 展示 Decision 卡

等待右侧 Decision 卡出现，重点展示：

- 推荐方案。
- 利弊取舍。
- 下一步动作。
- 页面提示“接受后只生成待确认动作，不直接改日程”。

讲解重点：

> “Jarvis 给出的是可执行建议，但最终执行权仍在用户手里。”

### 4. 接受建议

点击“接受建议”或“让 Maxwell 执行”。

预期用户可见结果：

- 页面提示已生成待确认卡。
- 不会立刻新增日历事件。
- 之后可去 Maxwell / 待确认动作中确认。

讲解重点：

> “它不会越权修改我的时间表，只是把建议变成一个待确认动作。”

## 成功标准

路演时观众应该能理解：

- Jarvis 能把一个模糊问题变成多角色决策。
- 角色发言有区分度，不是单一助手回复。
- 最终结果是结构化建议，不是长文本堆叠。
- 接受建议后进入待确认动作，不直接改日程。

## 安全边界

- Mira 不做心理诊断。
- 高风险场景只提供安全支持、可信任的人和当地紧急渠道建议。
- 页面文案不恐吓、不承诺治愈。
