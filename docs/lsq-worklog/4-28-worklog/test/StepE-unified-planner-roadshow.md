# Demo 9 ASCII Copy: missed days trigger Maxwell auto-reschedule

Goal: show Jarvis does not pretend a missed task was completed. It marks the old day as missed, asks Maxwell/LLM to rebuild future days, syncs calendar events, and tells the user what changed.

## Steps

1. Prepare a multi-day plan with one old unfinished plan day and at least one future unfinished plan day.
2. Open Calendar -> Tasks and show the original plan state.
3. Trigger maintenance:
   - Product path: after local 01:00, routine calls daily maintenance once per date.
   - Roadshow path: call `POST /jarvis/planner/daily-maintenance/once?today=YYYY-MM-DD`.
4. Refresh Calendar -> Tasks.
5. Confirm the old unfinished day is now `missed`.
6. Confirm future plan days moved to new dates/times generated by Maxwell/LLM.
7. Open weekly calendar view and confirm linked calendar events moved too.
8. Open proactive messages and show Maxwell's explanation message.

## Success Criteria

- Old unfinished day is persisted as `missed`.
- Future day updates are persisted in `jarvis_plan_days`.
- Existing calendar event is synced to the new date/time.
- `plan.rescheduled` event is recorded.
- Maxwell proactive message explains the reschedule.

# Demo 10 ASCII Copy: persistent calendar, conflict detection, and audit history

Goal: show the schedule system is no longer process-memory only and can be operated from normal user pages.

## Steps

1. Create two overlapping calendar events from the Calendar page.
2. Refresh the page and confirm both events are still visible.
3. Restart the AI backend if desired, then open the same date range again; events should still be present because they are stored in SQLite.
4. Observe the conflict summary on the Calendar page.
5. Open Calendar -> Tasks, select a plan, edit a plan day date/time, and save.
6. Confirm the linked calendar event moves with the plan day.
7. Check the plan change history section to see recorded planner events.
8. Trigger missed auto-reschedule and show the care/proactive message telling the user that unfinished plan items were detected and the plan was already rescheduled.

## Success Criteria

- Calendar events survive refresh/restart.
- Conflict/free-window information appears in the normal Calendar page.
- Plan day edit works from the normal Tasks page.
- Plan change history is visible to the user.
- Care reminder is user-facing, not a debug panel.
