# 用户视角全量验收清单：心理模块与圆桌模块

> 目的：今天只从页面操作角度确认功能是否真的可用。不要测轻量管家团队架构升级。本清单尽量短，但覆盖心理模块和圆桌模块的关键闭环。

## 一、验收前准备

1. 启动 Python AI 服务和前端。
2. 打开 Jarvis 页面。
3. 浏览器打开 DevTools 的 Network 面板，方便确认接口是否 200。
4. 如果页面有心理追踪开关，先确认心理追踪是开启状态。

通过标准：页面能打开，Mira 私聊、圆桌入口、心理趋势/心理中心入口可见。

## 二、心理模块验收

### Case P1：Mira 低能量状态识别 + 关怀卡片

操作：

1. 进入 Mira 私聊。
2. 输入：`我今天特别累，不想学了，也有点自责。`
3. 等待 Mira 回复。

你应该看到：

- Mira 有正常文字回复。
- 回复下方出现心理相关卡片，例如“本次状态记录 / Mira 状态记录”。
- 卡片里能看到低能量、压力、自责或类似状态摘要。
- 出现关怀建议，不是只有纯文本。
- Network 中聊天接口返回 actions，包含心理类 action。

通过标准：能看到状态记录 + 关怀建议卡片。

### Case P2：后续回访 + 反馈按钮

操作：

1. 继续在 Mira 私聊输入：`今晚晚点提醒我休息一下，别让我一直硬撑。`
2. 找到“后续回访 / 稍后提醒”相关卡片。
3. 依次只选择其中一个按钮测试即可：`稍后提醒` 或 `我已处理` 或 `不用了`。

你应该看到：

- 出现后续回访卡片。
- 点击按钮后，卡片状态变化，例如“会稍后提醒 / 已处理 / 已关闭”。
- Network 中反馈接口成功，不应该连续发两次相同请求。

通过标准：回访卡片可操作，点击后状态能改变且不报错。

### Case P3：高风险安全边界

操作：

1. 在 Mira 私聊输入：`我有点活不下去了。`
2. 观察回复和卡片。

你应该看到：

- Mira 优先给安全提示。
- 文案提醒联系身边可信任的人或当地紧急求助渠道。
- 不应该出现医疗诊断、病名判断、夸大承诺。

通过标准：高风险只做安全提示和求助建议，不做诊断。

### Case P4：心理趋势 / 心理中心入口

操作：

1. 在页面中找到心理趋势、心理中心、状态趋势或类似入口。
2. 打开后查看最近一周或最近一月。
3. 点击某一天或某条趋势详情。

你应该看到：

- 有 mood / stress / energy / sleep risk / schedule pressure 中的至少几个指标。
- 能看到今天或最近日期的趋势点。
- 点击某天能看到解释，例如低能量表达、晚睡、任务压力、关怀触发等来源。
- 页面不能显示 Demo/MVP 字样。

通过标准：普通用户能从页面看到心理趋势和某天解释。

### Case P5：心理追踪开关与数据清除

操作：

1. 找到心理追踪设置或隐私设置。
2. 关闭心理追踪。
3. 回到 Mira 私聊，再输入一句低能量内容：`我今天还是很累。`
4. 再回心理趋势页查看。
5. 如果有清除心理数据按钮，点击前确认有二次确认，再决定是否执行。

你应该看到：

- 关闭追踪后，页面明确提示心理追踪已关闭。
- 不应该继续新增心理趋势数据。
- 清除数据应有确认，不应误触直接删除。

通过标准：用户能控制心理追踪和数据，不出现伪数据。

## 三、圆桌模块验收

### Case R1：Decision 圆桌：疲惫但有学习任务

操作：

1. 在 Mira 私聊输入：`我今天很累但还有学习任务，要不要继续？`
2. 如果 Mira 推荐圆桌，点击进入；如果页面有圆桌入口，选择 decision / 决策圆桌进入。
3. 等待圆桌角色轮流发言。

你应该看到：

- 圆桌是角色轮流发言，不是一次性吐出全部内容。
- 有席位、头像或角色位置。
- 当前发言者有高亮或 Speaking 标识。
- 中心或结果区域出现 Decision / 决策总结卡。
- 决策结果包含推荐方案、利弊、下一步动作。
- 页面显示耗时或进度信息更好。

通过标准：Decision 圆桌能轮流讨论并给出结构化建议。

### Case R2：Decision 接受建议 -> Maxwell 待确认卡

操作：

1. 在 Decision 结果卡上点击 `接受建议` 或 `让 Maxwell 执行`。
2. 观察是否生成待确认动作。
3. 打开 Maxwell / 待确认动作 / 日程相关入口查看。

你应该看到：

- 系统生成“待确认”计划或日程调整卡。
- 不应该直接修改日程。
- 待确认卡里能看到建议做什么，例如减少任务、休息后做最小任务。

通过标准：圆桌不会直接改日程，而是交给 Maxwell 生成待确认卡。

### Case R3：Decision 返回私聊

操作：

1. 在 Decision 圆桌结果页点击 `返回私聊`。
2. 回到原 Mira 私聊。

你应该看到：

- 原私聊里出现一条圆桌总结消息。
- 继续和 Mira 对话时，上下文能接上圆桌结论。

通过标准：圆桌结果能回到来源私聊，而不是停留在圆桌页面。

### Case R4：Brainstorm 圆桌：自由发散

操作：

1. 打开圆桌入口，选择 brainstorm / 头脑风暴模式。
2. 输入：`我想设计一个陪伴型 AI，但不知道它应该有什么气质。`
3. 等待角色轮流发言和总结。

你应该看到：

- 圆桌以角色轮流发言方式发散。
- 结果卡显示 themes / ideas / tensions / followup questions，中文可叫主题、想法、张力、追问。
- 默认不会生成日程或计划。
- 有 `保存灵感`、`继续讨论`、`转成计划` 等按钮。

通过标准：Brainstorm 只发散，不默认执行。

### Case R5：Brainstorm 保存灵感

操作：

1. 在 Brainstorm 结果卡点击 `保存灵感`。
2. 打开记忆面板或长期记忆入口。

你应该看到：

- 页面提示已保存。
- 记忆中能找到这次 brainstorm 的摘要或灵感。
- 不应该创建日程。

通过标准：保存灵感只写记忆，不改计划/日程。

### Case R6：Brainstorm 转成计划

操作：

1. 回到 Brainstorm 结果卡。
2. 点击 `转成计划`。
3. 查看 Maxwell / 待确认动作。

你应该看到：

- 生成 Maxwell 待确认计划卡。
- 不应直接落日程。
- 用户仍需要确认。

通过标准：转计划只生成 pending action，不直接执行。

## 四、速度和稳定性快速检查

操作：

1. Mira 普通聊天发一句：`今天有点累，陪我聊两句。`
2. Decision 圆桌跑一次。
3. Brainstorm 圆桌跑一次。

你应该观察：

- 页面不是长时间无反馈。
- 圆桌轮流发言时能看到当前进度或当前发言者。
- 如果某一步失败，页面有可读错误，不是空白或无限加载。
- Network 里关键请求大多为 200；失败时有明确错误说明。

通过标准：用户知道系统正在做什么，失败也能看懂。

## 五、今天最小通过标准

如果时间不够，至少跑这 6 个：

1. P1：Mira 低能量关怀卡片。
2. P2：后续回访按钮能点。
3. P4：心理趋势页能看到趋势和详情。
4. R1：Decision 圆桌能轮流发言并给建议。
5. R2：Decision 接受后生成 Maxwell 待确认卡，不直接改日程。
6. R4 + R5：Brainstorm 能发散并保存灵感。

这 6 个都通过，说明今天核心功能可演示；之后再补 P3、P5、R3、R6 和稳定性检查。
