## 二、当前进度判断

### 1. 心理模块当前已具备

- 情绪 observation 表与规则保存 MVP。
- 日级 mood snapshot 表与聚合 MVP。
- 行为 observation 表与前端事件 MVP。
- 日程压力 signal 表与压力计算 MVP。
- `/care/trends` 趋势接口与前端组件 MVP。
- care trigger / care intervention / feedback / snooze MVP。

### 2. 心理模块距离全量还差

- emotion observation 缺真实 `turn_id` 关联。
- 情绪分析缺 LLM 结构化 fallback 和完整情绪谱系。
- mood snapshot 缺稳定自动调度、backfill、正向事件来源和详情解释。
- 行为采集缺 Electron/桌面端与异常关闭容错。
- 压力信号缺连续高压天数、计划重排日志、`jarvis_plan_days` 完整接入验证。
- 心理趋势缺稳定产品入口、day detail 交互和关闭追踪时的隐私状态。
- care trigger 需要确认自动接入 proactive/routine。
- CareCard 可能存在反馈双请求风险，需要检查和修复。
- 关怀反馈需要按 trigger_type 精细降频。

### 3. 圆桌模块当前已具备

- `decision_result` MVP。
- `brainstorm_result` MVP。
- `accept` 生成 Maxwell pending action。
- `save` 保存 brainstorm 灵感。
- `plan` 将 brainstorm 转 Maxwell pending action。
- 前端已有 decision / brainstorm 结果卡、席位、发言高亮、主持总结卡、长度控制。
- 保留角色轮流发言并加 roundtable timing。

### 4. 圆桌模块距离全量还差

- `/roundtable/{session_id}/return` 仍未产品化。
- `roundtable_results` 的 `user_choice / handoff_status / result_json` schema 仍需规范。
- `title / user_prompt / source_session_id / source_agent_id` 需要独立稳定保存。
- `/roundtable/{session_id}/run` 是否保留未决策；如果不用，需要从设计中明确合并到 start/continue。
- 圆桌结果作为普通消息写回来源私聊的链路未完整。
- 轮流发言模式下的错误降级、单 Agent 失败继续、用户可见进度还需补齐。
- 圆桌与心理状态、日程压力、今日计划的可解释连接还需前端展示。

## 三、第一批 P0：今天优先补齐

### P0-1：心理 observation 补真实 turn_id 关联（已完成）

目标：emotion observation 必须能追溯到具体用户 turn。

实现要求：

- 修改 `save_chat_turn()` 或新增返回 id 的保存函数。
- Jarvis 私聊中先保存用户 turn，再将 turn id 传给 `persist_mood_care()`。
- `jarvis_emotion_observations.turn_id` 新数据不再为 `None`。
- 历史数据允许为空，但新数据必须关联。

验收：

- 用户向 Mira 发送心理相关消息后，`agent_chat_turns` 和 `jarvis_emotion_observations` 可通过 turn_id 对上。
- 测试覆盖 turn_id 不为空。

worklog：`P0-1-心理observation-turn-id关联-worklog.md`

### P0-2：关怀触发自动运行与反馈降频（已完成）

目标：关怀触发不能只存在函数里，必须能自动运行，并尊重用户反馈。

实现要求：

- 确认并接入 `evaluate_care_triggers()` 到 proactive engine / routine scheduler。
- 检查并修复 `CareCard` 与父组件可能重复发送 feedback 请求的问题。
- 降频策略按 trigger_type 生效，不只是全局 daily budget。
- 用户选择 `too_frequent / not_needed` 后，同类型提醒降频。
- snooze 到期后才重新出现在 feed。

验收：

- 连续高压/晚睡/任务过载会自动生成 care intervention。
- 用户反馈“太频繁”后，同类提醒减少。
- 同一次点击只发送一次 feedback 请求。

worklog：`P0-2-关怀自动触发与反馈降频-worklog.md`

### P0-3：圆桌 return 返回私聊链路（已完成）

目标：圆桌结果可以回到来源私聊，作为普通消息继续上下文。

实现要求：

- 新增 `POST /roundtable/{session_id}/return`。
- 根据 `source_session_id / source_agent_id` 找回原私聊。
- 将圆桌总结保存为原私聊中的一条 agent/system 消息。
- 返回前端跳转所需 payload：agent_id、session_id、message_id、summary。
- 如果来源缺失，返回可读错误，不要 500。

验收：

- Decision / Brainstorm 圆桌点击“返回私聊”后，原私聊出现圆桌总结。
- 继续和原 Agent 对话时，能接上圆桌结论。

worklog：`P0-3-圆桌return返回私聊-worklog.md`

### P0-4：规范 roundtable_results schema（已完成）

目标：不要长期用 `status / pending_action_id / options_json / tradeoffs_json / context_json` 拼凑所有语义。

实现要求：

- `roundtable_results` 增加或规范：`result_json`、`user_choice`、`handoff_status`、`handoff_target`、`accepted_at`、`returned_at`、`saved_at`。
- Decision 和 Brainstorm 都用 `result_json` 保存完整结构化结果。
- 保持旧字段兼容读取。
- 更新 persistence、router、前端类型和测试。

验收：

- `decision_result / brainstorm_result` 能完整从 `result_json` 恢复。
- 用户执行 accept/save/plan/return 后，`user_choice` 和 `handoff_status` 正确变化。

worklog：`P0-4-roundtable-results-schema规范化-worklog.md`

### P0-5：轮流发言错误降级和进度反馈（已完成）

目标：保留轮流发言体验，但单 Agent 失败不能导致整场圆桌崩掉。

实现要求：

- 每个 `agent_turn` 有 timing、status、error。
- 某个 Agent 失败时，主持人给出降级说明并继续后续流程。
- SSE 输出 progress / agent_error / roundtable_timing 或等价事件。
- 前端显示当前第几位、共几位、谁失败或被跳过、是否正在生成总结。

验收：

- 模拟一个 Agent 发言失败，圆桌仍能继续并生成总结。
- 用户能看到失败说明，而不是无限加载或空白。

worklog：`P0-5-圆桌轮流发言错误降级-worklog.md`

## 四、第二批 P1：补全核心体验

### P1-1：LLM 结构化情绪分析 fallback（已完成）

目标：心理分析不能只靠关键词规则。

实现要求：

- 保留规则快速层。
- 对规则低置信度、多情绪混合、长文本倾诉、含糊表达调用 LLM。
- LLM 输出 JSON：`primary_emotion`、`secondary_emotions`、`valence`、`arousal`、`stress_score`、`fatigue_score`、`risk_level`、`confidence`、`evidence_summary`。
- LLM 失败降级到规则结果，不阻塞聊天。
- 不保存用户心理原文全文。

验收：

- 混合表达能输出多情绪结构。
- 高风险场景仍走安全边界。

worklog：`P1-1-LLM结构化情绪分析-worklog.md`

### P1-2：mood snapshot 自动调度与 backfill（已完成）

目标：日级心理快照必须自动形成，不依赖手动访问接口。

实现要求：

- 用户启动或服务启动时检查当天 snapshot。
- 每日固定时间聚合前一天 snapshot。
- 支持最近 N 天缺失 snapshot backfill。
- snapshot 聚合包含 emotion、behavior、stress、care intervention、任务完成情况。
- positive_events 接入正向来源：任务完成、表达开心/放松、按时休息、减少任务负载。

验收：

- 不访问 `/care/snapshots` 也会生成当天/前一天 snapshot。
- 缺失日期可补齐。
- snapshot 里有正向和负向事件。

worklog：`P1-2-mood-snapshot自动调度-worklog.md`

### P1-3：心理趋势 day detail 解释链路（已完成）

目标：用户能看到“为什么这天压力高/能量低”。

实现要求：

- `/care/trends` 或新增 `/care/days/{date}` 返回详情。
- 详情包含关联 emotion observations、behavior observations、stress signals、care triggers、snapshot summary。
- 前端点击某一天展开解释。
- 解释要可读：任务密度、逾期、晚睡、低能量表达、连续高压等。

验收：

- 用户点击某天能看到压力来源。
- 关闭心理追踪时不展示伪数据。

worklog：`P1-3-心理趋势详情解释-worklog.md`

### P1-4：圆桌 source/title/user_prompt 独立保存（已完成）

目标：圆桌 session 自身保存完整来源和用户问题，不依赖 conversation_history 间接拼。

实现要求：

- `roundtable_sessions` 或相关持久化补齐：mode、source_session_id、source_agent_id、title、user_prompt、status。
- `/roundtable/start` 必须写入这些字段。
- 历史恢复和前端展示使用这些字段。

验收：

- 刷新后圆桌能显示标题、用户原问题、来源 Agent。
- return 能稳定找到来源私聊。

worklog：`P1-4-圆桌session来源字段补齐-worklog.md`

### P1-5：圆桌与心理/日程上下文的可解释展示（已完成）

目标：圆桌不是凭空建议，要说明参考了什么。

实现要求：

- Decision 圆桌展示参考上下文摘要：今日心理状态、任务压力、关键日程、长期目标。
- 不暴露敏感心理原文。
- 用户可展开“为什么推荐这个方案”。
- Brainstorm 默认不读取过多私密上下文，除非用户明确要求。

验收：

- 用户知道建议来源。
- 心理隐私不泄漏给不该看的角色或前端详情。

worklog：`P1-5-圆桌上下文解释-worklog.md`
