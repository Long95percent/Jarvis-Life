# 08-智能体回复速度调优任务

## 一、任务目标

当前 Jarvis 已经具备多 Agent 私聊、Agent 私下咨询、长期记忆召回、长期任务拆解、主动打扰和基础圆桌能力。下一阶段需要把“能跑通”推进到“响应稳定、用户觉得快、可定位慢点”。本任务聚焦智能体回复速度，围绕智能体间路由、长期记忆检索、LLM 调用链路、前后端端到端延迟建立测试方案和优化方案。

目标不是盲目减少能力，而是在不破坏心理机制、圆桌会议和长期记忆闭环的前提下，把默认私聊、任务路由、圆桌启动和主动建议的延迟控制在可接受范围内。

## 二、需要测试的核心链路

### 1. 默认私聊链路

- 用户在任意 Agent 私聊中发送普通聊天。
- 后端完成会话落盘、上下文装配、偏好画像注入、长期记忆召回、LLM 回复、agent turn 写库。
- 重点指标：首 token 时间、完整回复时间、数据库写入耗时、记忆召回耗时。

### 2. 跨 Agent 路由链路

- 用户和 Mira / Leo / Athena 聊天时输入计划、日程、复盘、心理压力等混合意图。
- 系统需要判断是否路由 Maxwell、是否触发 Agent consult、是否只由当前 Agent 回复。
- 重点指标：规则路由耗时、LLM 分类耗时、consult 额外耗时、错误路由率。

### 3. 长期记忆检索链路

- 用户输入涉及过往偏好、长期目标、情绪历史、任务背景的问题。
- 系统从 `jarvis_memories` / preference profile / recent turns 中取上下文。
- 重点指标：召回条数、召回耗时、prompt token 增量、相关性、是否造成明显慢响应。

### 4. 圆桌会议链路

- 用户启动 decision 圆桌或 brainstorm 圆桌。
- 系统预取必要上下文，然后一次生成固定轮次讨论和主持总结。
- 重点指标：上下文预取耗时、单次 LLM 生成耗时、前端可感知等待时间、是否出现多次串行 LLM 调用。

### 5. 端到端前端链路

- 从用户点击发送到前端出现第一段反馈、最终回复、卡片渲染完成。
- 重点指标：浏览器请求耗时、接口耗时、JSON 解析和状态更新耗时、重复请求或重复渲染。

## 三、测试方案

### 1. 埋点与日志

为 Jarvis 主链路增加统一 timing span，至少覆盖：

- `request_received`：后端收到请求。
- `conversation_persisted`：会话和用户 turn 已落盘。
- `route_decided`：完成意图/Agent 路由。
- `memory_recalled`：完成长期记忆和偏好画像召回。
- `llm_started` / `llm_finished`：主 LLM 调用耗时。
- `consult_finished`：私下咨询耗时。
- `actions_built`：pending action / calendar / task card 构建完成。
- `response_sent`：后端返回。
- 前端记录 `send_clicked`、`response_headers`、`first_render`、`final_render`。

### 2. 基准用例集

建立 20 条固定测试 prompt，每次优化前后重复跑：

- 5 条普通陪伴聊天。
- 5 条日程/计划路由到 Maxwell。
- 4 条需要长期记忆召回的问题。
- 3 条心理关怀相关问题。
- 3 条圆桌启动或圆桌继续问题。

每条记录：Agent、session_id、是否新会话、是否有历史记忆、是否触发 consult、总耗时、首响耗时、错误情况。

### 3. 数据规模测试

分别构造三档本地数据：

- 小数据：10 条会话、50 条记忆。
- 中数据：100 条会话、1000 条记忆。
- 大数据：1000 条会话、10000 条记忆。

重点观察长期记忆召回、历史会话查询、proactive feed、background task 查询是否随着数据量线性变慢。

### 4. 验收指标

初版建议指标：

- 普通私聊首响：目标 1.5 秒内，完整回复 6 秒内。
- 不触发工具的 Agent 路由：目标 300ms 内完成规则判断，LLM 分类只在不确定时触发。
- 长期记忆召回：目标 300ms 内完成 Top-K 召回和裁剪。
- Agent consult：只在必要场景触发，额外耗时可见且可降级。
- 圆桌：默认一次 LLM 生成整轮，避免角色逐个串行请求。

## 四、优化方案

### 1. 智能体间路由优化

- 先用规则和轻量分类器判断明显日程、长期计划、心理风险、圆桌推荐意图。
- 只有规则置信度不足时才调用 LLM 做结构化分类。
- 对 Maxwell 计划类、Mira 心理类、Athena 分析类、Leo 体验类建立明确路由边界。
- Agent consult 默认异步或限流：普通聊天不触发，复杂/高风险/跨领域问题才触发。

### 2. 长期记忆检索优化

- 召回前先按 `visibility`、`owner_agent_id`、`allowed_agent_ids`、`memory_tier` 做 SQL 预过滤。
- 默认只取 Top-K 小集合，避免把大量记忆塞进 prompt。
- 近期原文、久远压缩、偏好画像分层注入：普通私聊只注入摘要，必要时再展开原文。
- 为近期活跃 session 和常用 agent profile 做短 TTL 缓存。

### 3. LLM 调用链路优化

- 默认私聊尽量一轮 LLM 完成回复；工具判断、草稿、最终回复不要无条件串行两轮。
- 日程/计划写库前需要结构化输出，但缺字段时只追问，不进入完整规划。
- 圆桌采用“一次生成固定轮次 JSON”的方式，不按角色逐个请求。
- 对低风险系统消息、主动 routine、历史标题生成等任务使用更小模型或延后执行。

### 4. 数据库与接口优化

- 高频列表接口增加分页和时间范围，避免一次拉全量历史。
- `conversation_history.session_id`、`agent_chat_turns.session_id`、`jarvis_memories.owner_agent_id/status/created_at`、`background_task_days.plan_date/status` 建索引检查。
- pending action、calendar events、background task day 的组合查询尽量一次接口返回，减少前端瀑布请求。

### 5. 前端体验优化

- 用户点击发送后立即显示“已收到/正在思考”状态，降低主观等待。
- 把首段文本、工具卡片、最终结论分阶段渲染。
- 防止历史恢复、主动消息 feed、日历面板在发送消息时重复刷新。
- 圆桌场景允许先显示席位和主持开场，再等待完整讨论结果。

## 五、交付物

- 一份 `jarvis_latency_benchmark` 测试脚本或手动测试清单。
- 一份固定 prompt 基准集。
- 后端 timing 日志字段。
- 前端关键交互 timing 记录。
- 一轮优化前后对比表：普通私聊、路由、长期记忆、圆桌、端到端。
