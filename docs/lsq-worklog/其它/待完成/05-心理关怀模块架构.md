5.心理关怀模块架构：从“情绪观测”到“长期趋势”再到“温和干预”的闭环

心理关怀模块应该建立在前面那套计划和多智能体事件架构之上，而不是单独做一个“心情建议弹窗”。它的本质是一个长期状态追踪系统：持续观察用户在聊天、作息、日程、任务完成、深夜使用、压力密度等方面留下的信号，把这些信号转成可解释的心理状态指标，再按天、周、月、年形成趋势，最后由关怀层在合适时机触发温和、克制、有边界的心理建议。这个模块不应该诊断用户，更不能替代医生；它的定位是陪伴、提醒、减压和帮助用户觉察。

整体可以分成七层：情绪采集层、行为采集层、压力评估层、心理状态聚合层、趋势分析层、关怀触发层、关怀交互层。情绪采集层从聊天内容中提取情绪；行为采集层记录用户最后操作时间、深夜使用、连续在线、打开/关闭程序时间；压力评估层读取日程密度、任务堆积、逾期任务和计划重排次数；心理状态聚合层把这些信号合成每天的 mood snapshot；趋势分析层生成周/月/年的心情波动图；关怀触发层判断是否需要弹出建议；关怀交互层由 Mira 或专门的 care agent 给出建议、陪伴、呼吸练习、休息提醒或求助提示。

第一条主链路是聊天情绪观测链路。每次用户发送消息后，除了保存到 `agent_chat_turns`，还应该进入一个轻量的 emotion analyzer。这个 analyzer 可以先用规则和词典做快速判断，例如疲惫、焦虑、愤怒、低落、开心、兴奋、无助、拖延、自责等；再在必要时调用 LLM 输出结构化 JSON：`primary_emotion`、`secondary_emotions`、`valence`、`arousal`、`stress_score`、`fatigue_score`、`risk_level`、`confidence`、`evidence_summary`。这些结果写入 `jarvis_emotion_observations`。注意这里保存的不是用户原文全文，而是摘要和结构化指标，原文仍在聊天表里，避免心理模块重复存储隐私内容。

第二条主链路是作息行为观测链路。用户在个人资料里已经能写入作息，例如 bedtime / wake。系统需要增加一个 activity tracker，记录用户最后操作程序时间、当天首次打开时间、最后关闭或最后活跃时间、深夜活跃时长、是否超过设定 bedtime。比如用户设置 23:00 睡觉，但连续多天 01:30 之后仍在聊天或安排任务，就写入 `jarvis_behavior_observations`：`observation_type=late_night_usage`、`expected_bedtime=23:00`、`actual_last_active_at=01:35`、`deviation_minutes=155`。这个信号本身不是心理结论，但它会进入疲劳和压力评估。

第三条主链路是日程压力观测链路。心理关怀不能只看用户说了什么，还要看用户被安排了什么。压力评估层需要读取 `jarvis_plan_days`、`calendar_events`、`jarvis_workbench_items` 和逾期/重排日志，计算当天日程密度、连续高压天数、任务逾期数量、未完成任务数量、休息窗口是否不足、晚上是否安排过重。比如一天内超过 8 小时任务、连续 3 天没有明显休息窗口、凌晨 1 点重排多次，都应该形成 `schedule_pressure_observation`。这些观测进入 `jarvis_stress_signals`，并能解释“为什么系统认为今天压力偏高”。

第四条主链路是每日心理快照链路。系统每天至少生成一条 `jarvis_mood_snapshots`，代表当天用户的综合状态。它不是单条聊天情绪，而是综合情绪、作息、日程、任务完成和历史基线后的结果。字段可以包括：`date`、`mood_score`、`stress_score`、`energy_score`、`sleep_risk_score`、`schedule_pressure_score`、`dominant_emotions`、`positive_events`、`negative_events`、`risk_flags`、`summary`、`confidence`。比如用户当天说话偏焦虑、日程很满、昨天任务未完成且凌晨还在使用程序，那么当天 snapshot 的 `stress_score` 会升高，`energy_score` 会降低，`risk_flags` 可能包含 `late_night_overuse` 和 `schedule_overload`。

第五条主链路是长期趋势链路。月度/年度心情波动图不应该前端临时算，而应该由后端提供聚合接口。后端从 `jarvis_mood_snapshots` 聚合出 `GET /care/trends?range=month|year`，返回每天/每周的 mood、stress、energy、sleep risk 和 schedule pressure。前端可以画折线图、热力图或日历热度图：横轴是日期，纵轴是心情/压力分数，颜色代表风险等级。点击某一天可以展开“这天为什么压力高”：比如“当天 5 个任务、2 个逾期、最后活跃时间 01:42、聊天中多次表达焦虑”。这样用户看到的不是冷冰冰的分数，而是可解释的自我观察。

第六条主链路是关怀触发链路。关怀层不应该每次情绪低落都打扰用户，而是使用触发规则和冷却时间。典型触发条件包括：连续 3 天 stress_score 高于阈值；连续 2 天超过 bedtime 90 分钟仍活跃；当天日程密度过高且休息窗口不足；用户聊天中出现强烈自责、无助、崩溃表达；任务连续 miss 导致重排；月度趋势显示心情持续下降。触发后写入 `jarvis_care_triggers`，包含 `trigger_type`、`severity`、`reason`、`evidence_ids`、`cooldown_until`、`status`。同类关怀必须有冷却时间，例如轻度提醒 12 小时内不重复，中度关怀 24 小时内不重复，避免系统变成新的压力源。

第七条主链路是关怀建议链路。触发关怀后，不应该直接弹出长篇说教，而应该由 Mira 或 care agent 根据触发原因生成短、柔和、可执行的建议。轻度触发可以是非阻塞 toast：“你今天的任务排得有点满，要不要留 20 分钟空档？”中度触发可以是卡片：“这两天你都在睡觉时间后很久还在使用 Jarvis，我建议今晚把最后一个任务挪到明天。”强触发可以建议用户联系信任的人或专业帮助，但必须用谨慎语言，并声明系统不能替代专业心理支持。所有建议都写入 `jarvis_care_interventions`，记录是否展示、用户是否接受、是否关闭、是否稍后提醒。

第八条主链路是主动减负链路。心理关怀模块不能只给建议，它还应该能和 planner service 互动。比如检测到日程过紧时，Mira 不能直接改日程，但可以发起一个 `care.suggest_schedule_relief` 事件给 Maxwell。Maxwell 收到后，基于 planner service 找出可移动任务，生成一个待确认的“减负调整卡”：把某个低优先级任务挪到明天、缩短某个任务、插入休息窗口。用户确认后才真正修改 `jarvis_plan_days` 和 `calendar_events`。这样心理关怀和日程系统形成闭环：不是只安慰用户，而是帮用户把压力源结构性降低。

第九条主链路是隐私和安全边界。心理关怀涉及高度敏感数据，因此必须有明确边界。用户应能在设置里开启/关闭心理关怀、选择是否分析聊天情绪、是否记录深夜使用、是否生成长期趋势、是否允许弹窗提醒。数据库中的心理观测应尽量保存结构化指标和摘要，不额外复制完整聊天文本。高风险内容只能触发支持性建议和求助提醒，不能输出诊断结论，例如不能说“你得了抑郁症”，只能说“我注意到你最近多次表达了强烈低落和无助，如果这种状态持续，建议联系信任的人或专业支持”。

第十条主链路是多智能体分工。Mira 是心理关怀主 agent，负责情绪解释、温和建议、减压练习和关怀语气；Maxwell 负责把关怀建议转成可确认的日程调整；Shadow 负责长期偏好学习，例如用户更接受文字提醒还是卡片提醒、喜欢晚上复盘还是早晨总结；其他 agent 可以贡献上下文，但不能直接写心理状态表。所有 agent 的关怀相关动作都通过 `jarvis_agent_events` 交换，例如 `care.triggered`、`care.intervention_shown`、`care.relief_requested`、`care.relief_confirmed`。

数据表可以这样设计：
1. `jarvis_emotion_observations`：保存单次聊天或文本片段的情绪识别结果，包括 `id, session_id, turn_id, agent_id, primary_emotion, valence, arousal, stress_score, fatigue_score, risk_level, confidence, evidence_summary, created_at`。
2. `jarvis_behavior_observations`：保存用户行为信号，包括 `id, observation_type, observed_at, expected_value, actual_value, deviation_minutes, payload_json, created_at`。
3. `jarvis_stress_signals`：保存日程和任务压力信号，包括 `id, signal_type, signal_date, severity, score, reason, related_plan_id, related_day_id, payload_json, created_at`。
4. `jarvis_mood_snapshots`：保存每日综合心理状态，包括 `date, mood_score, stress_score, energy_score, sleep_risk_score, schedule_pressure_score, dominant_emotions_json, risk_flags_json, summary, confidence, created_at, updated_at`。
5. `jarvis_care_triggers`：保存触发记录，包括 `id, trigger_type, severity, reason, evidence_ids_json, status, cooldown_until, created_at, resolved_at`。
6. `jarvis_care_interventions`：保存关怀建议和用户反馈，包括 `id, trigger_id, agent_id, intervention_type, content, suggested_action_json, status, user_feedback, shown_at, acted_at, created_at`。

接口可以这样设计：
1. `GET /care/snapshot/today`：返回今天的综合状态、主要风险和建议。
2. `GET /care/trends?range=month&date=2026-04`：返回月度心情、压力、能量、作息风险曲线。
3. `GET /care/trends?range=year&year=2026`：返回年度趋势和每月摘要。
4. `GET /care/interventions?status=pending`：返回待展示或待处理的关怀建议。
5. `POST /care/interventions/{id}/ack`：用户确认已读。
6. `POST /care/interventions/{id}/snooze`：稍后提醒。
7. `POST /care/interventions/{id}/accept`：接受建议；如果涉及日程调整，则转交 planner service 生成待确认调整卡。
8. `PATCH /care/settings`：开关心理关怀、情绪分析、行为追踪、趋势图和弹窗提醒。

前端展示上可以增加一个“心理关怀 / 心情趋势”面板。首页只展示轻量摘要，例如“今天压力偏高，建议留一段休息时间”；详细页展示月度折线图、年度热力图、压力来源拆解、睡眠作息偏移、日程负荷趋势。关怀弹窗应该分级：轻度用角落 toast，中度用卡片，强风险用更明显但仍然温和的提示。任何提示都应该有“我知道了”“稍后提醒”“不要再提示此类内容”“调整今天日程”这些操作，避免让关怀变成干扰。

最终用户体验应该是这样的：用户连续几天在 23:00 作息后仍然 01:30 使用 Jarvis，同时聊天里多次表达“好累”“压力好大”，日程里又连续排满学习和工作。系统不会立刻诊断用户，而是先在后台形成 emotion observation、late night behavior observation 和 schedule pressure signal。当天或次日生成 mood snapshot，发现 stress 连续升高，于是触发 care trigger。Mira 弹出一张温和卡片：“我注意到你这两天都很晚还在处理任务，而且今天安排很满。要不要我帮你留出 20 分钟恢复时间？”如果用户接受，Maxwell 生成一个待确认日程调整卡，把低优先级任务挪开并插入休息窗口。到了月底，用户可以打开趋势图，看到哪些天压力高、原因是什么、作息是否偏移、心情是否在改善。

这个模块最重要的原则是“长期、克制、可解释、可退出”。长期意味着它不是一次性聊天建议，而是持续形成趋势；克制意味着它不会频繁打扰用户；可解释意味着每个分数和触发都有证据来源；可退出意味着用户可以随时关闭任何追踪或提醒。心理关怀模块只有和会话、日程、任务、作息、多 agent 事件系统打通，才会真正有价值：它不仅知道用户说自己累，也知道用户为什么累、什么时候更容易累、哪些安排正在加重负担，以及系统能如何温和地帮用户减轻负担。
