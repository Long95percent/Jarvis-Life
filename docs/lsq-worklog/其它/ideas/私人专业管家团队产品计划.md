# 私人专业管家团队产品计划

> 创建时间：2026-04-24 20:20（Asia/Shanghai）  
> 文档位置：`ideas/私人专业管家团队产品计划.md`  
> 产品定位：个人私人专业管家团队  
> 设计原则：从产品体验出发，保留 `docs/worklog/` 中已经落地的 Jarvis / Be IronMan 架构成果，在现有角色、圆桌、主动关怀、工具、记忆和 Java 三层网关基础上升级；其中“多智能体真实协作”允许进行较大幅度架构升级，但目标是增强协作层，而不是推翻现有产品与服务分层。

## 1. 计划目标

本计划的目标，是把当前产品从“AI 工作台 / AI 助手”进一步明确为：

**一个长期陪伴用户、理解用户生活节律与心理状态、能协调多个专业角色、能低成本高效率主动关心用户的个人私人专业管家团队。**

它不是单一聊天机器人，而是一个由多个专业智能体组成的私人团队。用户面对的是一个可信赖、有秩序、有记忆、有分工、有协作能力的团队，而不是一堆孤立的功能按钮。

最终希望达成的效果：

- 用户感觉自己拥有一个“私人专业管家团队”，而不是一个普通 AI 对话框。
- 团队能理解用户长期的生活节律、心理状态、工作负荷、偏好、关系和目标。
- 团队内不同角色会主动协作，例如心理管理员向秘书了解近期日程，健康管家向生活管家了解睡眠和饮食，HR 管家判断是否需要新增角色。
- 每个角色都有清晰的人设、职责、边界、记忆和低成本信息管理方式。
- 用户能看见团队如何协作：谁询问了谁、谁提供了什么信息、最终建议如何形成。
- 系统在不大改现有架构的前提下，逐步把已有 Agent、记忆、工具、Jarvis、圆桌能力产品化。

本计划的修订原则：

- `docs/worklog/2026-04-23-daily-summary.txt` 和 `docs/worklog/2026-04-24-daily-summary.txt` 中已经完成的架构、角色、能力和验证结果，全部视为现有基础，不重新推倒。
- 已有的 Alfred、Maxwell、Nora、Mira、Leo、Shadow 角色体系应被继承，并在此基础上扩展“私人专业管家团队”。
- 已有的场景圆桌、主动关怀、偏好学习、本地生活聚合、专属工具链、共享协作记忆、SQLite 持久化、Java 三层网关和前端 Jarvis 主页应被保留。
- 真正需要大幅升级的是“多智能体协作层”：从当前的摘要型共享记忆、圆桌讨论和后台协作，升级为可路由、可追踪、可授权、可复盘、可扩展的协作智能体系统。

## 1.1 现有架构基座

根据 `docs/worklog/`，当前产品不是从零开始，已经具备一套可继承的 Jarvis / Be IronMan 基座。

### 已有产品与角色基础

- 品牌与产品形态已经从 ShadowLink 逐步转向 Jarvis / Be IronMan。
- 核心隐喻已经形成：用户是“老板”，多个专业智能体为用户服务。
- 已有 5+1 角色：Alfred 总管家、Maxwell 秘书、Nora 营养师、Mira 心理师、Leo 生活顾问、Shadow 偏好学习器。
- 角色人设、语气和说话风格已经开始分化，不再是同质化商务助手。

### 已有协作与主动能力

- 已有 Shadow Roundtable 引擎，支持后台多 Agent 讨论并产出 JSON action 决策。
- 已有 5 个场景圆桌：今日日程协调、本地生活活动、情绪压力疏导、周末恢复规划、工作头脑风暴。
- 已有多轮圆桌和用户插话能力，用户可以以“老板”身份加入讨论。
- 已有主动关怀引擎，包含压力升高、日程过载、睡眠不佳、空闲窗口、情绪下降等触发规则。
- 已有自动升级机制，私聊命中复杂/高风险场景时可升级为圆桌。

### 已有工具、记忆与落地能力

- 已有专属角色工具链，工具能真实落地到 calendar、context、journal、follow-up 等模块。
- 已有 Preference Learner / Shadow 偏好学习器，能静默观察并提取用户偏好。
- 已有本地生活聚合服务，接入新闻、附近活动、日程密度等生活上下文。
- 已有多角色 roundtable transcript 记忆和私聊共享协作记忆注入。
- 已有 SQLite 持久化，并已接回原有 Java 三层网关架构。

### 已有边界与升级缺口

- 共享协作记忆仍主要是“摘要型共享记忆”，还不是高质量长期语义记忆系统。
- 私聊后台协作已经成立，但协作过程对用户仍不够结构化、可视化和可审计。
- 圆桌偏会议形态，适合讨论；但真正的管家团队还需要任务路由、跨角色询问、授权边界、协作图谱和角色能力演化。
- Shadow 偏好学习器仍偏轻量，还没有承担团队 HR、角色招募、能力缺口识别等职责。

## 2. 产品重新定位

### 2.1 新定位

**Jarvis-Life 是一个面向个人长期生活与成长管理的私人专业管家团队。**

它关注的不只是“回答问题”，而是持续帮助用户：

- 过得更有节律。
- 情绪更稳定。
- 安排更合理。
- 决策更清晰。
- 长期目标更可执行。
- 生活、工作、心理、健康、人际、学习之间更协调。

### 2.2 和普通 AI 助手的差异

| 维度 | 普通 AI 助手 | 私人专业管家团队 |
| --- | --- | --- |
| 关系 | 一次性问答工具 | 长期陪伴型团队 |
| 组织方式 | 单一助手 | 多角色协作团队 |
| 关注重点 | 回答用户问题 | 管理用户生活质量与长期状态 |
| 记忆方式 | 主要依赖对话上下文 | 每个角色有本地长期记忆与精简摘要 |
| 主动性 | 用户问才答 | 能在合适时机主动关心和提醒 |
| 可解释性 | 只看到最终答案 | 可查看角色协作链路和任务云图 |
| 成本策略 | 每次塞入大量上下文 | 按角色加载必要信息，压缩记忆，高效调用 |

### 2.3 产品关键词

- 私人管家

- 长期记忆

- 心理陪伴
- 主动关心
- 角色协作
- 用户可控
- 低成本高职能
- 可视化协作云图

## 3. 目标用户与核心场景

### 3.1 目标用户

第一阶段建议聚焦以下用户：

1. **高压知识工作者 / 创业者 / 独立开发者**
   - 工作任务多，日程复杂，容易焦虑或节律混乱。
   - 需要有人帮助梳理安排、提醒节奏、观察状态。

2. **希望改善生活秩序的人**
   - 想管理睡眠、饮食、运动、学习、情绪和长期目标。
   - 需要长期陪伴，而不是一次性建议。

3. **高频 AI 用户**
   - 已经习惯与 AI 沟通，希望 AI 更像一个团队，而不是单一聊天助手。
   - 愿意配置角色、记忆、偏好和自动化流程。

4. **需要私人化支持但不想泄露太多隐私的人**
   - 关注本地存储、本地模型、可控记忆和可审计行为。

### 3.2 核心场景

#### 场景 A：每日节律管理

用户早上打开产品，管家团队自动汇总：

- 昨晚睡眠或用户自述状态。
- 今天日程和重要任务。
- 用户近期情绪趋势。
- 今天适合的工作强度。
- 是否需要安排休息、运动、饮食提醒。

最终输出：一份“今日生活与工作安排建议”。

#### 场景 B：心理状态关心

当用户表达疲惫、焦虑、拖延、压力大时：

- 心理管理员分析用户表达和近期记录。
- 向秘书询问近期日程压力。
- 向任务管家询问未完成任务堆积情况。
- 向健康管家询问睡眠/运动/饮食趋势。
- 形成温和、可执行、不过度诊断的关心与建议。

最终输出：安抚 + 状态判断 + 可执行小行动 + 是否需要降低今日任务强度。

#### 场景 C：生活安排与任务协调

用户说：“帮我安排一下今天。”

- 秘书接收任务。
- 向日程管家获取固定安排。
- 向任务管家获取待办优先级。
- 向心理管理员了解当前状态是否适合高压任务。
- 向生活管家确认吃饭、休息、运动窗口。
- 输出一个合理日程。

最终输出：日程表 + 优先级 + 休息安排 + 风险提醒。

#### 场景 D：团队自动扩展

用户多次提出某类需求，例如“帮我管理健身计划”“帮我记录饮食”“帮我管理财务”。

- HR 管家观察到重复需求。
- 分析现有团队是否已有角色覆盖。
- 如果缺少专职角色，提出“招募新角色”的建议。
- 用户同意后，系统生成角色配置文件、职责范围、记忆目录和协作接口。

最终输出：新增一个可管理特定功能的专业智能体。

#### 场景 E：任务协作云图

用户向秘书下达复杂任务后，可以看到：

- 秘书拆解了哪些子问题。
- 秘书询问了哪些角色。
- 这些角色又调用了哪些其他角色或工具。
- 每个节点产出了什么中间结论。
- 最终建议如何合成。

最终输出：一个可点击、可追溯的角色协作关系图。

## 4. 最终产品效果设计

### 4.1 用户看到的不是“模型”，而是“团队”

产品首页应逐步从普通聊天框升级为团队工作台：

- 顶部显示当前值班总管 / 秘书。
- 左侧显示团队成员：秘书、心理管理员、日程管家、健康管家、生活管家、任务管家、HR 管家等。
- 中间是主对话区。
- 右侧是用户状态面板：今日节律、情绪趋势、任务压力、近期关注事项。
- 底部或独立视图显示任务协作云图。

### 4.2 用户能自由修改每个角色

每个智能体都应有本地配置文件，用户可以修改：

- 角色名称。
- 人设与性格。
- 说话风格。
- 工作职责。
- 不能做什么。
- 可询问哪些角色。
- 可调用哪些工具。
- 记忆保存规则。
- 主动关心频率。
- 成本预算和模型偏好。

用户也可以通过自然语言说：

- “心理管理员以后说话更温柔一点。”
- “秘书不要太啰嗦，只给我重点。”
- “健康管家不要每天提醒我体重，除非我主动问。”

系统应把这些建议沉淀到对应角色的配置或偏好记忆中。

### 4.3 用户能信任它不会过度打扰

主动关心需要非常克制：

- 不要频繁弹窗。
- 不要制造焦虑。
- 不要未经允许做重大判断。
- 不要假装具备医疗诊断能力。
- 对心理健康相关内容要温和、支持性表达，并在必要时建议寻求专业帮助。

理想体验是：用户感觉“它很懂我，也很有边界”。

## 5. 角色体系设计

### 5.1 三层协作结构

在不大改当前架构的前提下，建议采用三层结构：

#### 第一层：总管 / 调度层

角色：`总管 Agent` 或 `秘书长 Agent`

职责：

- 作为用户任务的默认入口。
- 理解用户意图。
- 判断需要哪些角色参与。
- 拆解任务。
- 汇总各角色结果。
- 控制成本和上下文大小。
- 决定是否需要询问用户确认。
- 记录任务协作过程。

注意：总管不应亲自做所有专业判断，而是负责协调。

#### 第二层：专业管家层

专业角色按需启用，不需要每次全部加载。

建议初始团队：

| 角色 | 核心职责 | 典型协作对象 |
| --- | --- | --- |
| 秘书 / 总助 | 接收任务、整理安排、生成日程建议、对接用户 | 所有角色 |
| 心理管理员 | 关注情绪、压力、动力、拖延、内耗和心理安全 | 秘书、日程管家、任务管家、健康管家 |
| 日程管家 | 管理日历、时间块、提醒、节奏安排 | 秘书、任务管家、生活管家 |
| 任务管家 | 管理待办、目标、优先级、执行进度 | 秘书、心理管理员、日程管家 |
| 健康管家 | 关注睡眠、运动、饮食、身体状态 | 生活管家、心理管理员、日程管家 |
| 生活管家 | 管理生活杂事、居家、出行、购物、日常安排 | 秘书、健康管家、日程管家 |
| 学习/成长管家 | 管理学习计划、阅读、技能成长 | 任务管家、心理管理员 |
| 关系管家 | 帮助维护重要人际关系、纪念日、沟通提醒 | 秘书、心理管理员、日程管家 |
| HR 管家 | 评估团队能力缺口、建议新增/调整角色 | 总管、所有角色 |
| 记忆管理员 | 维护长期记忆质量、摘要、归档、去重 | 所有角色 |

### 5.2 HR 管家的特殊设计

HR 管家不是现实意义上的人事，而是“团队能力设计师”。

它负责：

- 观察用户重复提出的需求。
- 判断当前团队是否覆盖这个需求。
- 判断是否应该新增角色，还是修改现有角色职责。
- 生成新角色草案。
- 请求用户确认。
- 用户同意后创建角色配置文件和角色目录。

HR 管家不能自动无限扩张团队，必须有约束：

- 新增角色必须用户同意。
- 每次新增角色要说明理由、成本和预期收益。
- 同类角色应优先合并，避免团队膨胀。
- 角色数量超过一定阈值时，HR 应建议合并或停用低频角色。

### 5.3 角色之间的关系

角色之间不应是完全孤立的。建议用“允许协作矩阵”管理。

示例：

| 发起角色 | 可询问角色 | 允许询问的信息 |
| --- | --- | --- |
| 心理管理员 | 秘书 | 近期日程、今日压力、重要事件 |
| 心理管理员 | 任务管家 | 任务堆积、延期、优先级冲突 |
| 心理管理员 | 健康管家 | 睡眠、运动、饮食、身体疲劳迹象 |
| 秘书 | 日程管家 | 可用时间、冲突安排、提醒窗口 |
| 秘书 | 生活管家 | 通勤、吃饭、生活杂事安排 |
| HR 管家 | 所有角色 | 角色能力边界、频繁无法处理的请求 |
| 记忆管理员 | 所有角色 | 记忆摘要、过期信息、重复信息 |

每次跨角色询问都应生成协作事件，用于后续云图展示。

## 6. Agent 架构选择建议

你提到 subagent 和 multiagent 的架构选择，这里建议不要走极端。

同时结合 `docs/worklog/` 中已经落地的能力，当前系统已经不是“没有多智能体”的状态，而是已经具备：角色 roster、圆桌、私聊后台协作、专属工具链、主动触发、共享协作记忆和 Java 网关集成。所以下一步不是简单“新增 multiagent”，而是把现有多角色能力升级为真正的“协作智能体架构”。

### 6.1 不建议所有角色全量 multiagent 常驻

原因：

- Token 成本高。
- 协作链路容易失控。
- 每个任务都全员讨论会变慢。
- 容易产生重复建议和互相污染上下文。

### 6.2 不建议只有单 Agent + 工具

原因：

- 很难形成真实的专业分工。
- 角色个性和长期记忆难以独立维护。
- 用户很难感受到“专业管家团队”。

### 6.3 推荐架构：Supervisor + On-demand Subagents + Event Graph

推荐采用：

**一个总管调度 Agent + 按需唤醒的专业 Subagents + 统一事件图谱记录协作过程。**

特点：

- 用户入口统一，不混乱。
- 专业角色按需调用，节省成本。
- 每个角色有独立 prompt、记忆、配置和职责边界。
- 总管只给角色传递必要摘要，不传递全量上下文。
- 所有协作事件写入任务图，用于可视化。
- 简单任务只调用一个角色，复杂任务再多角色协作。

### 6.3.1 基于现有架构的协作层大升级

在不推翻现有 Java 后端、Python AI 服务、前端 Jarvis 页和已有角色体系的前提下，多智能体协作层允许做较大升级。建议新增一个“Team Collaboration Layer”，位于现有 `jarvis` 能力和底层 `agent/tools/memory` 之间。

目标不是让每个角色自由乱聊，而是建立一套可控协作协议：

```text
用户 / 前端 Jarvis
  ↓
Alfred / 总管调度层
  ↓
Team Collaboration Layer
  ├─ Agent Registry：角色注册、能力、状态、预算、权限
  ├─ Task Router：任务理解、角色选择、协作模式选择
  ├─ Collaboration Protocol：角色间询问、回应、委托、升级
  ├─ Shared Context Bus：只传必要摘要和授权信息
  ├─ Memory Broker：按角色/任务/全局策略读取与写入记忆
  ├─ Graph Recorder：记录节点、边、事件、中间结论
  └─ Governance Guard：隐私、心理安全、成本、用户确认
  ↓
专业角色 Agents + 专属工具链 + 现有 Roundtable / Proactive / Preference 能力
```

这意味着：

- Alfred 仍然是总管，但不再把所有协作逻辑硬写在 prompt 或单个流程里。
- Maxwell、Nora、Mira、Leo 等现有角色保留，并接入统一协作协议。
- Shadow 从轻量偏好学习器升级为“观察者 + 偏好学习 + 团队 HR 建议”的候选基础，但初期仍需保持低打扰。
- Roundtable 保留为一种“会议型协作模式”，但不再承担所有复杂任务；日常任务更多走轻量的“点对点询问 / 委托 / 汇总”模式。
- 每次协作都必须生成结构化事件，未来用于协作云图。

### 6.3.2 协作模式升级

建议把多智能体协作拆成 5 种模式，而不是所有情况都开圆桌：

| 协作模式 | 说明 | 适用场景 |
| --- | --- | --- |
| Direct | 单角色直接回答 | 简单问题、角色私聊 |
| Consult | 主角色向其他角色询问信息 | 心理管理员询问秘书近期日程 |
| Delegate | 主角色把子任务委托给其他角色 | 秘书让日程管家找空档 |
| Roundtable | 多角色共同讨论并达成共识 | 复杂决策、冲突权衡、情绪+日程+健康综合问题 |
| Escalate / Recruit | 升级给用户确认或 HR 建议新增角色 | 高风险心理内容、能力缺口、新工具需求 |

当前 worklog 中已有 Roundtable、自动升级和私聊后台协作；下一步要补强的是 Consult、Delegate、Graph Recorder、Memory Broker 和 Governance Guard。

### 6.3.3 当前确认的架构取舍：保留现有基座，新增轻量协作层

经过对比，当前确认不推翻已有架构。现有 Jarvis / ShadowLink 架构在记忆、工具、圆桌、主动关心和调试可控性上已经有效，且更省 token；新设计的多智能体团队架构作为长期上限，应该以“增量协作层”的方式接入。

确认原则：

- 默认单 Agent 或单角色工具调用，简单任务不进入多智能体。
- 中等复杂任务采用轻量协作，例如 Alfred 只询问 Maxwell / Mira / Nora / Leo 中必要的 1-2 个角色。
- 复杂任务才进入 Roundtable 或 Deep Team 协作。
- 圆桌保留，但不作为所有复杂任务的唯一入口。
- 多智能体升级重点是 Task Router、Collaboration Protocol、Context Broker、Graph Recorder，而不是重写 Java / Python / React 三层结构。
- token 成本控制优先级高于“全员参与的表面智能感”。真正专业的管家团队应当克制调度：该一个人办就一个人办，该询问专家时才询问专家。

成本判断：

| 架构形态 | 智能表现 | Token 成本 | 当前建议 |
| --- | --- | --- | --- |
| 现有单 Agent / 工具架构 | 稳定、直接 | 最低 | 保留为默认路径 |
| 现有圆桌 | 有团队感 | 中等 | 保留为会议型模式 |
| 轻量协作层 | 专业且可控 | 较低到中等 | 作为下一阶段主升级 |
| 完整 multi-agent 常驻 | 上限高但复杂 | 高 | 后置，仅用于复杂任务 |
| 所有 Agent 默认参与 | 看似热闹但低效 | 最高 | 明确不采用 |

### 6.4 何时使用哪种协作模式

| 任务类型 | 推荐模式 | 示例 |
| --- | --- | --- |
| 简单问答 | 单角色响应 | “今天我该先做什么？” |
| 日程安排 | 总管 + 秘书 + 日程/任务 | “帮我安排今天。” |
| 情绪支持 | 心理管理员 + 必要信息查询 | “我最近很焦虑。” |
| 复杂决策 | 圆桌 / 多 Agent | “我要不要换工作？” |
| 团队扩展 | HR 管家 + 用户确认 | “我想长期管理健身。” |
| 记忆整理 | 记忆管理员后台任务 | 每日/每周摘要归档 |

## 6.5 多智能体协作升级的验收效果

这部分是本计划中允许较大架构升级的重点。完成后应达到：

- 角色之间不是“共享同一段 transcript”，而是可以明确地询问、委托、回应和引用彼此结论。
- 每个角色知道自己为什么被调用、需要回答什么、能访问哪些信息、不能越界做什么。
- 总管可以根据任务类型选择 Direct / Consult / Delegate / Roundtable / Escalate。
- 每次跨角色协作都会形成结构化事件：`who asked whom, why, with what context, got what answer`。
- 用户可以在协作云图里看到协作过程，而不是只能看到最终回复。
- 系统能追踪哪些角色经常被调用、哪些能力缺口反复出现，为 HR 管家“招人”提供依据。

## 7. 角色配置与 Prompt 工程设计

### 7.1 每个角色一个本地配置文件

建议每个角色本地拥有独立目录：

```text
data/agents/
  secretary/
    profile.yaml
    prompt.md
    memory/
      long_term.md
      working_summary.md
      preferences.json
      facts.json
    inbox/
    logs/
  psychology_manager/
    profile.yaml
    prompt.md
    memory/
    inbox/
    logs/
```

为了不大改当前架构，第一阶段可以先用 Markdown / JSON / YAML 文件实现，不急着引入复杂数据库。

### 7.2 角色配置字段

`profile.yaml` 建议包含：

```yaml
id: psychology_manager
name: 心理管理员
version: 1
status: active
personality:
  tone: 温柔、稳定、尊重边界
  style: 简洁、支持性、少评判
mission: 关注用户情绪、压力、动力、拖延和心理安全
responsibilities:
  - 识别用户近期心理状态变化
  - 在必要时向秘书、任务管家、健康管家询问信息
  - 给出温和、可执行、非医疗诊断式建议
boundaries:
  - 不做医学诊断
  - 不替代心理医生
  - 遇到严重风险时建议用户联系现实中的专业支持
collaboration:
  can_ask:
    - secretary
    - schedule_manager
    - task_manager
    - health_manager
memory_policy:
  save:
    - 长期压力源
    - 用户明确表达的重要偏好
    - 反复出现的情绪模式
  avoid_saving:
    - 一次性情绪宣泄细节
    - 不必要的隐私细节
budget:
  default_model: low_cost
  max_context_items: 8
  summary_first: true
```

### 7.3 Prompt 结构

每个角色的 prompt 不应该是一大段不可维护文本，而应模块化：

1. 你是谁。
2. 你服务谁。
3. 你的职责是什么。
4. 你的性格和表达方式。
5. 你可以询问谁。
6. 你可以调用哪些工具。
7. 你必须遵守哪些边界。
8. 你什么时候应该保存记忆。
9. 你什么时候应该拒绝或升级给总管。
10. 你输出时应多简洁。

### 7.4 用户可修改性

产品上应提供“角色编辑器”：

- 基础模式：用户通过自然语言调整角色。
- 高级模式：用户直接编辑配置文件。
- 修改预览：展示本次修改会影响哪些行为。
- 回滚能力：角色配置保留历史版本。

## 8. 长期记忆设计

### 8.1 记忆目标

长期记忆不是把所有对话都保存，而是保存“未来会反复有用的信息”。

记忆应帮助系统：

- 记住用户偏好。
- 记住用户重要目标。
- 记住生活节律。
- 记住常见压力源。
- 记住重要关系和事件。
- 记住角色自己的工作经验。

### 8.2 记忆分层

建议采用四层记忆：

| 记忆类型 | 内容 | 保存位置 | 调用方式 |
| --- | --- | --- | --- |
| 用户档案 | 基础偏好、长期目标、生活节律 | 全局用户记忆 | 总管和授权角色可读 |
| 角色记忆 | 角色领域内的重要事实和摘要 | 各角色目录 | 仅该角色默认读取 |
| 任务记忆 | 某次任务过程、协作图、结论 | task graph / logs | 用户查看或复盘时读取 |
| 临时上下文 | 当前会话短期信息 | 运行时 | 任务结束后摘要或丢弃 |

### 8.3 记忆压缩策略

为了低 token 成本，每个角色需要自动维护高精简信息：

- 每次任务结束后只保存摘要，不保存完整冗余对话。
- 把事实、偏好、模式、待确认信息分开保存。
- 定期由记忆管理员去重、过期、合并。
- 调用角色前先取“工作摘要”，必要时再检索详细记忆。
- 每个角色设置最大上下文条数和摘要长度。

### 8.4 记忆写入必须谨慎

记忆写入建议分级：

| 等级 | 规则 |
| --- | --- |
| 自动保存 | 明确稳定的低敏偏好，如“用户喜欢简洁回答”。 |
| 待确认保存 | 涉及心理、健康、关系、身份等敏感信息。 |
| 不保存 | 一次性宣泄、临时情绪、无长期价值隐私。 |

产品上可以让用户看到：

- 今天系统记住了什么。
- 哪些记忆待确认。
- 可以删除、编辑或禁止某类记忆。

## 9. 低成本高职能策略

### 9.1 成本控制原则

- 默认不全员讨论。
- 默认不加载完整历史。
- 默认先用摘要，再按需检索详情。
- 简单任务用低成本模型。
- 复杂心理/规划/决策任务再使用更强模型。
- 角色输出默认短而有用。
- 每个角色每次只回答自己职责范围内的问题。

### 9.2 任务路由策略

总管接到任务后先判断：

1. 用户意图是什么？
2. 是否需要多角色？
3. 哪些角色是必要角色？
4. 哪些角色只是可选角色？
5. 是否需要工具？
6. 是否需要用户授权？
7. 是否需要写入记忆？

然后生成一个轻量执行计划。

### 9.3 信息包机制

角色之间不要互相传完整聊天记录，而是传“信息包”：

```json
{
  "task_id": "task_20260424_001",
  "from": "secretary",
  "to": "psychology_manager",
  "question": "用户今天任务较多，是否建议降低安排强度？",
  "context_summary": "用户昨晚睡得较晚，今天有三个高优先级任务，刚表达有些焦虑。",
  "requested_output": "给出不超过 5 条建议。"
}
```

这样可以减少 token，也方便可视化。

## 9.5 日程安排与日历板块需求

日程安排是私人管家团队的核心落地场景。凡是用户明确要求“安排、规划、添加、调整、移动、删除、提醒、排期”的事项，系统不应只在聊天里给建议，还应当写入日程安排板块，形成可查看、可编辑、可复盘的真实安排。

### 9.5.1 产品目标

- 聊天中的日程建议可以沉淀为真实日程，而不是停留在对话文本。
- 用户可以在日程板块看到每天一整天的安排。
- 日程板块可以像日历一样打开，支持按日、按周、按月查看和翻页。
- 管家团队在安排时间时，应结合用户已有日程、精力状态、通勤时间、恢复窗口和任务优先级。
- 后续如果涉及路线规划，可在日程详情里展示大概路线图；路线规划图属于后续增强，不作为第一阶段硬性要求。

### 9.5.2 触发规则

以下类型应进入日程板块：

- 用户明确说“帮我安排 / 加到日程 / 设个提醒 / 排一下时间”。
- Alfred 或 Maxwell 给出具体时间块，并获得用户确认。
- 主动关心发现日程过载后，用户同意系统调整日程。
- 圆桌或协作任务最终产出可执行时间安排。

以下类型默认不写入日程：

- 只是泛泛建议，例如“你可以下午休息一下”。
- 用户没有确认的敏感安排。
- 模型不确定具体时间、地点、事项的内容。
- 一次性讨论中未形成明确行动项的内容。

### 9.5.3 日历视图能力

日程安排板块应逐步具备：

- 今日视图：从早到晚展示一天安排、空闲窗口、冲突和恢复时间。
- 日视图：用户可以切换任意日期，查看当天完整时间线。
- 周视图：查看一周任务密度、压力高峰和空闲窗口。
- 月视图：查看长期节奏、重要节点、周期性安排。
- 翻页能力：上一天 / 下一天，上一周 / 下一周，上一月 / 下一月。
- 详情弹窗：点击某个日程可查看来源、创建它的 Agent、相关对话、备注和可修改项。
- 状态标记：待确认、已确认、已完成、已延期、冲突、建议调整。

### 9.5.4 与管家团队的关系

- Maxwell 是日程安排的主要负责人。
- Alfred 负责判断是否需要把聊天建议升级为日程操作，并向用户确认。
- Mira 可以参考日程密度判断心理压力，但不应直接改日程，除非用户授权。
- Nora 可以建议恢复、睡眠、饮食相关时间块。
- Leo 可以为活动、出行、生活事项提供建议。
- Shadow 可以学习用户对日程安排的偏好，例如喜欢上午做深度工作、晚上不安排高压任务。

### 9.5.5 数据与可解释性

每个由 Agent 创建或修改的日程都应保存：

- 标题、开始时间、结束时间、地点、备注。
- 来源：用户直接创建 / Alfred 建议 / Maxwell 安排 / 圆桌结论 / 主动关心。
- 创建原因：为什么建议这个时间。
- 关联任务或对话。
- 是否需要用户确认。
- 是否涉及路线规划。

### 9.5.6 路线规划后续增强

如果日程包含地点或连续地点移动，后续可以增加路线规划能力：

- 根据地点顺序生成大概路线。
- 展示通勤时间估算。
- 在日程中预留出行缓冲。
- 在地图或简化路线图中展示当天移动路径。
- 当路线过紧时提醒用户调整安排。

第一阶段不要求完整地图能力，可以先做“地点 + 通勤时间 + 缓冲建议”的文本化路线计划。

## 10. 主动关心设计

### 10.1 主动关心的触发条件

系统可以在以下情况下主动关心：

- 用户连续高强度工作。
- 用户长期未完成重要任务。
- 用户多次表达焦虑、疲惫、拖延。
- 用户日程中出现密集安排。
- 用户长期未休息或生活节律异常。
- 用户设置了主动提醒偏好。

### 10.2 主动关心的表达原则

- 简短。
- 温柔。
- 具体。
- 可选择。
- 不评判。
- 不制造焦虑。
- 不越界诊断。

示例：

> 我注意到你今天安排比较满，而且刚才提到有点累。要不要我帮你把下午任务压缩成 2 个重点，并留出 20 分钟恢复时间？

### 10.3 用户控制权

用户必须能设置：

- 哪些角色可以主动提醒。
- 提醒频率。
- 安静时段。
- 哪些主题不要提醒。
- 只在严重冲突时提醒。
- 完全关闭主动关心。

## 11. 协作云图设计

### 11.1 产品目标

协作云图让用户看到“管家团队是如何完成任务的”。

用户可以直观看到：

- 任务从哪里开始。
- 哪个角色参与。
- 谁问了谁。
- 调用了哪些工具。
- 每个节点输出了什么。
- 最终结论由哪些信息支撑。

### 11.2 云图节点类型

| 节点类型 | 示例 |
| --- | --- |
| 用户指令 | “帮我安排今天” |
| 总管节点 | 任务理解、拆解、汇总 |
| 角色节点 | 秘书、心理管理员、健康管家 |
| 工具节点 | 日历、文件、RAG、搜索、本地资源 |
| 记忆节点 | 用户偏好、近期状态、历史摘要 |
| 决策节点 | 是否需要询问用户、是否保存记忆 |
| 输出节点 | 最终建议、日程安排、提醒 |

### 11.3 云图交互

用户点击节点后可以查看：

- 节点名称。
- 角色身份。
- 收到的问题。
- 使用的上下文摘要。
- 输出结果。
- 是否写入记忆。
- token / 成本估算。
- 耗时。

### 11.4 第一阶段实现建议

不需要一开始做复杂图数据库。可以先把每次任务的协作过程保存为 JSON：

```json
{
  "task_id": "task_20260424_001",
  "title": "安排今天",
  "nodes": [
    {"id": "user", "type": "user", "label": "用户请求"},
    {"id": "secretary", "type": "agent", "label": "秘书"},
    {"id": "schedule", "type": "agent", "label": "日程管家"}
  ],
  "edges": [
    {"from": "user", "to": "secretary", "label": "下达任务"},
    {"from": "secretary", "to": "schedule", "label": "询问可用时间"}
  ],
  "events": []
}
```

前端再用关系图组件渲染即可。

## 12. 用户体验流程设计

### 12.1 新用户初始化

首次使用时，不要让用户配置太多。

建议流程：

1. 选择使用目标：生活节律 / 工作安排 / 心理陪伴 / 学习成长 / 全部。
2. 选择管家团队风格：温柔陪伴型、专业高效型、严格督促型、自定义。
3. 初始化 3 个核心角色：秘书、心理管理员、任务/日程管家。
4. 询问是否开启主动关心。
5. 生成第一版用户档案。

### 12.2 日常使用流程

1. 用户向秘书或总管说需求。
2. 总管判断是否需要多角色。
3. 必要角色按需读取自己的精简记忆。
4. 角色之间通过信息包协作。
5. 总管汇总结果。
6. 展示最终回复和协作云图。
7. 询问是否保存重要记忆或调整角色偏好。

### 12.3 角色成长流程

1. 用户给角色反馈。
2. 角色生成自我调整建议。
3. 用户确认。
4. 写入角色配置版本。
5. 后续输出风格和行为发生变化。

## 13. 基于现有架构的升级方式

### 13.1 必须保留和复用的现有能力

当前仓库已有以下基础，可以优先复用：

- `shadowlink-ai/app/agent/`：Agent、ReAct、Plan-Execute、多 Agent 能力。
- `shadowlink-ai/app/jarvis/`：Jarvis 场景、偏好、主动引擎、圆桌、上下文总线。
- `shadowlink-ai/app/agent/memory/`：短期、长期、语义、情节记忆。
- `shadowlink-ai/app/tools/`：系统工具、文件工具、本地搜索、知识搜索、网页搜索。
- `shadowlink-web/src/components/agent/`：Agent 思考过程、工具调用卡片，可扩展为协作云图。
- 根目录 PyQt 版：可先作为 MVP 展示角色配置、对话、任务和本地资源。

同时必须保留 `docs/worklog/` 已经验证过的产品能力：

- Alfred / Maxwell / Nora / Mira / Leo / Shadow 的现有角色基础。
- 5 个场景圆桌和多轮用户插话体验。
- 主动关怀触发规则、冷却机制和 interrupt budget。
- 自动升级机制，从私聊升级到圆桌。
- 专属角色工具链和工具真实落地能力。
- 共享协作记忆注入。
- Java 三层网关与 Jarvis API Router 的集成方式。
- 前端 Jarvis 主页、设置页、Zustand Store 和已有 agent UI 组件。

### 13.2 允许大幅升级的部分：多智能体协作层

用户明确希望多智能体协作真正成熟，因此这一层不应只是小修小补。允许新增结构化协作模块，但要以“加一层协作中台”的方式做，而不是推翻现有服务。

建议新增模块：

```text
shadowlink-ai/app/jarvis/team/
  registry.py              # 角色注册表：角色能力、权限、状态、预算
  router.py                # 任务路由：判断调用哪个角色、采用哪种协作模式
  protocol.py              # 协作协议：ask / delegate / respond / escalate / recruit
  collaborator.py          # 协作执行器：驱动多角色按协议协作
  context_broker.py        # 上下文代理：只分发必要摘要和授权信息
  memory_broker.py         # 记忆代理：统一读取/写入全局、角色、任务记忆
  memory_policy.py         # 记忆策略：敏感信息确认、摘要、过期、去重
  graph_recorder.py        # 协作云图事件记录
  governance.py            # 成本、隐私、心理安全、权限与用户确认
  hr_manager.py            # HR 管家：能力缺口分析、角色草案、招募建议

data/agents/
  alfred/
  maxwell/
  nora/
  mira/
  leo/
  shadow/
  custom_agents/

data/team_graphs/

```

这些模块的定位：

- 不替代现有 `roundtable_sessions.py`，而是把 roundtable 作为一种协作模式纳入统一路由。
- 不替代现有 `preference_learner.py`，而是让 Shadow 的偏好学习结果被 Memory Broker 和 HR 管家使用。
- 不替代现有工具链，而是通过角色权限和任务协议更精确地调用工具。
- 不强制一次性迁移所有功能，可以先让 Alfred + Maxwell + Mira 跑通结构化协作。

## 14. 分阶段路线图

### 14.1 Phase 0：产品定义与信息架构

目标：把 `docs/worklog/` 已有 Jarvis / Be IronMan 成果正式定义为“私人专业管家团队”的第一版基座，而不是重新规划一套新产品。

需求：

- 更新产品定位文档，说明新定位不是替代已有 Jarvis，而是升级已有 Jarvis。
- 整理现有角色 Alfred、Maxwell、Nora、Mira、Leo、Shadow 的职责、工具、记忆和边界。
- 明确哪些能力已经完成，哪些属于升级目标。
- 明确角色配置格式。
- 明确记忆策略，特别是共享协作记忆向长期语义记忆升级的路径。
- 明确协作云图数据结构。
- 明确主动关心边界。

验收标准：

- 有清晰的产品说明。
- 有基于现有 5+1 角色的职责边界。
- 有可执行的 MVP 范围。

### 14.2 Phase 1：现有团队协作结构化

目标：不是从零做最小团队，而是把当前已经存在的团队协作结构化，让它从“能协作”升级为“可追踪、可解释、可复盘地协作”。

核心角色沿用现有 5+1：

- Alfred：总管家 / 调度者。
- Maxwell：秘书 / 日程任务协调。
- Nora：营养与恢复。
- Mira：心理支持。
- Leo：生活活动与本地生活。
- Shadow：偏好学习与观察者。

核心功能：

- 保留 Alfred 作为默认总管入口。
- 为 Alfred → Maxwell / Mira / Nora / Leo 的后台协作建立结构化协议。
- 将当前“共享协作记忆注入”升级为“任务协作事件 + 摘要记忆”。
- 每次跨角色询问都生成协作 JSON。
- 用户先能查看文字版协作链路。
- 每个角色逐步迁移到本地配置文件。
- 每个角色有精简长期记忆文件。

不做：

- 不做复杂自动招募，只保留 HR 管家设计。
- 不做完整图形化云图，先做 JSON + 文字链路。
- 不做全自动生活数据采集，继续复用现有本地生活聚合。
- 不做高风险心理干预。

验收标准：

- 用户说“帮我安排今天”，Alfred 能按协议询问 Maxwell，并按需询问 Mira / Nora / Leo。
- 用户说“我最近很焦虑”，Mira 能向 Maxwell 获取日程压力、向 Shadow/记忆获取近期偏好或状态摘要。
- 用户能看到本次任务的协作链路。
- 协作链路中能看到每个角色被调用的原因和简短结论。

### 14.3 Phase 2：协作云图与主动关心升级

目标：在已有剧场式圆桌、主动关怀和前端 Jarvis 主页基础上，把“协作过程”可视化，把“主动关心”做得更准确、更克制。

核心功能：

- 协作 JSON 渲染为关系图。
- 点击节点查看角色中间输出。
- 主动关心偏好设置，继承已有 agent 开关、预算滑条和 Shadow 总开关。
- 基于已有 stress_spike / schedule_overload / sleep_poor / free_window_detected / mood_declining 触发规则继续升级。
- 记忆写入确认机制。
- 圆桌会议厅继续保留，作为复杂任务的可视化形态；协作云图用于展示更细的角色关系链路。

验收标准：

- 每个复杂任务都可查看协作云图。
- 主动提醒可关闭、可调频率。
- 敏感记忆写入前会请求用户确认。

### 14.4 Phase 3：HR 管家与角色自动扩展

目标：在不破坏现有 5+1 核心团队的前提下，让团队具备“发现能力缺口并建议招人”的能力。

核心功能：

- HR 管家分析重复需求。
- 判断新增角色或扩展现有角色。
- 自动生成角色草案。
- 用户确认后创建本地角色目录与配置。
- 新角色加入协作注册表。
- HR 的判断依据来自：协作云图、无法处理的任务、用户重复请求、角色自报能力缺口、Shadow 偏好学习结果。

验收标准：

- 当用户多次提出同类需求，HR 能提出新增角色建议。
- 用户同意后，系统创建角色配置并可被总管调用。
- 用户可停用、合并或删除角色。

### 14.5 Phase 4：高级个性化与长期成长

目标：让团队真正长期适配用户。

核心功能：

- 每周生活节律总结。
- 用户状态趋势分析。
- 角色自我复盘与调整建议。
- 低频角色自动休眠。
- 记忆质量评分。
- 团队能力面板。

验收标准：

- 用户能看到“这个团队这周帮我管理了什么”。
- 系统能提出角色调整建议。
- 长期记忆越来越精简，而不是越来越臃肿。

## 16. 产品风险与边界

### 16.1 心理健康边界

心理管理员必须明确边界：

- 不能做医学诊断。
- 不能替代心理咨询师或医生。
- 不能在危机情况下只给 AI 建议。
- 如果用户表达自伤、伤害他人或严重危机，应建议立即联系现实中的可信赖人员或专业机构。

### 16.2 隐私边界

产品会涉及生活、心理、健康、关系等敏感信息，所以必须：

- 默认本地保存。
- 允许用户删除任何记忆。
- 敏感记忆写入前确认。
- 角色只能访问职责相关的信息。
- 协作云图可隐藏敏感节点。

### 16.3 自动化边界

系统不应未经用户同意：

- 新增角色。
- 保存敏感长期记忆。
- 执行外部高风险操作。
- 发送消息给真实联系人。
- 下单、支付或做不可逆操作。

## 17. 还能进一步优化的产品想法

### 17.1 团队仪表盘

展示：

- 今日用户状态。
- 当前活跃角色。
- 最近团队做过的事。
- 待用户确认的记忆。
- 待用户确认的角色调整。
- 本周节律趋势。

### 17.2 角色关系编辑器

用户可以可视化设置：

- 谁可以问谁。
- 谁不能访问哪些信息。
- 哪些角色需要用户授权才能参与。
- 哪些角色默认沉默，只在被问到时工作。

### 17.3 管家团队周报

每周自动生成：

- 本周完成了什么。
- 哪些任务拖延了。
- 情绪和节律有什么变化。
- 下周建议如何安排。
- 团队建议新增/调整哪些能力。

### 17.4 角色人格市场 / 模板

未来可以内置角色模板：

- 温柔心理陪伴型。
- 高效秘书型。
- 严格教练型。
- 学习导师型。
- 健康管家型。
- 创业合伙人型。

用户可一键启用，再本地个性化修改。

### 17.5 低成本模式与高级模式

用户可选择：

- 省钱模式：少角色、短输出、低成本模型。
- 平衡模式：按需多角色。
- 深度模式：多 Agent 圆桌、强模型、更完整分析。

## 18. 成功指标

### 18.1 产品体验指标

- 用户是否愿意每天打开。
- 用户是否愿意告诉它真实状态。
- 用户是否觉得它“懂我”。
- 用户是否能从建议中直接行动。
- 用户是否愿意保留和调整角色。

### 18.2 功能指标

- 复杂任务平均调用角色数可控。
- 角色协作链路可追溯率。
- 长期记忆有效率。
- 用户确认保存的记忆数量。
- 主动关心被用户接受的比例。
- 角色配置被用户修改后的满意度。

### 18.3 成本指标

- 单次简单任务 token 成本。
- 多角色任务 token 成本。
- 角色记忆摘要长度。
- 无效角色调用比例。
- 重复上下文传递比例。

## 19. 总结

这个产品最重要的不是“有多少功能”，而是能不能形成一种可信赖的团队感：

- 有人负责接待你。
- 有人关心你的状态。
- 有人安排你的时间。
- 有人管理你的任务。
- 有人维护你的记忆。
- 有人判断团队还缺什么能力。
- 所有人协作时，你能看见过程，也能控制边界。

因此，下一步不建议推翻现有产品与服务架构，而是保留 `docs/worklog/` 已经形成的 Jarvis / Be IronMan 基座；但对于多智能体协作层，需要进行一次较大幅度升级，把现有圆桌、私聊后台协作、共享协作记忆和专属工具链整合成“角色注册 + 任务路由 + 协作协议 + 记忆代理 + 协作事件图谱 + 权限治理”的真正协作智能体系统。

只要第一版能让用户真实感受到“我不是在和一个 AI 聊天，而是在被一个专业私人管家团队服务”，这个方向就成立。
