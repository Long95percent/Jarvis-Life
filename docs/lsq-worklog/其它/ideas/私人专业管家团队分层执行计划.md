# 私人专业管家团队分层执行计划

> 创建时间：2026-04-24 22:00（Asia/Shanghai）  
> 文档位置：`ideas/私人专业管家团队分层执行计划.md`  
> 关联文档：`ideas/私人专业管家团队产品计划.md`、`docs/worklog/2026-04-23-daily-summary.txt`、`docs/worklog/2026-04-24-daily-summary.txt`  
> 当前原则：先修通 Agent 对话，再分层升级；保留已有 Jarvis / Be IronMan 基座，但对多智能体协作层做系统性升级。

## 1. 总体判断

私人专业管家团队不能一蹴而就。当前产品已经有 Alfred、Maxwell、Nora、Mira、Leo、Shadow、圆桌、主动关怀、专属工具链、共享协作记忆、前端 Jarvis 主页和 Java 网关等基础，因此不应该推倒重来。

但当前系统要变成真正成熟的“私人专业管家团队”，必须按层推进：

1. 先保证 Agent 对话链路稳定可用。
2. 再把已有角色能力稳定下来。
3. 再把多智能体协作从“能讨论”升级为“可路由、可追踪、可授权、可复盘”。
4. 再做长期记忆、角色配置、协作云图、HR 招募和主动关心优化。

本计划把后续工作拆成 7 层，每层都有目标、任务和验收标准。

## 2. 第 0 层：Debug 与稳定性基线

### 2.1 目标

在任何产品升级之前，先让 Agent 对话跑通，并且失败时能明确知道失败原因。

### 2.2 当前已定位问题

本次排查发现：

- Python Jarvis 对话端点在 Mock LLM 下可以正常返回。
- 真实 LLM 配置下失败，错误为 `401 Unauthorized`。
- 当前 `.env` 使用：`SHADOWLINK_LLM_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1`、`SHADOWLINK_LLM_MODEL=deepseek-v3.2`，API Key 被服务商拒绝。
- 之前接口把真实错误包装成泛化 `Agent response failed`，导致前端只能看到“跑不通”，无法判断是代码、网关、模型还是 Key 问题。
- 部分测试/ASGI 直连场景没有触发 FastAPI lifespan，导致 `get_llm_client()` 可能返回 `None`。

### 2.3 已完成修复

- `shadowlink-ai/app/core/dependencies.py`：为 `get_llm_client()` 增加懒加载兜底，避免 lifespan 未初始化时返回 `None`。
- `shadowlink-ai/app/api/v1/jarvis_router.py`：Agent 对话失败时返回更明确的 `502` 和 LLM 配置错误说明。
- `shadowlink-web/src/services/jarvisApi.ts`：前端 API 层读取后端 `detail`，不再吞掉真实错误。
- `shadowlink-web/src/stores/jarvisStore.ts`：聊天失败时把错误原因写入聊天记录，让用户在界面上能看到具体原因。

### 2.4 必须完成的配置修复

代码已经能诊断问题，但真实对话仍需要修正 LLM 配置：

- 确认 `SHADOWLINK_LLM_API_KEY` 是否有效。
- 确认 DashScope 是否支持当前 `SHADOWLINK_LLM_MODEL=deepseek-v3.2`。
- 如使用 DashScope 兼容模式，模型名应换成该平台真实可用的模型名。
- 如果使用 DeepSeek 官方接口，应同步切换 `SHADOWLINK_LLM_BASE_URL` 与模型名。
- 修完后访问 `/api/v1/jarvis/llm-status`，直到 `test_result.ok=true`。

### 2.5 验收标准

- `POST /api/v1/jarvis/chat` 对 `alfred` 能返回 200。
- 前端私聊任意核心 Agent 时能看到正常回复。
- LLM 配置错误时前端能显示具体原因，而不是泛化失败。
- 前端 `npm run type-check` 通过。
- 前端 `npm run build` 通过。

## 3. 第 1 层：现有 Jarvis 基座冻结与整理

### 3.1 目标

把 `docs/worklog/` 中已有成果整理成稳定基线，避免后续升级时破坏已有功能。

### 3.2 要保留的能力

- Alfred 总管家。
- Maxwell 秘书。
- Nora 营养师。
- Mira 心理师。
- Leo 生活顾问。
- Shadow 偏好学习器。
- 5 个场景圆桌。
- 私聊后台多角色协作。
- 主动关怀触发规则。
- 自动升级到圆桌。
- 专属角色工具链。
- 共享协作记忆注入。
- Java 三层网关代理。
- 前端 Jarvis 主页和设置页。

### 3.3 执行任务

- 为现有 5+1 角色建立一份“角色基线表”。
- 为每个角色记录：职责、工具、边界、主动性、是否可被其他角色询问。
- 把现有场景圆桌整理成“协作模式”的一种，而不是唯一协作方式。
- 标记当前已有 API 与前端入口，避免重复造轮子。
- 给关键路径补基础回归测试。

### 3.4 验收标准

- 有清晰的角色基线文档。
- 有现有功能清单和不能破坏的验收点。
- Agent 对话、圆桌、主动消息、设置页能跑通。

## 4. 第 2 层：角色配置与 Prompt 工程文件化

### 4.1 目标

让每个智能体不再只靠硬编码 prompt，而是有本地角色配置文件，方便用户修改性格、职责和边界。

### 4.2 执行任务

- 为 Alfred、Maxwell、Nora、Mira、Leo、Shadow 建立本地配置目录。
- 每个角色包含 `profile.yaml`、`prompt.md`、`memory/working_summary.md`、`memory/preferences.json`。
- 现有硬编码角色定义先迁移为配置文件，运行时加载。
- 保留默认配置，用户改坏时可恢复。
- 前端先做只读角色详情，再做自然语言修改角色风格。

### 4.3 验收标准

- 修改某个角色的语气配置后，下一次对话能体现变化。
- 角色配置不影响其他角色。
- 配置错误时系统能回退到默认角色配置。

## 5. 第 3 层：多智能体协作层大升级

### 5.1 目标

这是本产品的核心升级层。把现有“圆桌 + 私聊后台协作 + 共享协作记忆”升级成真正的协作智能体系统。

### 5.1.1 当前架构决策

当前确认采用混合路线：保留现有 Jarvis / ShadowLink 主体架构，不重写 Java Gateway、Python AI 服务和 React 前端；在现有 Jarvis 能力之上新增轻量 Team Collaboration Layer。

执行原则：

- 简单任务走 Direct，不启动多智能体。
- 中等任务走 Consult / Delegate，只唤醒必要角色。
- 复杂任务才走 Roundtable / Deep Team。
- token 成本、响应速度和可调试性优先于“全员参与”。
- 先把日程、记忆、工具、协作日志打通，再做 HR 招募和完整云图。

### 5.2 核心设计

新增 Team Collaboration Layer：

- Agent Registry：角色注册、能力、状态、权限、预算。
- Task Router：判断任务需要哪个角色、哪种协作模式。
- Collaboration Protocol：支持 ask、delegate、respond、escalate、recruit。
- Context Broker：只给角色传必要摘要，不传完整上下文。
- Memory Broker：统一读写全局记忆、角色记忆、任务记忆。
- Graph Recorder：记录角色协作节点、边和事件。
- Governance Guard：控制隐私、心理安全、成本和用户确认。

### 5.3 分步执行

第一步：只做 Alfred 调度 Maxwell / Mira / Nora / Leo 的结构化询问。

第二步：把当前圆桌作为 `Roundtable` 协作模式接入 Task Router。

第三步：把工具调用结果和角色回复统一记录成协作事件。

第四步：允许角色之间点对点询问，例如 Mira 询问 Maxwell 用户近期日程压力。

第五步：实现协作失败兜底，例如角色不可用、工具失败、信息不足时回到总管确认。

### 5.4 验收标准

- 用户问“帮我安排今天”，Alfred 能按协议询问 Maxwell，并按需询问 Nora / Mira / Leo。
- 用户说“我最近很累”，Mira 能获取日程压力、任务压力、健康恢复建议后再回答。
- 每次跨角色协作都有结构化日志。
- 简单任务不会全员讨论，复杂任务才进入多角色协作。

## 6. 第 4 层：长期记忆与低成本信息管理

### 6.1 目标

让每个角色有长期记忆，但不无限堆上下文；用低成本做到高职能化。

### 6.2 执行任务

- 建立全局用户档案记忆。
- 建立每个角色的专业记忆。
- 建立任务级协作记忆。
- 建立记忆写入策略：自动保存、待确认保存、不保存。
- 建立摘要压缩机制，定期去重、过期、合并。
- 角色调用前先读工作摘要，必要时再检索详细记忆。

### 6.3 验收标准

- 角色能记住用户明确允许保存的长期偏好。
- 敏感心理、健康、关系信息写入前需要确认。
- 多轮对话不会因为记忆越来越多导致 token 爆炸。
- 用户可以查看、编辑、删除记忆。

## 6.5 第 4.5 层：日程安排与日历板块落地

> 当前状态：MVP 已落地并进入可操作版本。已完成后端日程字段扩展、按日期范围查询、前端日历侧栏、日/周/月视图切换、日视图前后翻页、周/月翻页、日程详情来源/原因展示；已完成“Agent 建议日程 → 后端持久化待确认队列 → 前端待确认卡片/列表 → 用户确认或修改后确认 → 真实写入日程 API → 日历可见”的闭环；已支持用户手动新增、修改、删除、设置几点到几点、划掉/恢复完成，并向 Agent 注入本地当前时间用于制定计划。待补：路线图、待确认队列批量处理和更完整的跨端同步。

### 6.5.1 目标

把“管家团队的建议”转化为用户能真实管理的日程。日程类需求不能只停留在聊天记录中，而应写入日程安排板块，并能以日历方式查看。

### 6.5.2 执行任务

- 梳理现有 calendar 工具、日程数据结构和前端入口。
- 定义“聊天建议 → 待确认日程 → 已确认日程”的状态流。
- Alfred / Maxwell 在生成具体时间安排时，必须判断是否需要写入日程。
- 涉及新增、修改、删除日程时，默认需要用户确认。
- 日程板块增加日历视图需求：日视图、周视图、月视图、前后翻页。
- 当日视图展示一整天安排，包括空闲窗口、冲突、恢复时间和待确认项。
- 日程详情展示来源 Agent、创建原因、相关对话和可修改项。
- 路线规划先作为后续增强：第一阶段只保留地点、通勤时间、缓冲建议字段。

### 6.5.3 验收标准

- 用户说“帮我把明天下午 3 点安排健身”，确认后能在日程板块看到。
- 用户说“帮我安排今天”，Alfred / Maxwell 给出的具体时间块可以转成待确认日程。
- 用户可以打开日历查看某一天从早到晚的完整安排。
- 用户可以翻到前一天 / 后一天，后续扩展到周/月。
- 由 Agent 创建的日程能显示来源和原因。
- 未确认的建议不会直接写入正式日程。

## 7. 第 5 层：协作云图与可解释性

### 7.1 目标

让用户直观看到管家团队如何协作，而不是只看到最终答案。

### 7.2 执行任务

- 先保存每次任务的 `team_graph.json`。
- 前端先展示文字版协作链路。
- 再升级为节点关系图。
- 节点包括用户、总管、角色、工具、记忆、决策、输出。
- 点击节点可以看输入摘要、输出结论、工具结果、成本估算。

### 7.3 验收标准

- 每个复杂任务都有可查看协作链路。
- 用户能看见“谁问了谁，为什么问，回答了什么”。
- 协作云图不泄露被隐藏或未授权的敏感记忆。

## 8. 第 6 层：主动关心升级

### 8.1 目标

让系统能高准确度、低打扰地关心用户。

### 8.2 执行任务

- 继承已有 stress_spike、schedule_overload、sleep_poor、free_window_detected、mood_declining。
- 引入用户主动关心偏好。
- 引入安静时段。
- 引入提醒频率控制。
- 引入“先询问是否需要帮助”的温和模式。
- 让心理相关提醒遵守安全边界。

### 8.3 验收标准

- 主动关心可以关闭。
- 主动关心不会频繁打扰。
- 主动关心内容具体、温和、可执行。
- 心理高风险表达会建议现实专业支持，而不是仅给 AI 建议。

## 9. 第 7 层：HR 管家与团队自扩展

### 9.1 目标

让团队能发现能力缺口，并在用户同意后新增角色。

### 9.2 执行任务

- Shadow 或 HR 管家分析重复需求。
- 统计哪些任务经常无法被现有角色很好处理。
- 生成新角色草案，包括职责、性格、工具、记忆策略、预算。
- 用户确认后创建角色配置目录。
- 新角色加入 Agent Registry。
- 支持停用、合并、删除角色。

### 9.3 验收标准

- HR 不会未经用户同意新增角色。
- 新角色能被 Alfred 调用。
- 角色数量过多时系统建议合并或休眠低频角色。

## 10. 推荐执行顺序

### 第一周：先修通与冻结基线

- 修通 Agent 对话。
- 修 LLM 配置诊断。
- 整理现有角色基线。
- 固定最小回归测试。

### 第二周：角色配置文件化

- 迁移 5+1 角色配置。
- 做角色配置加载器。
- 支持用户改角色语气。

### 第三周：日程写入与日历板块 MVP

- 打通聊天建议到待确认日程。
- 日程板块支持日视图和前后翻页。
- 日程详情显示来源 Agent 和创建原因。
- Maxwell 作为日程安排主负责人接入。

### 第四至五周：协作层 MVP

- 做 Agent Registry。
- 做 Task Router。
- 做 ask/delegate/respond 协议。
- 做结构化协作日志。
- 简单任务保持 Direct，复杂任务才升级。

### 第六周：协作可视化第一版

- 文字版协作链路。
- JSON team graph。
- 前端节点详情面板。

### 第七周：记忆与主动关心升级

- Memory Broker。
- 记忆确认。
- 主动关心偏好。
- 提醒频率控制。

### 第八周以后：HR 管家与团队扩展

- 能力缺口分析。
- 新角色草案生成。
- 用户确认后招募。

## 11. 当前马上要做的事

1. 确认 Python AI、Java Gateway、Web 三端稳定运行，Agent 私聊和头脑风暴可用。
2. 冻结现有 5+1 角色基线，保留当前有效架构。
3. 梳理现有 calendar 工具、日程数据结构和前端日程入口。
4. 设计“聊天建议 → 待确认日程 → 已确认日程”的最小闭环。
5. 开始第 2 层角色配置文件化，同时为 Maxwell 补齐日程写入职责说明。
