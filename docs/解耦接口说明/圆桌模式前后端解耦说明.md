# 圆桌模式前后端解耦说明

> 本文是给后续圆桌前端、后端、主界面维护者协作用的解耦说明。
>
> 重要：`roundtable-api-contract.md` 和 `roundtable-frontend-design-guide.md` 是硬契约文件，不在本文中修改。本文只解释如何按现有契约拆边界、协作和落地。

## 1. 解耦目标

圆桌模式要从主 Jarvis 私聊里拆成相对独立的业务包：

```text
主 Jarvis 私聊 / Home
  -> 只负责进入圆桌、关闭圆桌、接收返回结果
圆桌前端包
  -> 负责圆桌 UI、SSE 展示、结果卡、场景交互
圆桌后端包
  -> 负责多智能体回合、场景协议、工具调用、结果生成
其它业务模块
  -> 只通过 pending action / confirmed write 被圆桌间接影响
```

解耦后，主私聊不需要知道圆桌内部状态机；圆桌也不能直接改日历、计划、关怀等模块的数据。

## 2. 当前边界结论

### 2.1 已经比较清晰的边界

- 后端入口集中在 `shadowlink-ai/app/api/v1/jarvis/roundtable/router.py`。
- 请求模型集中在 `shadowlink-ai/app/api/v1/jarvis/roundtable/schemas.py`。
- 圆桌后端核心逻辑集中在 `shadowlink-ai/app/jarvis/shadow_roundtable.py`、`roundtable_graph.py`、`roundtable_protocols.py` 等文件。
- 前端圆桌 UI 主要在 `shadowlink-web/src/components/jarvis/RoundtableStage.tsx`。
- 前端圆桌纯逻辑 helper 在 `shadowlink-web/src/components/jarvis/roundtableStageLogic.ts`。
- 前端 API 门面目前仍在 `shadowlink-web/src/services/jarvisApi.ts`。

### 2.2 仍然耦合的地方

- `JarvisHome.tsx` 仍负责私聊、圆桌、日历等入口切换。
- `jarvisStore.ts` 仍承载跨模块状态和入口跳转。
- `jarvisApi.ts` 仍是较大的兼容门面，圆桌 API 还没有完全拆成独立 service。
- 圆桌返回私聊仍依赖主私聊 session / agent 语义。

这些地方暂时不要大改，除非明确要做入口层重构。

## 3. 文件职责划分

### 3.1 圆桌前端开发可改

```text
shadowlink-web/src/components/jarvis/RoundtableStage.tsx
shadowlink-web/src/components/jarvis/roundtableStageLogic.ts
shadowlink-web/src/components/jarvis/roundtableStageLogic.test.ts
```

可以做：
- 圆桌 UI 排版。
- 直播会议发言展示。
- SSE payload 的纯解析。
- 结果卡展示。
- 场景专属 UI。
- pending confirmation 数量展示。
- next_routes 展示。

不要做：
- 改主私聊状态机。
- 改入口跳转规则。
- 改圆桌 URL。
- 改后端字段名。
- 在组件里新增直接 `fetch`。

### 3.2 圆桌后端开发可改

```text
shadowlink-ai/app/api/v1/jarvis/roundtable/router.py
shadowlink-ai/app/api/v1/jarvis/roundtable/schemas.py
shadowlink-ai/app/jarvis/shadow_roundtable.py
shadowlink-ai/app/jarvis/roundtable_graph.py
shadowlink-ai/app/jarvis/roundtable_protocols.py
shadowlink-ai/app/jarvis/roundtable_sessions.py
```

可以做：
- 增加内部场景协议。
- 调整 agent turn 组织方式。
- 增加内部工具上下文。
- 增强 SSE 事件 payload，但要保持向后兼容。
- 增加 result metadata。

不要做：
- 改已有接口路径。
- 让圆桌直接写日历/计划/关怀数据库。
- 绕过 pending action 确认链路。
- 让 `jarvis_task_plan_decompose` 在圆桌里直接落计划或日历。

### 3.3 主界面维护者负责

```text
shadowlink-web/src/components/jarvis/JarvisHome.tsx
shadowlink-web/src/stores/jarvisStore.ts
shadowlink-web/src/services/jarvisApi.ts
shadowlink-web/src/services/sse.ts
```

负责：
- 从私聊进入圆桌。
- 从圆桌返回私聊。
- 保持 source_session_id / source_agent_id。
- 跨模块状态切换。
- SSE 基础设施。

圆桌业务开发不应随意修改这些文件。

## 4. 前后端通信边界

圆桌前端只通过 service 调用后端，不直接拼 URL。

当前稳定接口包括：

```text
POST /api/v1/jarvis/roundtable/start
POST /api/v1/jarvis/roundtable/continue
GET  /api/v1/jarvis/roundtable/{session_id}/decision-result
POST /api/v1/jarvis/roundtable/{session_id}/accept
GET  /api/v1/jarvis/roundtable/{session_id}/brainstorm-result
POST /api/v1/jarvis/roundtable/{session_id}/save
POST /api/v1/jarvis/roundtable/{session_id}/plan
POST /api/v1/jarvis/roundtable/{session_id}/return
```

前端必须允许 SSE payload 增加字段。

后端必须保证：
- 旧字段不随意删除。
- 事件类型不随意改名。
- 新场景不要求前端新增必填请求字段。
- 写操作不绕过确认链路。

## 5. SSE 解耦规则

### 5.1 前端处理方式

前端只依赖通用事件语义：
- 发言开始。
- 发言片段。
- 角色完成。
- 圆桌完成。
- 决策结果。
- 脑暴结果。
- 场景阶段。
- 降级 / 错误。

前端遇到未知 event：
- 不白屏。
- 可以记录到完整记录。
- 不影响后续事件展示。

### 5.2 后端新增事件方式

后端可以新增增强事件，但必须满足：
- 不替换已有通用事件。
- 不要求前端必须处理才能继续流程。
- 关键业务结果仍通过 result endpoint 或最终 result event 获取。

## 6. 结果卡解耦规则

圆桌结果分两类：

```text
decision result
brainstorm result
```

前端结果卡只负责展示和触发用户动作。

后端负责生成结构化结果。

用户点击后的写入边界：
- 接受决策：生成 pending action，不直接改日历。
- 保存脑暴：用户点击后写记忆。
- 脑暴转计划：生成待确认计划，不直接静默落库。
- 返回私聊：只把圆桌摘要写回来源私聊 session。

## 7. 工具调用和写操作边界

圆桌可以智能调用工具，但必须分层：

```text
只读工具
  -> 可以直接作为上下文注入圆桌
写操作工具
  -> 只能生成 pending_confirmation
用户确认
  -> 业务模块正式写入
```

特别规则：
- `jarvis_task_plan_decompose` 在私聊里可能直接生成计划相关结构。
- 但在圆桌里必须按 deferred confirmation tool 处理。
- 圆桌不能直接写 plan / plan_day / calendar projection。

## 8. 圆桌和私聊的入口边界

主私聊只关心两个动作：

```text
进入圆桌
返回私聊
```

进入圆桌时需要带：
- `source_session_id`
- `source_agent_id`
- `scenario_id`
- `mode_id`
- 用户输入

返回私聊时后端负责把总结写回来源 session，前端只负责切回对应私聊视图。

不要让圆桌前端自己拼私聊消息，也不要让主私聊解析圆桌内部回合。

## 9. 建议的下一步解耦动作

### 9.1 前端 service 轻拆

新增：

```text
shadowlink-web/src/services/roundtableApi.ts
```

职责：
- 只放圆桌 API 类型和调用方法。
- `jarvisApi.ts` 保留兼容 re-export 或 thin wrapper。
- `RoundtableStage.tsx` 后续从 `roundtableApi.ts` 引入类型和方法。

好处：
- 圆桌前端不再依赖大门面。
- 改圆桌 service 不影响其它 Jarvis API。

注意：
- 不改接口路径。
- 不改请求/响应字段。

### 9.2 前端状态轻拆

如果后续继续做，可以新增：

```text
shadowlink-web/src/components/jarvis/roundtableStore.ts
```

只管理：
- 当前 roundtable session。
- SSE 事件列表。
- 当前阶段。
- result card state。

但第一版不强制做，避免动 `jarvisStore.ts` 太多。

### 9.3 后端内部服务轻拆

如果 `shadow_roundtable.py` 继续变大，可以拆：

```text
shadowlink-ai/app/jarvis/roundtable_service.py
shadowlink-ai/app/jarvis/roundtable_result_service.py
shadowlink-ai/app/jarvis/roundtable_tool_policy.py
```

分别负责：
- 会话流程编排。
- result 生成和读取。
- 圆桌工具权限 / pending confirmation 策略。

第一版不建议大拆，只在新增功能时顺手抽边界。

## 10. 验收清单

每次改圆桌时检查：

- 没有修改 `roundtable-api-contract.md` 和 `roundtable-frontend-design-guide.md`，除非明确要求。
- 没有改已有圆桌 URL。
- 没有新增组件内直接 `fetch`。
- `RoundtableStage.tsx` 能忽略未知 SSE 字段。
- 写操作仍走 pending action 或用户确认。
- 返回私聊仍通过 `/return` 接口。
- 主私聊不解析圆桌内部状态机。
- `JarvisHome.tsx` 和 `jarvisStore.ts` 没有被圆桌业务随意改动。
- 如果确实改了接口，必须同步写接口说明文件并明确告知。

## 11. 防乱码要求

圆桌相关文档和前端中文文案必须使用 UTF-8。

不要用 PowerShell 默认 `Set-Content` 写中文文件，除非显式指定 `-Encoding utf8`。

推荐：

```powershell
Set-Content -Encoding utf8 path content
```

或用编辑器保存为 UTF-8。
