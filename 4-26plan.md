# 2026-04-25 产品与架构执行计划

> 创建时间：2026-04-25（Asia/Shanghai）  
> 文档目的：在已完成的第 4.5 层日程模块基础上，重新明确后续管家团队、日程委托、单 Agent 基础能力、外部 skill / API / 文件与网页能力的产品与架构方向。  
> 当前原则：不推翻现有有效架构；保留单 Agent 能力、工具能力、RAG、MCP、文件处理、网页能力和未来外部 API 接入能力；多智能体团队能力采用渐进式升级。

## 1. 总体结论

当前产品不应该走“所有 Agent 都直接写日程、直接改数据”的方向。更合理的方向是：

- 每个 Agent 都可以识别用户是否存在日程化、任务化、文件处理、网页处理、代码处理、外部系统处理等需求。
- 但具体执行要由对应专职角色或专属 skill 管理。
- 日程写入由秘书 Maxwell 统一管理。
- 文件、网页、代码、外部 API 等能力作为可接入 skill / tool 系统，不应被削弱。
- 单 Agent 基础能力必须保留，包括 ReAct、Plan-and-Execute、工具调用、外部 skill 接入、文件操作、网页操作、简单代码实现等。
- 多智能体团队架构要逐步做，但不需要先完成完整团队架构再修日程。

后续主路线：

```text
保留现有单 Agent + 工具基础能力
  ↓
先做轻量日程委托协议：任意 Agent → Maxwell
  ↓
再升级 Team Collaboration Layer
  ↓
再做完整管家团队协作、协作云图、HR 招募、外部 skill 市场
```

## 2. 当前日程写入逻辑的问题

当前已实现的日程链路是：

```text
用户 → 当前对话 Agent → Agent 判断是否调用 calendar 工具
  → 后端生成 pending_confirmation
  → 前端展示待确认卡
  → 用户确认
  → 写入日程
```

这个链路已经比“直接写入”安全，但仍有几个产品和架构问题：

1. **职责不够清晰**：任何拥有日程工具的 Agent 都可能提出写日程。
2. **秘书 Maxwell 的中心角色不突出**：日程本应由秘书统一安排。
3. **冲突检查不足**：Agent 可能只按用户说法创建日程，没有充分检查已有安排。
4. **日程格式过于依赖模型输出**：如果模型没有给出严格 `start/end`，容易失败。
5. **其它 Agent 不应被要求自己规划精确时间**：例如心理 Agent、营养 Agent 可以提出意图，但不应该承担最终排期。

## 3. 新日程委托原则

### 3.1 日程不是所有 Agent 直接写

日程应作为 Maxwell 的核心 skill，由秘书管理。

其它 Agent 可以识别“这可能需要加入日程”，但不直接写入正式日程，也不直接决定最终时间。

推荐职责：

| 角色 | 可以做什么 | 不应该做什么 |
| --- | --- | --- |
| Alfred | 判断是否需要交给 Maxwell；协调跨领域安排 | 直接跳过秘书写日程 |
| Maxwell | 读取日程、检查冲突、规划时间、生成待确认日程 | 未经用户确认直接写入 |
| Mira | 识别情绪/恢复相关安排需求 | 自己决定最终时间并写日程 |
| Nora | 识别饮食、睡眠、运动、恢复安排需求 | 自己直接写正式日程 |
| Leo | 识别生活活动、出行、本地生活安排需求 | 自己直接写正式日程 |
| Shadow | 学习用户日程偏好 | 直接创建或修改日程 |

### 3.2 日程意图格式可以宽松

用户或其它 Agent 不必提供严格时间。

允许输入：

- “明天提醒我健身。”
- “这周找个时间和朋友吃饭。”
- “今天下午安排一段休息。”
- “晚上记得让我复盘。”
- “帮我把这个任务排进日程。”

这些都可以形成日程意图，但不要求一开始就有 `start/end`。

### 3.3 最终时间由 Maxwell 规划

Maxwell 接收日程意图后，应负责：

1. 读取当前时间。
2. 读取已有日程。
3. 判断是否已有冲突。
4. 判断用户给的时间是否已经过去。
5. 如果时间不明确，根据上下文和偏好规划候选时间。
6. 如果有多个可能时间，给用户 1-3 个选项。
7. 生成待确认日程。
8. 用户确认后才写入正式日程。

## 4. 轻量日程委托协议

在完整团队架构完成前，先做一个轻量协议。

### 4.1 协议流

```text
用户和任意 Agent 对话
  ↓
当前 Agent 识别到日程意图
  ↓
当前 Agent 生成 schedule_intent
  ↓
后端/协作层转交 Maxwell
  ↓
Maxwell 读取已有日程、当前时间、用户偏好
  ↓
Maxwell 规划时间并检查冲突
  ↓
前端弹出 Maxwell 待确认卡 / 对话框
  ↓
用户确认、修改或取消
  ↓
确认后写入正式日程
```

### 4.2 schedule_intent 建议格式

这个格式不要求用户自己填完整时间，重点表达“意图”。

```json
{
  "type": "schedule_intent",
  "from_agent": "mira",
  "to_agent": "maxwell",
  "source_session_id": "session_xxx",
  "user_request": "我最近很累，明天提醒我散步恢复一下",
  "suggested_title": "恢复散步",
  "time_preference": "明天，具体时间由 Maxwell 安排",
  "duration_minutes": 30,
  "deadline": null,
  "location": null,
  "notes": "Mira 建议用于低压力恢复",
  "constraints": [
    "避免和已有日程冲突",
    "不要安排得太晚",
    "优先选择低压力时间段"
  ],
  "needs_route_planning": false,
  "needs_user_confirmation": true
}
```

### 4.3 Maxwell 输出待确认格式

Maxwell 处理后，再形成更严格的待确认日程：

```json
{
  "type": "calendar.add",
  "status": "pending_confirmation",
  "owner_agent": "maxwell",
  "title": "恢复散步",
  "start": "2026-04-26T16:30:00+08:00",
  "end": "2026-04-26T17:00:00+08:00",
  "reason": "明天下午 16:30 后没有高强度安排，适合低压力恢复",
  "conflict_check": {
    "has_conflict": false,
    "checked_events": 3
  },
  "alternatives": [
    {
      "start": "2026-04-26T19:30:00+08:00",
      "end": "2026-04-26T20:00:00+08:00",
      "reason": "晚饭后轻量活动"
    }
  ],
  "needs_user_confirmation": true
}
```

## 5. 前端交互目标

### 5.1 不论用户和谁对话，都由 Maxwell 出面确认日程

用户和 Mira 对话时，如果 Mira 识别到日程意图，界面应出现 Maxwell 的确认框：

> Maxwell：我建议把「恢复散步」安排在明天 16:30-17:00。这个时间没有和已有安排冲突，也不会太晚。要写入日程吗？

用户可以：

- 确认写入。
- 修改时间后确认。
- 查看替代时间。
- 取消。

### 5.2 待确认日程进入日历侧栏

所有待确认日程都进入“待确认安排”列表。

刷新页面后仍然存在。

用户可以在日历侧栏中修改后确认。

## 6. 当前 bug 的分析方向

用户现在让秘书安排日程时仍出现：

> 对话暂时没有跑通：Internal server error

这说明当前后端私聊链路仍有未捕获异常。可能原因包括：

1. Python AI 服务没有重启，仍在跑旧代码。
2. Windows 时区数据库缺失导致当前时间注入失败。
3. 模型输出的工具 JSON 不完整或不合法。
4. 日程 pending action 保存时字段缺失。
5. pending action 表未创建或旧进程未加载新 schema。
6. Provider / LLM 调用本身失败，但错误被统一包装成 Internal server error。

修复原则：

- 先看后端日志拿到真实异常。
- 所有日程工具调用失败都应返回可读错误，不应 500。
- 如果时间缺失，不应报错，应转成 `schedule_intent` 交给 Maxwell 补齐。
- 如果 JSON 不完整，应让 Maxwell 追问用户或自动规划候选时间。

## 8. 单 Agent 基础能力必须保留

管家团队不是要替代单 Agent 能力。每个 Agent 仍必须具备基础执行能力。

必须保留和强化：

- ReAct：边思考边调用工具。
- Plan-and-Execute：复杂任务拆步骤执行。
- Direct Chat：简单问题快速回答。
- Tool Calling：调用日程、文件、网页、搜索、知识库、外部 API。
- Memory：读写长期偏好和任务上下文。

## 9. 外部 skill 和电脑操作能力必须保留

未来产品不能只做生活管家聊天，还要保留“个人专业管家团队”的实际执行能力。

需要支持未来接入：

这些能力应以 skill / tool / MCP server 的方式接入，不应写死在 Agent prompt 里。

## 10. Skill 架构原则

### 10.1 Skill 是可注册能力

每个 skill 应有：

- 名称。
- 描述。
- 输入 schema。
- 输出 schema。
- 权限级别。
- 是否需要用户确认。
- 可调用角色。
- 成本预算。
- 风险等级。

### 10.2 Skill 不等于 Agent

Agent 是角色，skill 是能力。

例如：

- Maxwell 是秘书 Agent。
- `schedule_manager` 是日程 skill。
- `calendar.add` 是具体工具。
- `feishu.create_task` 是外部 API 工具。

### 10.3 高风险 skill 必须确认

必须确认的操作：

- 写入日程。
- 删除日程。
- 修改文件。
- 移动文件。
- 调用外部 API 发送消息。
- 创建外部任务。
- 任何不可逆或影响外部世界的操作。

#### 3.5.1 为什么需要这一步

普通日程只解决“某天某时做什么”。但真正的私人专业管家团队需要处理更复杂的任务，例如：

- “我要考雅思。”
- “暑假后想去日本玩。”
- “一个月后开始准备搬家。”
- “之后有空帮我慢慢整理作品集。”
- “今年想把身体状态调整好。”

这些不是单条日程，而是一个长期目标、项目、计划或后台任务。秘书不能只创建一个日历事件，而要判断：

1. 这是短期任务、长期项目、等待型任务，还是周期型任务。
2. 是否需要拆解为多个阶段。
3. 哪些步骤需要变成日程。
4. 哪些步骤只是后台提醒、待办或里程碑。
5. 哪些时间点必须由用户确认。
6. 是否需要其它 Agent 提供专业建议，例如 Mira/Nora/Leo。

#### 3.5.2 新的任务入口原则

用户直接跟 Maxwell 说，是最优路径：

```text
用户 → Maxwell → 任务拆解 skill → 后台任务清单 / 待确认日程卡 / 长期计划
```

用户跟其它 Agent 说，也不能丢失：

```text
用户 → Mira/Nora/Leo/Alfred
  ↓
当前 Agent 识别到长期或短期安排意图
  ↓
生成 task_intent / schedule_intent
  ↓
路由给 Maxwell
  ↓
Maxwell 判断任务类型、拆解、安排、追问或生成确认卡
```

其它 Agent 不直接创建长期任务计划，只负责识别意图、补充专业信息、转交 Maxwell。

#### 3.5.4 Maxwell 专属 Task Planning Skill

#### 3.5.5 雅思示例

用户说：“我要考雅思。”

Maxwell 不应该只回复一句建议，而应该：

1. 识别为 `long_project`。
2. 询问关键缺失信息：目标分数、考试日期、每天可投入时间。
3. 如果用户不知道日期，则建议先创建“雅思备考项目”。
4. 给出可编辑计划：
   - 每周一/三/五 19:30-20:30 听力/阅读。
   - 每周六 10:00-11:30 写作训练。
   - 每两周一次模拟测试。
   - 到某日期完成第一阶段。
5. 弹出“长期任务计划确认卡”，用户可以改开始日期、结束日期、频率、单次时长。
6. 用户确认后：
   - 写入后台任务清单。
   - 只把近期 1-2 周日程写入日历，避免一次性塞满日历。
   - 后续由 Maxwell 滚动安排。

#### 3.5.6 旅行示例

用户说：“放暑假后想去某地玩。”

Maxwell 应该：

1. 识别为 `future_project`。
2. 登记为后台旅行计划。
3. 询问目的地、预算、出行人数、偏好。
4. 给出任务拆解：
   - 几号开始查机票。
   - 几号前定酒店。
   - 几号整理证件/行李。
   - 出发前几天确认天气和路线。
5. 只把关键节点作为待确认日程，不要把所有任务直接写死。
6. 必要时后续让 Leo 负责本地活动建议，让 Maxwell 负责时间和预订节点。

#### 3.5.7 与 Skill Registry 的关系

这一步确实依赖 Skill Registry 的基础完善，因为长期任务拆解不是纯 prompt 能稳定完成的。至少需要：

- skill schema：定义输入输出格式。
- confirmation policy：哪些任务需要用户确认。
- persistence：后台任务清单要长期保存。
- permissions：哪些 Agent 可以创建 task_intent，谁能写入 task list。
- tool whitelist：Maxwell 能调用 task planner，其他 Agent 只能提交 intent。
- cost control：长期任务拆解不应每次都全量调用高成本模型。

因此 Step 3.5 的执行方式应是：

```text
先做最小 Task Planning Skill schema
  ↓
再做后台任务清单 persistence
  ↓
再做 Maxwell 调用 task planner
  ↓
再做 task_intent → Maxwell
  ↓
最后接入更完整 Skill Registry 权限、成本、外部 API 能力
```

也就是说，Step 3.5 是 Skill Registry 强化前的“关键业务 skill 切片”，而 Step 4 是把这个模式泛化成完整 skill 系统。

### 【部分实现：已落地 Demo Run、Demo Trace、Demo Memory、测试画像重置，以及主私聊 LLM JSON + 规则兜底记忆提取/保存/召回/删除闭环；还差混合存储和敏感记忆趋势治理】Step 3.6：完善记忆提取层与长期记忆治理

目标：在现有 Shadow 偏好学习器、协作记忆、聊天历史和长期/语义/情节记忆模块基础上，补齐一个轻量但可靠的 Memory Extraction Layer。它不追求一次性做成复杂记忆系统，而是先解决“什么值得记、记到哪里、怎么召回、什么时候忘”的核心问题，让私人管家团队逐步形成稳定、低 token、可维护的长期理解能力。

#### 3.6.1 当前问题

项目已有记忆相关能力，但还不够成熟：

- `PreferenceLearner` 能每隔若干轮提取偏好，但偏重用户偏好，且持久化和主 prompt 注入还不完整。
- `collaboration_memories` 能保存用户约束、工具动作和协作总结，但主要是规则型保存，不是统一的记忆准入层。
- `LongTermMemory`、`SemanticMemory`、`EpisodicMemory` 已存在，但与 Jarvis 私聊主流程连接不够紧。
- 当前缺少统一的记忆准入、分类、去重、重要性评分、召回和遗忘机制。

#### 3.6.2 记忆准入原则

每轮对话结束后，不应把所有文本都长期保存，而是先经过准入判断：

1. **事实性信息优先长期保存**
   - 例如会议时间、约饭时间、旅行时间、考试日期、重要联系人、长期目标。
   - 这类信息如果涉及未来行动，应同步交给 Maxwell 或后台任务/日程系统管理。

2. **高结构化 Entity & Relation 必须提取**
   - 例如“小王是我的同事”“小李是我的好朋友”“技术问题我通常找小王”。
   - 应提取成实体和关系，而不是只保存原文。
   - 示例结构：`用户 --同事--> 小王`、`用户 --好友--> 小李`、`用户 --技术求助对象--> 小王`。

3. **个性化信息需要稳定积累**
   - 包括用户画像、作息、习惯、偏好、厌恶、常用编号、沟通风格、健康/心理边界、生活节律。
   - 只有重复出现、明确表达或高置信度推断的信息才升级为长期偏好。

4. **表面话语背后的意图要谨慎记录**
   - 例如用户频繁说“我最近不想见人”，可能反映社交能量下降。
   - 这类不应直接写成绝对事实，而应保存为低置信度洞察，并允许后续反思修正。

5. **临时推理和低价值闲聊默认不保存**
   - 玩笑、随口吐槽、一次性情绪、临时推理过程、模型中间分析，不进入长期记忆。
   - 如果短期内对当前任务有用，可只保留在短期上下文或当前会话摘要中。

#### 3.6.3 记忆分类与分流存储

不要把所有记忆都塞进同一种数据库，按数据结构分流：

- **关系型 DB / SQLite**
  - 保存确定事实、日程、后台任务、用户约束、待确认记忆、重要性分数、来源、时间戳、置信度。
  - 适合查询“某件事是否存在、何时发生、是否确认”。

- **图结构 / Semantic Graph**
  - 保存 Entity & Relation，例如人际关系、组织关系、职责关系、偏好关联。
  - MVP 阶段可以先用现有 `SemanticMemory` 文件图或 SQLite 边表，不必立刻引入独立图数据库。

- **向量索引 / RAG Memory**
  - 保存长文本摘要、重要对话片段、项目背景、文档型知识。
  - 用于语义召回，但不作为唯一事实来源。

- **用户画像 Profile**
  - 保存高置信、稳定、低频变化的个人信息和偏好。
  - 进入 Agent prompt 时必须高度精简，避免 token 膨胀。

#### 3.6.4 记忆提取流水线

建议新增统一流程，不要散落在各处重复判断：

```text
对话/工具/日程/后台任务事件
  ↓
Memory Candidate Extractor：提取候选记忆
  ↓
Admission Gate：判断是否值得保存
  ↓
Classifier：事实 / 关系 / 偏好 / 约束 / 洞察 / 项目背景
  ↓
Normalizer：结构化、去重、合并同义项
  ↓
Storage Router：写入 SQLite / Semantic Graph / Vector Memory / Profile
  ↓
Retrieval Builder：按 Agent、任务和当前意图精简召回
```

其中 `Admission Gate` 必须输出原因，避免黑箱记忆：

```json
{
  "should_save": true,
  "memory_type": "relation",
  "confidence": 0.86,
  "importance": 0.72,
  "reason": "用户明确说明小王是同事，并且技术问题会找他"
}
```

#### 3.6.5 心理与生活节律适配

由于产品定位是“个人私人专业管家团队”，记忆层不能只服务问答和偏好，还必须为后续心理分析师、生活节律管理员、秘书 Maxwell 提供可分析的数据底座。这里要注意：系统不是医疗诊断工具，但可以持续记录生活节律、情绪波动和异常信号，帮助 Agent 更准确、更温和地关心用户，并在必要时建议用户寻求现实支持。

需要为心理与节律增加专门的结构化记忆类型：

- **Mood Signal / 情绪信号**
  - 记录用户自述的压力、焦虑、疲惫、低落、兴奋、烦躁、孤独、社交能量等。
  - 必须带时间戳、来源原文、置信度、强度区间、是否用户明确表达。
  - 不把单次情绪直接固化为人格标签。

- **Rhythm Signal / 生活节律信号**
  - 记录睡眠、起床、饮食、运动、学习/工作时段、社交频率、休息不足、熬夜等。
  - 重点保存“趋势”和“偏离常态”，而不是无限保存每一句原话。

- **Stressors & Support / 压力源与支持系统**
  - 记录引发波动的压力源，例如考试、工作、人际、身体不适、长期任务堆积。
  - 记录支持资源，例如朋友、家人、同事、常用放松方式、有效安抚方式。

- **Care Preference / 关心方式偏好**
  - 记录用户希望被怎样关心：直接提醒、温柔询问、少打扰、晚上不要打扰、先给方案还是先共情。
  - 这类信息对“自动高准确度关心用户”非常关键。

- **Risk Marker / 风险标记**
  - 对极端低落、自伤暗示、长期失眠、明显功能受损等内容做高重要性标记。
  - 风险标记不应被普通遗忘衰减快速抹掉，但也不能被随意注入所有 Agent prompt；只给具备权限的心理/安全相关 Agent 使用。

建议新增一个轻量 `wellbeing_memory` 或在 `extracted_memories` 中增加 wellbeing 分类字段：

```json
{
  "memory_type": "mood_signal",
  "source": "chat",
  "timestamp": "2026-04-26T22:30:00+08:00",
  "signal": "anxiety",
  "intensity": 0.7,
  "confidence": 0.82,
  "trigger": "考试准备压力",
  "duration_hint": "最近几天",
  "user_words": "我最近一想到雅思就很焦虑",
  "care_suggestion": "先共情，再给小步行动建议",
  "privacy_level": "sensitive"
}
```

心理与节律记忆的召回策略要更谨慎：

- 只向相关 Agent 精简召回，例如心理管理员、秘书、生活节律管理员。
- 普通 Agent 默认只拿到必要摘要，不拿敏感原文。
- 关心用户时优先使用趋势和偏好，不要频繁引用敏感原话造成压力。
- 如果检测到异常波动，应先温和询问和确认，不要直接下判断。
- 用户必须能查看、修改、删除这类敏感记忆。

心理与节律分析的核心目标：

```text
连续信号记录
  ↓
趋势与异常检测
  ↓
结合日程/后台任务/睡眠/社交数据
  ↓
生成低打扰、高准确度的关心建议
  ↓
必要时提醒用户寻求现实支持
```

#### 3.6.6 动态维护机制

长期使用时，记忆必须能更新、降权和遗忘：

- **重要性加权**：越影响生活安排、心理边界、长期目标、人际关系、健康安全，权重越高。
- **置信度机制**：用户明确说出的事实高于模型推断；一次性推断低置信，多次出现后再升级。
- **反思总结**：定期把一段时间内的记忆压缩成深度洞察，例如“用户近期晚上社交意愿下降”。
- **遗忘衰减**：低重要性、长期未使用、无后续验证的信息逐步降权，不再默认注入 prompt。
- **分级存储**：热记忆进入 prompt，温记忆按需召回，冷记忆只在搜索或用户追问时查。
- **冲突处理**：新记忆与旧记忆冲突时不要直接覆盖，应标记冲突并在必要时询问用户。

#### 3.6.7 成本与 token 控制

- 每轮对话只做轻量规则判断，只有命中高价值候选时才调用 LLM 提取。
- 对每个 Agent 注入的记忆不超过少量高相关条目。
- 长文本只保存摘要和引用，不把完整聊天历史反复塞进 prompt。
- 记忆提取结果必须结构化，便于后续压缩、去重和召回。

#### 3.6.8 测试与路演画像模式

为了长期开发和路演稳定性，需要增加一个“测试画像重置/种子模式”。平时测试会产生大量无效用户画像、偏好、日程、情绪信号和协作记忆，如果不清理，会导致演示时 Agent 行为不一致、记忆来源混乱、无法追溯每一步 demo 为什么触发。因此需要一个受保护的一键功能：

```text
一键清空测试记忆
  ↓
选择/导入路演画像种子
  ↓
从今天开始重新记录所有 demo 事件
  ↓
每一步路演脚本都能追溯到输入、提取记忆、生成动作和确认结果
```

这个能力不只是“删除数据”，而是面向路演和回归测试的可控环境管理：

- **一键清空用户画像与测试记忆**
  - 清空 Shadow 学到的 preferences。
  - 清空或归档测试产生的 `agent_chat_turns`、`collaboration_memories`、`extracted_memories`、`wellbeing_memory`、测试日程、测试后台任务。
  - 对真实用户数据必须二次确认，默认只支持 `demo/test mode`，避免误删真实长期记忆。

- **路演画像种子**
  - 支持导入一个鲜明、稳定、可复现的人物画像，例如：备考雅思、轻微焦虑、作息容易后移、喜欢温柔但直接的提醒、近期计划暑假后旅行。
  - 画像种子应写入用户 profile、初始偏好、初始生活节律、初始后台任务或示例约束。
  - 种子文件应保存在 `worklog-lsq/test` 或专门 `demo_profiles` 目录，便于和路演脚本对应。

- **从今天开始的一致性测试窗口**
  - 重置后记录 `demo_start_at`。
  - 所有新产生的记忆、日程、后台任务、协作事件都带 `demo_run_id` 或 `trace_id`。
  - 前端可以显示“当前处于 Demo 模式 / 从某时刻开始记录”。

- **每一步 demo 可追溯**
  - 路演脚本中的每个 Demo Step 都分配 `demo_step_id`。
  - 后端记录该步骤的用户输入、Agent、工具调用、记忆提取结果、确认动作、写入结果。
  - 方便路演前复盘：为什么这一步出现这个卡片、写入了什么记忆、触发了哪个 skill。

- **高一致性输出**
  - Demo 模式下尽量固定关键上下文：用户画像、所在地、当前日期策略、初始日程和后台任务。
  - LLM 输出仍可自然变化，但关键业务事件必须稳定：该出日程卡就出日程卡，该出长期任务卡就出长期任务卡，该路由 Maxwell 就路由 Maxwell。

MVP 可先做成设置页里的危险操作区：

```text
[清空测试画像与记忆]
[加载路演画像：雅思备考 + 生活节律 + 暑假旅行]
[开始新的 Demo Run]
[导出本次 Demo Trace]
```

#### 3.6.9 MVP 落地顺序

1. 建立 `memory_candidates` / `extracted_memories` 的 SQLite 表，记录来源、类型、置信度、重要性、状态。
2. 把现有 `PreferenceLearner` 升级为统一 Memory Extractor，支持事实、关系、偏好、约束、洞察五类输出。
3. 先用规则 + 小模型/低成本模型做准入；高价值或复杂关系再调用主 LLM。
4. 把明确的 Entity & Relation 写入 `SemanticMemory` 或 SQLite relation 表。
5. 把稳定偏好写入用户画像，但必须保留来源和更新时间。
6. 在私聊 prompt 中只注入与当前 Agent 和当前任务相关的精简记忆。
7. 增加用户可查看、修改、删除记忆的前端入口，避免“黑箱记忆”。
8. 增加 Demo/Test Profile Reset：清空测试画像、加载路演种子、记录 demo_run_id 和 step trace。

#### 3.6.10 验收

- 用户说“小王是我的同事，以后技术问题我一般会问他”，系统能提取出同事关系和技术求助偏好。
- 用户说“我下周三晚上和小李约饭”，系统能识别为事实性未来安排，并交给 Maxwell 生成待确认日程或任务。
- 用户开玩笑或随口吐槽时，系统不会把它保存成长期事实。
- 用户多次表达类似偏好后，系统能形成稳定偏好，而不是每次都重复提取。
- Agent 回复时能使用相关记忆，但不会把大量无关历史塞进 prompt。
- 用户能看到系统记住了什么，并能删除或修改。
- 用户连续几天表达焦虑、疲惫、熬夜或节律紊乱时，系统能形成趋势信号，供心理管理员或生活节律管理员温和关心。
- 心理与节律类敏感记忆有隐私等级，不会被无关 Agent 随意读取或在回复中频繁引用原文。
- 开发者能一键清空测试画像与测试记忆，加载固定路演画像，从今天开始重新测试。
- 每个路演脚本步骤都能追溯到对应输入、Agent 回复、工具调用、记忆提取、确认动作和写入结果。

### 【部分实现：已有工具 schema、白名单和确认卡机制；还差统一 Skill Registry、权限、成本和外部 skill 治理】Step 4：Skill Registry 强化

目标：保留并强化外部 skill / API / 文件 / 网页 / 代码能力。

验收：

- skill 有 schema、权限、确认策略。
