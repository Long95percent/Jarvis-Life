# Jarvis-Life 比赛演示 Demo 方案

## 0. 演示定位

Jarvis-Life 是一个面向个人工作与生活场景的 AI 工作台。它不是单一聊天机器人，而是一个由多角色 Agent、工具执行、记忆系统、日程计划、本地生活信息和圆桌协作共同组成的“个人 AI 生活操作系统”。

比赛演示应重点回答三个问题：

1. **它解决什么痛点？** 个人每天面对日程、任务、学习、健康、情绪、生活选择时，信息分散、行动难落地、长期偏好无法被记住。
2. **AI 起什么关键作用？** AI 不只是生成回答，而是理解意图、选择角色、调用工具、提取记忆、组织多 Agent 协作，并把建议转成可确认的行动。
3. **是否完整闭环？** 从用户输入，到 Agent 分析，到工具执行，到待确认动作，到日历/计划/记忆/主动提醒，形成可落地闭环。

建议演示数量控制在 5-6 个。以下 6 个 demo 覆盖评分维度中的完整性、价值、创新性与技术实现性。

---

## Demo 1：多角色私人管家私聊 —— 从自然语言到可执行行动

### 演示目标

展示 Jarvis-Life 不是普通聊天框，而是一个“角色化 AI 管家团队”。用户可以和不同专家私聊，系统根据角色能力、工具白名单、本地意图识别和上下文，给出可执行结果。

### 场景痛点

用户常见需求不是单纯问答，而是：

- “帮我安排一个任务”；
- “提醒我某件事”；
- “今天该先做什么”；
- “我现在有点累，怎么安排比较合理”。

普通 AI 回答停留在建议层，无法连接日程、任务、记忆和后续提醒。

### 推荐演示脚本

1. 打开 Jarvis 首页。
2. 选择 Maxwell 私聊。
3. 输入：
   > 明天下午三点提醒我和产品同学开一个路演复盘会，提前半小时准备材料。
4. 展示 Maxwell 回复中出现待确认日程卡片。
5. 点击“确认写入日程”。
6. 打开日历面板，展示该安排已进入日程。

### 可展示能力

- 多 Agent 角色：Maxwell 作为秘书角色处理日程与任务。
- 本地 intent + slot：从自然语言中提取时间、事件、提醒意图。
- 工具调用：调用日程/待确认动作相关工具。
- 安全闭环：写入前通过 pending confirmation 让用户确认，不直接替用户改日程。
- 结果落地：确认后进入日历或计划视图。

### 对评分维度的价值

- **完整性与价值：** 从自然语言到日程写入形成闭环。
- **创新性：** 角色化 Agent + 工具白名单 + 待确认动作，兼顾智能与可控。
- **技术实现性：** 前端私聊、后端 `/chat`、本地 intent、工具执行、持久化、日历展示打通。

### 演示注意点

- 不要演示过于复杂的多轮日程冲突，先走确定性较强的日程创建。
- 如果模型速度较慢，可以提前准备好本地或云端 LLM 配置。
- 强调“确认后才执行”，这是产品可信度亮点。

## Demo 2：日程与长期计划闭环 —— 从大目标到每日任务

### 演示目标

展示 Jarvis-Life 能把模糊目标拆解成长期任务/每日计划，并和日历、后台任务、Maxwell 工作台联动。

### 场景痛点

用户常常有长期目标，例如备考、项目交付、健身计划，但难点在于：

- 目标太大，无法拆成每日行动；
- 每天任务和日历脱节；
- 过期任务没有自动处理；
- 用户需要一个持续推进的计划系统。

### 推荐演示脚本

1. 选择 Maxwell 或 Athena 私聊。
2. 输入：
   > 我想在 30 天内准备一次 AI 产品路演，每天安排一点准备任务，帮我拆成可执行计划。
3. 展示 AI 生成的后台任务计划卡片。
4. 点击“确认加入后台任务”。
5. 打开日历面板或任务视图，展示后台任务、计划日、每日任务。
6. 可选：展示 Maxwell 工作台或 planner task 列表。

### 可展示能力

- 长期任务拆解。
- 计划日模型。
- 后台任务与日历面板联动。
- Maxwell 秘书式工作台。
- 计划确认机制。

### 对评分维度的价值

- **完整性与价值：** 从目标输入到任务拆解再到日历/任务展示，覆盖真实执行链路。
- **创新性：** 把 LLM 生成能力接入长期计划系统，而不是只生成一段建议。
- **技术实现性：** 后端已有 `/plans`、`/plan-days`、`/background-tasks`、`/planner/tasks`、`/planner/calendar-items` 等接口支撑。

### 演示注意点

- 计划类 demo 可能涉及较多数据，建议提前清理或准备一组演示数据。
- 重点展示“确认加入后台任务”和“日历可见”，不要把时间花在复杂编辑上。
- 如果页面存在历史脏数据风险，提前跑一次日历面板确认不黑屏。

## Demo 3：场景圆桌会议 —— 多 Agent 协作决策，而不是单模型独白

### 演示目标

展示 Jarvis-Life 的差异化能力：多个专业 Agent 不是简单轮流聊天，而是在预设生活场景中进行结构化讨论，形成建议、共识、分歧和可确认行动。

### 场景痛点

现实问题往往不是单一维度，例如“今天很累但还有学习任务”，既涉及：

- 学习收益；
- 日程压力；
- 情绪与恢复；
- 总体取舍。

单个 AI 角色容易给出片面建议，多 Agent 圆桌更适合复杂决策。

### 推荐演示脚本

1. 打开 Jarvis 首页的场景卡片。
2. 选择“疲惫学习决策”或“今日日程协调”。
3. 输入：
   > 我今天很累，但晚上还想推进路演准备，到底该继续学还是休息？
4. 展示 Mira、Maxwell、Athena、Alfred 依次发言。
5. 展示轮次总结、共识、分歧、待确认问题。
6. 根据页面支持，点击继续讨论或接受建议。

### 可展示能力

- 六个预设圆桌场景。
- 角色按场景组合：Mira / Maxwell / Athena / Alfred 等。
- 圆桌流式事件：角色逐个发言，前端有会议舞台体验。
- 决策型圆桌：输出结构化建议。
- Brainstorm 型圆桌：输出 themes / ideas / tensions / followups。

### 对评分维度的价值

- **完整性与价值：** 从复杂问题输入，到多专家讨论，再到结构化结果，形成决策辅助闭环。
- **创新性：** 场景化多 Agent 图编排，而不是单一 chatbot。
- **技术实现性：** 后端存在 `/roundtable/start`、`/roundtable/continue`、`/roundtable/{session_id}/accept/save/return`，前端存在 `RoundtableStage`。

### 演示注意点

- 优先选择“疲惫学习决策”或“今日日程协调”，这两个更贴近比赛中的实际价值表达。
- 不建议演示太长的多轮圆桌，控制在 1 轮 + 总结。
- 如果现场网络或模型慢，提前准备 prompt，减少输入时间。

## Demo 4：心理关怀与主动提醒 —— 从被动聊天到持续陪伴

### 演示目标

展示 Jarvis-Life 不只是任务工具，也能在心理压力、疲惫、低能量场景下进行轻量陪伴、状态记录、关怀反馈和主动提醒。

### 场景痛点

用户的压力、疲惫、情绪波动通常不会自动进入任务系统。传统日历只记录事项，不理解人的状态。

### 推荐演示脚本

1. 选择 Mira 私聊。
2. 输入：
   > 我最近有点焦虑，事情很多，晚上也睡不好。
3. 展示 Mira 的陪伴式回应和可能的情绪/关怀卡片。
4. 展示心理趋势或关怀中心入口。
5. 展示主动消息 feed 或 care feedback，例如“有帮助 / 稍后提醒 / 不需要”。

### 可展示能力

- Mira 心理陪伴角色。
- 心理状态识别与 mood snapshot。
- care trends / care days / stress signals。
- 主动关怀消息。
- 用户反馈闭环，避免过度打扰。
- 安全边界：不是医疗诊断，而是轻量支持与求助提醒。

### 对评分维度的价值

- **完整性与价值：** 情绪表达 → 状态识别 → 关怀建议 → 趋势查看 → 反馈调节。
- **创新性：** 把日程压力、行为信号和心理关怀结合，而不是孤立聊天。
- **技术实现性：** 后端存在 `/care/settings`、`/care/trends`、`/care/days/{day}`、`/care/behavior-events`、`/messages`、`/messages/{id}/care-feedback` 等接口，前端有 `CareTrendsPanel`、`CareCard`、`ProactiveMessageFeed`。

### 演示注意点

- 强调安全边界：产品做陪伴、记录、提醒，不做医疗诊断。
- 不要输入极端风险文本，以免现场输出不可控。
- 演示重点放在“人性化”和“闭环反馈”。

## Demo 5：本地生活机会雷达 —— 从日程空档到真实生活建议

### 演示目标

展示 Jarvis-Life 能结合位置、天气、空档、体力和用户状态，推荐本地生活活动，帮助用户把“恢复精力”落到现实行动。

### 场景痛点

普通推荐系统只推荐附近地点；普通日历只知道时间。用户真正需要的是：

- 今天有没有空？
- 天气适不适合？
- 我累不累？
- 哪个活动最适合现在的我？

### 推荐演示脚本

1. 打开首页 Dashboard，展示天气、时间、状态卡片。
2. 选择 Leo 私聊或“本地生活活动推荐”圆桌。
3. 输入：
   > 我今天有点累，晚上想放松一下，帮我找一个附近合适的活动。
4. 展示 Leo 的活动建议。
5. 如果走圆桌，展示 Leo 提案、Maxwell 评估时间、Nora 评估体力、Alfred 汇总。
6. 可选：把活动安排进日程或生成待确认动作。

### 可展示能力

- 本地生活搜索与缓存。
- 天气/位置/时间上下文。
- Leo 生活顾问角色。
- 场景圆桌中的跨角色评估。
- 活动和日程联动。

### 对评分维度的价值

- **完整性与价值：** 从用户状态和空档出发，给出可执行的现实生活建议。
- **创新性：** 本地生活机会不是孤立搜索，而是进入 Agent 决策和主动提醒。
- **技术实现性：** 后端存在 `local_life_search.py`、`local_life_aggregator.py`、`/local-life`、`jarvis_local_life_search`、`jarvis_plan_activity_slot` 等能力。

### 演示注意点

- 本地生活搜索依赖外部搜索质量，建议提前缓存或准备稳定城市/关键词。
- 如果实时搜索不稳定，可强调系统支持缓存和跨场景读取。
- 不要承诺地图级实时路线精度，当前更适合作为“机会雷达”。

## Demo 6：长期记忆与偏好学习 —— 让 AI 越用越懂你

### 演示目标

展示 Jarvis-Life 的长期价值：它可以从对话中提取长期记忆、用户约束和偏好，并在后续对话中使用，而不是每次从零开始。

### 场景痛点

普通 AI 工具最大问题之一是“无连续性”：

- 每次都要重复个人偏好；
- 不记得饮食限制、作息习惯、学习目标；
- 无法在多角色之间共享上下文。

### 推荐演示脚本

1. 和 Nora 私聊输入：
   > 我晚上尽量不喝咖啡，最近想控制糖分，但不想完全不吃甜食。
2. 和 Mira 或 Maxwell 再聊一个相关问题。
3. 打开 MemoryPanel，展示提取出的偏好/长期记忆。
4. 展示 Shadow 偏好学习或记忆列表。
5. 再次询问饮食/晚间安排，观察 AI 是否参考偏好。

### 可展示能力

- 长期记忆提取。
- 偏好学习器 Shadow。
- 协作记忆：角色之间共享协调结果。
- 记忆查看与删除。
- 后台记忆提取，不阻塞主回复。

### 对评分维度的价值

- **完整性与价值：** 对话 → 记忆提取 → 记忆管理 → 后续个性化回复。
- **创新性：** 将记忆系统与多 Agent、工具执行、主动提醒结合。
- **技术实现性：** 后端存在 `/memories`、`/shadow/profile`、`memory_extractor.py`、`memory_recall.py`、`preference_learner.py`，前端存在 `MemoryPanel`。

### 演示注意点

- 后台记忆提取可能需要一点时间，可提前准备一轮对话。
- 现场重点展示“可查看、可删除、可用于后续回复”，不要只说系统会记住。
- 强调用户可控：记忆不是黑箱，可以管理。

## 推荐演示顺序

如果现场时间有限，建议顺序如下：

1. Demo 1：私聊到日程行动闭环。
2. Demo 3：多 Agent 圆桌决策。
3. Demo 2：长期计划与日历联动。
4. Demo 4：心理关怀与主动提醒。
5. Demo 5：本地生活机会雷达。
6. Demo 6：长期记忆与偏好学习。

如果只能演示 3 个，建议保留：

1. 私聊行动闭环。
2. 多 Agent 圆桌。
3. 记忆/计划/主动关怀三选一，根据现场稳定性选择。

---

## 稳定演示建议

1. 演示前启动 Python AI 服务和 React 前端，确认 `/api/v1/jarvis/agents`、`/api/v1/jarvis/scenarios` 能正常返回。
2. 提前配置可用 LLM provider，避免本地 CPU 模型导致等待过长。
3. 演示前准备好 3-5 条固定 prompt，不现场临时发挥。
4. 涉及本地生活搜索时提前跑一次，让缓存可用。
5. 涉及记忆提取时提前准备一轮对话，避免等待后台任务。
6. 日历/计划 demo 前确认没有历史脏数据导致页面黑屏。
7. 如果某条链路不稳定，仍可从接口/页面展示能力，但不要现场硬等模型。
